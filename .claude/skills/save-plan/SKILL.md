# プロンプト内容

あなたは、会話の中で**すでに合意済みの実装計画**を計画ファイルとして書き出すアシスタントです。

このスキルは「計画を生み出す」スキルではありません。`/grill-me` などで固めた計画を、消えないようにファイルへ**結晶化**するのが唯一の責務です。
新たな設計判断を勝手に追加せず、会話で合意した内容だけを忠実に記録してください。

**最重要ルール:**
- 会話で未確定・曖昧なまま残っている点があれば、保存に進む前に必ずユーザーに確認する。想像で埋めない。
- ADR起票はユーザーが判断する。Claudeは勝手にADRを作成しない。

---

## Phase 1: Issue 番号の検出

`plan-issue` スキルの Phase 1 と**同一ロジック**を使う（規則を一元管理するため、ここでは概要のみ）。

1. `git branch --show-current` でブランチ名を取得する。
2. `*issue-{N}*` パターン（`feat/issue-123`, `fix/issue-456`, `docs/issue-789` など）から番号 `{N}` を抽出する。
3. 抽出できない場合（`develop`, `main` 等）は `$ARGUMENTS` から取得する（`#123` or `123`）。
4. どちらからも取得できない場合は、以下を表示して終了する:

```
❌ Issue番号を検出できませんでした。
使い方: `/save-plan` （issue-* ブランチ内で実行）または `/save-plan 123`
```

---

## Phase 2: 計画内容の組み立て

会話の文脈から、合意済みの計画内容を抽出してまとめる。

- フォーマットは `docs/claude-plans/PLAN_TEMPLATE.md` に**従う**（概要 / 設計判断 / ステップ）。テンプレートを Read して構造を取得すること。スキル内にフォーマットを再掲しない。
- 「設計判断」セクションには、会話で実際に選択した判断と**その理由**を記載する。判断がなければ「なし」と書く。
- ステップは実装の意味的なまとまりで分割し、各ステップにコミットメッセージ案を添える（CLAUDE.md のコミット規約に沿う）。

---

## Phase 3: ファイル名の決定

ファイル名は**実装内容を端的に表す kebab-case** にする。Issue番号やタイプ接頭辞は付けない（ディレクトリ名が既に `issue-{N}` のため）。

- 良い例: `create-estimate-command.md`, `implement-prisma-estimate-repository.md`
- 悪い例: `plan.md`（内容が分からない）, `issue-298-plan.md`（番号が冗長）, `見積コマンド作成.md`（日本語/英語混在は避け、英語 kebab-case に統一）

保存先パス: `docs/claude-plans/issue-{N}/{実装内容の英語kebab-case}.md`

同名ファイルが既に存在する場合は、上書きする前にユーザーへ確認する（別作業の計画を潰さないため）。

---

## Phase 4: 保存と完了報告

ディレクトリ `docs/claude-plans/issue-{N}/` が無ければ作成し、ファイルを書き出す。

```
✅ 実装計画を保存しました

📋 Issue: #{N}
📄 計画: docs/claude-plans/issue-{N}/{file-name}.md
🌿 Branch: {現在のブランチ名}

実装に進む場合は `/tdd` を使ってください。
```
