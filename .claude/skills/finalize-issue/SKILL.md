# プロンプト内容

あなたは、設計が確定した仮イシューを**本イシュー**へ更新するアシスタントです。

このスキルは「設計するスキル」ではありません。`/plan-issue` → `/save-plan` で
`docs/claude-plans/issue-{N}/` に**結晶化済みの計画を正**として、仮イシューの本文に
実装タスク・受け入れ条件を追記し、ラベルを `Status: draft` → `Status: ready` へ付け替えるのが唯一の責務です。

**最重要ルール:**
- 実装タスク・受け入れ条件は**計画ファイルから**生成する。会話や想像で新しい設計判断を足さない。
- 計画ファイルと仮イシューの記述が矛盾する場合は、保存に進む前に必ずユーザーへ確認する。
- ADR起票はユーザーが判断する。Claudeは勝手にADRを作成しない。

---

## Phase 1: Issue 番号の検出

`plan-issue` スキルの Phase 1 と**同一ロジック**を使う。

1. `git branch --show-current` でブランチ名を取得する。
2. `*issue-{N}*` パターン（`feat/issue-123` など）から番号 `{N}` を抽出する。
3. 抽出できない場合（`develop`, `main` 等）は `$ARGUMENTS` から取得する（`#123` or `123`）。
4. どちらからも取得できない場合は、以下を表示して終了する:

```
❌ Issue番号を検出できませんでした。
使い方: `/finalize-issue` （issue-* ブランチ内で実行）または `/finalize-issue 123`
```

---

## Phase 2: 計画ファイルの取得（本化の入力）

```bash
ls docs/claude-plans/issue-{N}/
```

- 計画ファイル（`PLAN_TEMPLATE.md` 形式: 概要 / 設計判断 / ステップ）を読む。
- 複数ある場合は、どれを正とするかユーザーに確認する。
- **計画ファイルが無い場合**は、以下を表示して終了する（勝手に設計しない）:

```
❌ docs/claude-plans/issue-{N}/ に計画が見つかりません。
本化には設計が確定した計画が必要です。先に `/plan-issue` → `/save-plan` で計画を作成してください。
```

---

## Phase 3: 現在のイシュー取得

```bash
gh issue view {N} --json title,body,labels,comments
```

- イシューが存在しなければエラー表示して終了する。
- `Status: draft` ラベルが無い場合は警告する（既に本化済みの可能性）。ユーザーが続行を望めば進む。
- 仮イシュー本文の「概要 / 背景・目的 / 解決したい課題」は**本化後も引き継ぐ**。
- 仮イシューの記述と計画ファイルの内容に食い違いがあれば、ここでユーザーに確認する。

---

## Phase 4: 本文の組み立て

計画ファイルを正として、以下の本イシュー本文を組み立てる。仮イシューの背景情報は保持する。

```markdown
## 概要

(計画の「概要」と仮イシューの「概要」を統合した要約)

## 背景・目的

(仮イシューから引き継ぐ)

## 設計方針

(計画の「設計判断」から、選択した方式とその理由を記載。判断がなければ「既存パターン踏襲のため特記なし」)

## 実装タスク

- [ ] (計画の「ステップ」を実装タスクに変換。各ステップが1タスクに対応)
- [ ] ...

## 受け入れ条件

- [ ] (計画から導かれる「完了の定義」を検証可能な形で記載)
- [ ] ...

## 関連 issue

- (仮イシューから引き継ぎ。あれば追記)
```

- **実装タスクは計画の「ステップ」と対応**させる。計画に無い作業を勝手に足さない。
- 受け入れ条件は「ユーザー視点で完了を検証できる条件」を書く（実装タスクの言い換えにしない）。
- bug の場合は「症状 / 再現手順 / 影響範囲」を残し、計画から「原因 / 解決策 / 実装タスク / 受け入れ条件」を追記する。

組み立てた本文をユーザーに提示し、本化してよいか確認する。

---

## Phase 5: イシューの更新とラベル付け替え

確認が取れたら、本文を差し替えラベルを付け替える。

```bash
# 本文は一時ファイル経由で渡す（改行・記号の崩れ防止）
gh issue edit {N} --body-file <(cat <<'EOF'
{組み立てた本文}
EOF
)
gh issue edit {N} --add-label "Status: ready" --remove-label "Status: draft"
```

- `Status: ready` が存在しなければ作成する: `gh label create "Status: ready" --color 0e8a16`
- タイトルの接頭辞（`feat:` など）は維持する。設計でタイプが変わった場合のみユーザー確認の上で変更する。

---

## Phase 6: 完了報告

```
✅ 仮イシューを本化しました

📋 Issue: #{N} {title}
🏷️  Status: draft → ready
📄 元計画: docs/claude-plans/issue-{N}/{file-name}.md
🔗 {issue URL}

実装に進む場合は `/tdd` を使ってください。
```
