# プロンプト内容

あなたは GitHub の**イシュー**作成の専門家です。
#$ARGUMENTS について、設計を詰める前の起票としてイシューを作成してください。

## このスキルの役割（最重要）

このスキルは**まだ設計を詰め切っていない段階**で起票するためのものです。

- **やること**: 「何を・なぜ・何が課題か・何を決める必要があるか」を残す
- **やらないこと**: 実装タスク・受け入れ条件を**書かない**。これらはブランチ内での設計後に PR で扱う

理由: 起票時点で「設計済みのフリ」をした実装タスク・受け入れ条件を書くと、ブランチを切って実際に設計したときの結論とズレ、混乱の原因になる。起票したイシューは設計の「入力」であって「成果物」ではない。

設計の流れ:
1. `/create-issue` … イシュー作成（このスキル）
2. `/plan-issue` → `/save-plan` … ブランチ内で設計し `docs/claude-plans/issue-{N}/` に計画を結晶化
3. 実装し、イシューに対して PR を作成する

## ステップ 1: タイプ自動判定

`.claude/references/commit-types.md` を読み、マッピングテーブルを取得する。
`$ARGUMENTS` のキーワードとテーブルの `keywords` 列を照合して issue タイプを推定する。
テーブルの `prefix` 列をタイトル接頭辞に、`label` 列を GitHub ラベルに使用する。

- 複数マッチした場合は最も適切なものを選択
- どれにも該当しない場合は `feat`（`Type: enhancement`）をデフォルトとする

## ステップ 2: ラベル存在チェック

`gh label list` でタイプラベル（`Type: xxx`）の存在を確認する。

タイプラベルが存在しない場合はそのラベルなしで作成する（確認不要）。

## ステップ 3: 優先度判定（オプション）

以下のキーワードがあれば優先度ラベルも付与する:

- 「緊急」「critical」「本番障害」「ブロッカー」→ `Priority: critical`
- 「低優先」「後回し」「いつか」→ `Priority: low`
- 上記以外 → 優先度ラベルなし（付与しない）

優先度ラベルも存在チェックを行う。

## ステップ 4: テンプレート選択

判定したタイプに応じて以下のテンプレートを使用する。
**いずれのテンプレートにも実装タスク・受け入れ条件は含めない**。代わりに「未決事項 / 要設計」に、設計時に決めるべき論点を列挙する。

### 実装系（feat / refactor / test / ci / agent / build / chore / perf）テンプレート

```markdown
## 概要

(何をしたいか)

## 背景・目的

(なぜこれが必要か。どういう状況で出てきたか)

## 解決したい課題

- (課題1)
- (課題2)

## 未決事項 / 要設計

- [ ] (設計時に決めるべき論点1: 例「承認ステップの生成方式」)
- [ ] (論点2)

## 関連 issue

- なし
```

### bug（fix）テンプレート

バグは観測事実が中心なので症状・再現手順は具体的に書く。ただし**原因と解決策は設計（調査）後**にブランチ内で扱う。

```markdown
## 症状

(何が起きているか)

## 再現手順

1. 手順1
2. 手順2

## 影響範囲

(どの機能・ユーザーに影響があるか)

## 未決事項 / 要調査

- [ ] 原因の特定
- [ ] 解決方針の決定

## 関連 issue

- なし
```

### docs / question / architecture テンプレート

これらは「議論・決定すること自体」が目的のイシュー。起票段階では論点と背景を残す。

```markdown
## 疑問・課題

(何を決める必要があるか)

## 背景・コンテキスト

(どういう状況で出てきた疑問か、関連するコードや設計)

## 検討の方向性（あれば）

(現時点で考えている選択肢。なければ「未検討」)

## 未決事項

- [ ] (決めるべきこと1)

## 関連 issue

- なし
```

## ステップ 5: issue 作成（確認不要）

```bash
gh issue create \
  --title "{prefix}: タイトル" \
  --label "Type: xxx" \
  --body "本文"
```

- 優先度ラベルがある場合は `--label "Priority: xxx"` も追加
- `$ARGUMENTS` の内容を分析してテンプレートの各セクションを埋める
- **実装タスク・受け入れ条件は書かない**。判断に迷ったら「未決事項 / 要設計」へ回す
- 関連 issue があれば本文中にリンクする

## ステップ 6: 結果を返す

作成した issue の番号と URL を簡潔に返す。
設計に進む際は `/plan-issue` → `/save-plan` を使う旨を一言添える。
