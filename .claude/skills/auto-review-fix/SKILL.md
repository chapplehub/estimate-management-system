# プロンプト内容

あなたは PR の自己修正ループを回す自動レビュー＆修正エージェントです。
`$ARGUMENTS`（`[PR番号] [深さ]` / いずれも省略可）を入力として、**`/code-review` → 評価 → 妥当かつ高重大度のみ修正 → push → 再レビュー** を収束するまで自動で繰り返してください。

## 重要な前提条件

- **対象ブランチの worktree 内から実行する**（feat/fix 等の作業ブランチ）。`develop` / `main` / `master` では実行しない。
- 自動 push は **作業ブランチのみ**。`pre-bash-branch-safety.sh` フックと整合させる。
- ユーザーへの逐次確認は行わない（ループ全体を自動実行する）が、**終了条件に到達したら必ず止まる**。
- **レビューは必ず組み込み `/code-review` を使う**（`SlashCommand` ツール経由）。独自レビューで代替しない。
  - 事前に許可リストへ `SlashCommand(/code-review*)` が必要な場合がある。拒否されたらユーザーに許可を依頼する。

---

## Phase 0: 入力解析・対象 PR の特定・安全確認

### 0.1 入力解析（PR番号と深さ）

`$ARGUMENTS` を空白区切りで解析する。

- **深さ**: `low` / `medium` / `high` / `xhigh` / `max` のいずれかのトークンがあればそれを採用。なければ既定 **`medium`**。
  - これは `/code-review {深さ}` の effort level に渡す。深いほど不確実な指摘も拾う代わりに時間がかかる。
- **PR番号**: `#123` / `123` 形式のトークンがあればそれを使う。省略時は 0.2 で特定する。

例: `/auto-review-fix #123 high` → PR 123 を深さ high で。`/auto-review-fix xhigh` → 現在ブランチの PR を深さ xhigh で。

### 0.2 PR 番号の特定（省略時）

```bash
git rev-parse --abbrev-ref HEAD
gh pr view --json number,title,headRefName,state
```

- PR が存在しない / クローズ済みの場合はエラーとして理由を返し終了する。

### 0.3 Issue 番号と計画ファイルの特定

- ブランチ名（`feat/issue-123` 等）から Issue 番号を抽出する。
- 計画ファイルの場所を控える（評価ステージで読む）:

```bash
ls docs/claude-plans/issue-{番号}/
```

- `plan.md` または自動命名の `*.md`、加えて `deviations.md`（あれば）。**計画ファイルが見つからない場合はその旨を記録し、評価は実装意図の推測に頼る**ことを明記する。

### 0.4 安全確認（ブロッカー）

- ブランチが `develop` / `main` / `master` の場合は **即終了**する。理由: 作業ブランチ以外への自動 push を禁止するため。
- worktree がクリーンでない（未コミットの変更がある）場合は一旦コミット or 報告して終了する。

## ループ本体（最大 3 ラウンド）

以下の Phase 1〜5 を、**終了条件に達するまで**繰り返す。ラウンド番号を `R`（1 始まり）とする。

### 終了条件（いずれか満たしたら停止）

1. **収束**: 評価で**採用された** High 以上の指摘が 0 件 → 成功として Phase 6 へ。
2. **回数上限**: `R > 3` → 未収束として Phase 6 へ（残課題を報告）。
3. **no-progress**: 前ラウンドと比べて採用 High 以上の件数が**増えている**（増加） → 発散とみなし Phase 6 へ（人間の判断が必要と報告）。

---

### Phase 1: レビュー（組み込み `/code-review` を確実に使う）

**`SlashCommand` ツールで `/code-review {深さ} --comment` を実行する。** ここはメイン文脈で実行する（スラッシュコマンドはサブエージェント内では動かないため）。

- **`--comment` を付ける**: レビュー結果を PR に残すため。生の `/code-review` 指摘が PR の記録として残る。
- `--fix` は **付けない**（評価前に勝手に直させない）。修正は評価を通った High 以上だけに限定する。
- findings はセッション内にも表示されるので、それを次の評価ステージへ渡す。
- `/code-review` が指摘ゼロを返したら「収束」。Phase 6 へ。

### Phase 2: 評価（fresh subagent で妥当性を judge）— **鵜呑みにしない**

`/code-review` の結果を**そのまま採用しない**。毎ラウンド新しいサブエージェントを `Agent`（Task）ツールで起動し、**評価者（judge）**として各指摘の妥当性を判定させる。
理由: レビューを出す者と採否を決める者を分離し、誤検知（false positive）の修正・計画と矛盾する修正を防ぐため。各ラウンド起動し直す（文脈を引き継がない）。

**judge は判定のみ。コード修正・ファイル編集・コミットを絶対に行わない**（Edit / Write / git commit を禁止）。judge の責務は「採否と理由を返すこと」だけで、実装には一切触れない。

judge サブエージェントへのプロンプトに以下を**すべて**渡す:

1. **`/code-review` の指摘リスト**（Phase 1 の出力そのまま）
2. **実装の変更内容**: `git diff develop..HEAD`
3. **計画ファイルの中身**: `docs/claude-plans/issue-{番号}/*.md`（`deviations.md` 含む。無い場合はその旨）
4. 下記「Severity Rubric」
5. judge の判定基準（下記）

#### judge の判定基準

各指摘について、次を判断して `{採否, 理由}` を返す:

- **採用**: 指摘が実装上の実害（バグ/設計逸脱/制約違反）を正しく突いており、計画とも矛盾しない。
- **却下**: 次のいずれか。
  - **計画準拠**: 指摘は「あるべき」と言うが、**計画ファイルで意図的にそう設計している**（＝計画が正、指摘が文脈無視）。
  - **誤検知**: コードを誤読している / 実際には成立しない経路。
  - **スコープ外**: この PR の責務外（別 Issue にすべき）。
- 採用したものには Severity Rubric に従って **severity を確定**させる。

##### Medium / Low の例外採用（安全な共通化のみ）

Medium / Low は原則このループでは直さないが、**次の 3 条件をすべて満たす指摘だけ**は例外的に「採用（安全共通化）」とタグ付けする:

1. **設計判断が不要**: どこに置くか・どう抽象化するか・責務をどう分けるかで迷う余地がない。判断が要る時点で対象外。
2. **構造・状態の追加を伴わない**: 新しい型 / クラス / インターフェース / フィールド / 抽象レイヤ / 状態変数を一切増やさない。**既存の構造・関数の範囲内で完結する**（例: インラインで再実装している箇所を既存ユーティリティ呼び出しに置換、完全重複した分岐の統合、デッドコード除去）。
3. **共通化による純減**: 差し引きで**コード行数が減る**（追加 < 削除）。新規ヘルパーの新設のように行数が増える共通化は対象外。

- いずれか 1 つでも満たすか怪しい場合は **却下**（Medium / Low のまま残課題へ）。**迷ったら直さない**を徹底する。
- 「安全共通化」採用は High とは別カウントにする（ループ継続/no-progress 判定には影響させない。詳細は Phase 3）。

judge の出力形式: 指摘ごとに `{severity, file:line, 問題, 採否, 理由}` の構造化リスト。安全共通化採用には `採否=採用(安全共通化)` と明記する。

#### judge 判定を PR に投稿（必須）

judge の結果を受け取ったら、メイン文脈から **PR に judge 判定を残す**:

```bash
gh pr comment {PR番号} --body "$(cat <<'EOF'
## ⚖️ judge 評価 ラウンド {R}（/code-review {深さ} の指摘を計画準拠で再評価）

### 採用（修正対象）
- [{severity}] {file:line} {問題} → 理由: {採用根拠}

### 却下（修正しない）
- {file:line} {問題} → 却下理由: {計画準拠 / 誤検知 / スコープ外}
EOF
)"
```

これにより「`/code-review` の生指摘（Phase 1 の `--comment`）」と「judge の採否判断」の**両方が PR に記録**として残る。

#### Severity Rubric（このリポジトリ固有）

- **Critical**: ビルド/テストが壊れる、データ破壊、認証・認可の欠落。
- **High**（= 採用されれば自動修正の対象）:
  - **DDD 層違反**: Domain 層が infrastructure / application / presentation に依存。Domain 層が Prisma / Next.js / 外部ライブラリを import している。Application 層が repository の具象実装に依存（interface でない）。
  - **楽観ロック欠落**: 更新系で version を伴わない書き込み（特に Employee は外部認証副作用ゆえ version 付き書き込みを先頭に）。
  - **NULL 方針違反**: kawasima デシジョンツリー準拠の NULL 排除方針に反する nullable 追加。
  - 明確なバグ、データ不整合、未処理の例外パス。
- **Medium**: テスト不足、命名/責務の乱れ、軽微な設計の歪み。
- **Low**: スタイル、コメント、好みの範囲。

### Phase 3: 振り分け

今ラウンドの修正対象は次の 2 種類:

1. **採用 High 以上** — judge が採用した High / Critical の指摘。
2. **採用(安全共通化)** — judge が Medium / Low から例外採用した、3 条件（設計判断不要・構造/状態の追加なし・純減）を満たす共通化。

- **収束判定・no-progress 判定は「採用 High 以上」の件数だけ**で行う（安全共通化は判定に含めない）。理由: 安全共通化は機会主義的な掃除であり、ループの継続根拠にしないため。
- 採用 High 以上が 0 件なら「収束」。ただし**今ラウンドに安全共通化があれば、それだけは Phase 4 で処理してから** Phase 6 へ。
- **却下した指摘**と、例外条件を満たさない **Medium / Low** はこのループでは修正しない（Phase 6 のサマリに理由付きで残す）。

### Phase 4: 高重大度の修正（計画ファイルを作ってから着手）

**いきなり修正しない。まず修正計画ファイルを作成してから実装に入る。**

#### 4.1 修正計画ファイルの作成

今ラウンドの修正対象（**採用 High 以上** および **採用(安全共通化)**）について、修正方針を計画ファイルへ書き出す。

- 保存先: `docs/claude-plans/issue-{番号}/`
- ファイル名: **ラウンド番号をプレフィックスにする**。ラウンド1なら `r1-`、ラウンド2なら `r2-`。
  - 例: `docs/claude-plans/issue-123/r1-fix-plan.md`
- 記載内容: 対象ごとに `{severity, file:line, 問題, 修正方針, 影響範囲, 想定テスト}` を書く。安全共通化は `純減見込み（削除-追加の行数）` も併記する。
- DDD 層配置の判断を伴う場合は、その配置理由も計画に明記する（後でコミットボディに転記する）。

#### 4.2 実装

- 計画ファイルに沿って、今ラウンドの修正対象のみを修正する。**DDD 層ルール（CLAUDE.md）を絶対に破らない。**
- 安全共通化は、実装後に**本当に純減になっているか（`git diff --stat` で追加 < 削除）を確認**する。増えていたら破棄し、却下扱いに戻す。
- 意味のあるまとまりごとにコミットする（一括コミットしない）。
  - High 修正は原則 `fix:`、安全共通化は `refactor:`（統合/共通化）でコミット。計画ファイル自体は `docs:`。
- コミットは `.claude/references/commit-types.md` の type に従う（修正なので原則 `fix:`、設計変更を伴うなら適切な type）。
- **設計判断を含む場合はコミットボディに理由を記載**（CLAUDE.md のルール）。
- 修正後、テストを実行して緑を確認する:

```bash
pnpm test
pnpm lint
```

- テストが赤のまま push しない。直せない場合は Phase 6 で正直に報告する。

### Phase 5: push と再レビューへ

```bash
git push
```

- 今ラウンドの対応内容を PR にコメントする:

```bash
gh pr comment {PR番号} --body "$(cat <<'EOF'
## 🔁 自動レビュー＆修正 ラウンド {R}（/code-review {深さ}）

### 対応した指摘（採用・High 以上）
- {file:line} {問題} → {対応内容}

### 却下した指摘（参考）
- {file:line} {問題} → 却下理由: {計画準拠/誤検知/スコープ外}

### 次ラウンドで再レビューします
EOF
)"
```

- 採用 High 以上の件数を記録（no-progress 判定に使う）し、`R` を +1 して Phase 1 へ戻る。

## Phase 6: 収束サマリ

ループ終了後、結果を PR にコメントする。

```bash
gh pr comment {PR番号} --body "$(cat <<'EOF'
## ✅ 自動レビュー＆修正 完了サマリ（/code-review {深さ}）

- ラウンド数: {実行数} / 上限 3
- 終了理由: {収束 / 上限到達 / no-progress}

### 解消した High 以上の指摘
- ...

### 却下した指摘（計画準拠・誤検知・スコープ外）
- {file:line} {問題} → {却下理由}

### 残課題（人間の判断が必要）
- [Medium] {file:line} {問題}
- [未解消 High] {file:line} {理由（テストが直せない等）}
EOF
)"
```

- ユーザーへの最終報告では、**自動で解消したもの / judge が却下したもの / 人間の判断に委ねるもの**を明確に分けて返すこと。
- テスト赤・未解消 High が残った場合は、それを隠さず正直に報告する。
