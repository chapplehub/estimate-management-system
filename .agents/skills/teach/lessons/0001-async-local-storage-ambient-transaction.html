レッスン 01 · 集約またぎトランザクション

引数なしで tx を流す

AsyncLocalStorage による ambient トランザクションの「どこで・何を」

今回 issue-440 で入れたのは、「申請 submit の version bump（見積）＋ 申請挿入」を単一トランザクションで原子化する仕組み（ADR-20260626-dee）。 狙いは、bump コミット〜insert コミットの間に開く TOCTOU 窓（別バリエーションが二重に前進する隙）を閉じること。 ここで難しいのは DDD のレイヤー規律 を破らずにそれをやる点 — トランザクションは Prisma の概念で、アプリ層・ドメイン層に漏らしたくない。 その解が AsyncLocalStorage（ALS）による ambient トランザクション伝播です。

このレッスンのゴール（スキル）： 1本の「集約またぎ原子書き込み」を、コマンド → ポート → アダプタ → ALS → リポジトリ とレイヤーを跨いで指でたどれるようになる。どのファイルがどの責務を持つかを、見ずに言えるようになる。

核心のアイデア

普通にやるなら、リポジトリのメソッドに tx を引数で渡して回す。だがそれは全シグネチャを汚染し、ドメインインターフェースに Prisma 型が滲み出す。 ALS は 「非同期呼び出しの連鎖に、引数を経由せず暗黙の文脈を流す」 Node 標準機構。 これで tx を箱（ALS）に載せておき、リポジトリは currentClient() でその箱を覗くだけにする。

ambient（アンビエント）＝「その場の空気に溶けている」。tx を手渡しせず、実行文脈に漂わせて拾わせる、という発想。

6人の登場人物

① TransactionRunner app 層・ポート

src/server/shared/application/transaction/TransactionRunner.ts run<T>(work) だけのインターフェース。Prisma 型ゼロ。 「この work は1単位で原子的」とアプリ層が宣言するための抽象。トランザクションは非業務概念なので ドメイン層ではなく shared/application に置く（ADR-0039 と同型の論法）。

② txContext.ts infra 層・ALS の心臓

src/server/shared/infrastructure/transaction/txContext.ts
  • als … 「今の tx ハンドル」を載せる箱（AsyncLocalStorage<TransactionClient>）
  • currentClient() … ambient tx があればそれ、無ければ global prisma。リポジトリが触る唯一の入口
  • runInTx(tx, work) … als.run(tx, work)。tx を seed して work を走らせる（Runner が使う）
  • runAtomically(work) … join-or-open：ambient があれば相乗り、無ければ自分で $transaction

③ PrismaTransactionRunner infra 層・アダプタ

src/server/shared/infrastructure/transaction/PrismaTransactionRunner.ts prisma.$transaction((tx) => runInTx(tx, work)) の1行。$transaction を開き、その tx を ALS に seed。 DIP：具象（infra）がポート（app）を実装し、ソース依存が infra→app に向く。

④ リポジトリ群 infra 層・参加者

PrismaEstimateRepository / …ApplicationRepository / …ApprovalExemptionRepository prisma.xxx → currentClient().xxx に総置換。自前 prisma.$transaction していた多文メソッド（update）は runAtomically に。単独でも原子的、Runner 配下では外側に合流する二面性を獲得。

⑤ SubmitApplicationCommand app 層・利用者

src/server/subdomains/estimate/application/commands/SubmitApplicationCommand.ts TransactionRunner を注入（ポートのみに依存）。 txRunner.run(async () => { version 関門(update) → 挿入(insert) }) で bump と insert を1 tx に包む。 bump 後の挿入失敗は丸ごとロールバック＝無害な version 空振りすら起きない。

⑥ ESLint オーバーライド ガードレール

eslint.config.mjs（atomicSubmitRepositoryFiles） 参加3リポジトリで素の @server/prisma import を追加禁止。 うっかり global prisma を使って ambient tx を逃げ、原子性を静かに壊す事故を機械的に防ぐ。

1リクエストの流れ

SubmitApplicationCommand.execute
 └─ txRunner.run(work)                      [① app: ポート呼び出し]
     └─ PrismaTransactionRunner.run         [③ infra: アダプタ]
         └─ prisma.$transaction((tx) =>
              runInTx(tx, work))             [② infra: als.run で tx を seed]
               └─ work 実行中は als.getStore() === tx
                    ├─ estimateRepository.update(...)    → currentClient() = tx
                    └─ applicationRepository.insert(...) → currentClient() = tx
              （同一 tx で両方が走る／throw で全部ロールバック）

肝は update も insert も 引数で tx を受け取っていないこと。ALS が非同期連鎖を構造的に隔離するため、 可変ホルダ（ClientManager 方式）が抱えるリクエストスコープ規律やシングルトン化の罠なしに並行安全になる。

なぜ join-or-open（④の runAtomically）が要るのか： リポジトリの update は単独でも呼ばれる（他のユースケース）。 そのとき ambient が無ければ自分でトランザクションを開いて原子性を守り、ある（＝submit 配下）ときは外側に相乗りする。 1つの関数で「単独でも安全・合流もできる」を両立させている。

想起練習

答えを開く前に、声に出して言ってみること（fluency ではなく storage strength を作るため）。

1. リポジトリが tx を拾う「唯一の入口」の関数名は？ そのファイルは？ currentClient()。txContext.ts。ambient tx があればそれ、無ければ global prisma を返す。
2. tx を ALS に「seed」しているのはどの行か？ PrismaTransactionRunner の prisma.$transaction((tx) => runInTx(tx, work))。 runInTx の実体は als.run(tx, work)。
3. TransactionRunner をドメイン層ではなく shared/application に置いた理由は？ トランザクションは非ドメイン（非業務）概念だから。version=非業務概念ゆえアプリ層に置いた ADR-0039 と同型の論法。
4. ESLint で素 prisma import を禁止したのは何を防ぐためか？ global prisma を使うと ambient tx を逃げ、bump+insert の原子性を静かに壊すため。currentClient()/runAtomically 経由を強制する。

まず読むべき一次資料

Node.js 公式：Asynchronous context tracking（AsyncLocalStorage） — als.run(store, callback) と als.getStore() の正確な意味論、なぜ非同期連鎖をまたいで文脈が保たれるのかを、まず一次情報で押さえる。
