4 つの API の使い分け

currentClient()

読み書きのたびに呼ぶ。ambient tx があればそれ、無ければ global prisma。
用途：リポジトリの全クエリ。const db = currentClient(); db.xxx.findMany(...)

runAtomically(work)

リポジトリ内の多文メソッド（楽観ロック付き update 等）を包む。
join-or-open：ambient あれば相乗り／無ければ自前 $transaction。

txRunner.run(work)

アプリ層から集約またぎを原子化したいとき。コマンドが注入された TransactionRunner 経由で呼ぶ。

runInTx(tx, work)

内部用。PrismaTransactionRunner だけが使う。普段アプリ／リポジトリから直接は呼ばない。

紛らわしい2つ：txRunner.run vs runAtomically

どちらも $transaction を開いて ALS に seed するので似て見えるが、層・粒度・ネスト挙動が違う。 本質は2つ — ① app 層は Prisma を import 不可ゆえポート（txRunner.run）が要る、 ② リポジトリは単独でも submit 配下でも呼ばれるので join-or-open が要る。

軸txRunner.run(work)runAtomically(work)
層app 層ポート経由infra 関数を直接呼ぶ
呼ぶ人ユースケース（コマンド）リポジトリ内部の多文メソッド
境界の粒度集約またぎ（bump＋insert）1リポジトリメソッド内（root＋子群）
ネスト挙動always-open（最外殻専用）join-or-open（入れ子で相乗り）
協調する： txRunner.run が天井で tx を1回開き、その下のリポジトリの runAtomically は als.getStore() で「もう開いてる」と気づき相乗りする。結果、リポジトリのコードは 単独でも submit 配下でも一字一句同じで正しく動く（same code, two contexts）。
もし update が always-open のままだったら： submit 配下で呼ばれたとき global prisma で 2つ目の別 tx が開き、bump と insert が別トランザクションに割れて原子性が壊れる。 これが今回の TOCTOU 修正で $transaction を runAtomically に替えた急所。

新しいリポジトリを tx に参加させる手順

1. import を `prisma` → `currentClient, runAtomically` に差し替え
2. 全 `prisma.xxx` を `currentClient().xxx` に置換
3. 自前 `prisma.$transaction(async (tx) => ...)` を
   `runAtomically(async () => { const db = currentClient(); ... })` に
4. eslint.config.mjs の `atomicSubmitRepositoryFiles` にファイルを追加
5. 単独呼び出しのテスト（自前 tx）と Runner 配下のテスト（合流）の両方を書く
落とし穴： work の中で await を挟まず別の非同期スケジューリング（setTimeout・切り離した Promise）に処理を逃すと、 ALS の文脈は引き継がれない。tx 文脈に乗せたい処理は work の await チェーン上に保つこと。
