# 小数計算を整数で扱う方式（Money パターン）と端数処理の関心事

作成日: 2026-05-23

## 概要

金額のような「小数を含む数値」を浮動小数点で扱うと `0.1 + 0.2 !== 0.3` のような誤差が出る。
そのため最小単位（例: 1/100 円 = 銭）の整数で内部保持して四則演算する、というのが定石（Money パターン）。

このとき「整数→小数に戻すときの端数処理」が新しい関心事になる、という素朴な理解は **半分正しい**。
より正確には **「割り算を含む演算をした瞬間に端数処理の関心事が発生する」**。加減算・整数倍では発生しない。
そして見積ドメインでは丸める単位が「銭」と「円」の **2 段階** で存在するのが特徴。

## 詳細

### 1. 内部表現は常に整数のまま

`Money` クラスは内部で `minorUnits`（銭の整数）と `currency` のペアで保持する。
小数（円）に戻すのは `majorUnits` ゲッターで `÷100` して見せるときだけ、つまり **表示用の変換**。
計算の途中で「整数→小数→整数」と往復したりはしない。

ここが「戻すときに端数処理が必要」という直感の誤りやすいポイント。
計算中はずっと整数で、表示時の `÷100` では端数は出ない（割り切れる）。

### 2. 端数処理の関心事は「演算の種類」で発生する

| 演算 | 例 | 端数が出るか | 関心事 |
|---|---|---|---|
| 加算 `add` | 100銭 + 50銭 = 150銭 | 出ない | なし |
| 減算 `subtract` | 150銭 - 50銭 = 100銭 | 出ない | なし |
| 整数倍 `times` | 100銭 × 3 = 300銭 | 出ない（整数×整数） | なし |
| 比率適用 `applyRate` | 100,000銭 × 9500 ÷ 10^4 | **出る**（割り算） | **あり** |

つまり「整数→小数→整数」の往復が問題なのではなく、
**比率（掛率・税率）の適用が割り算を含むため、最小単位の更に下に端数が出る** のが本質。

### 3. 見積ドメインでは丸めが 2 段階

設計書 §8.1 / §8.3 を整数演算に落とすと、丸めは 2 つの単位に対して別々に発生する:

| 段階 | 丸める単位 | 担当メソッド | いつ |
|---|---|---|---|
| ① 銭未満の処理 | 銭（minor unit, 小数2桁） | `applyRate` 内の `Math.trunc` | 比率演算の途中で必ず |
| ② 円未満の処理 | 円（major unit） | `truncateToMajorUnit` / `ceilToMajorUnit` / `roundToMajorUnit` | §8.1「1円未満切捨」、§8.3 税の選択式丸め |

§8.1 の「掛率適用後金額 = 基本金額 × 掛率（端数切捨）」は、コード上では
`baseAmount.applyRate(rate.numerator, DiscountRate.SCALE).truncateToMajorUnit()`
という **2 段階の丸めの連鎖** で表現される。①で銭に丸めて、②で円に丸める。

### 4. なぜこの設計が DDD 的にきれいか

- ドメイン層は外部ライブラリ禁止（CLAUDE.md のルール）。decimal.js などは使えない。
  → 整数演算 + Currency による scale 管理で、純粋な TypeScript 標準機能だけで誤差ゼロを実現できる。
- 「丸めをどの単位で・どう行うか」が業務ルール（§8.1 切捨／§8.3 選択式）であり、
  これらは Money（プリミティブ）に丸めメソッドを置き、ポリシー／値オブジェクト（TaxRoundingType）が呼び分ける形にすると、
  業務ルールが「どの丸めを使うか」の選択として表現できる。

### 5. コード例（実際の Money の挙動）

```typescript
// 100,000円 × 掛率 0.95 = 95,000円
Money.fromMajorUnits(100_000)
  .applyRate(9500, 4)       // ① 銭未満切捨: 950,000,000 銭
  .truncateToMajorUnit();   // ② 円未満切捨: 950,000,000 銭（変化なし）
// → 95,000円

// 端数の例: 286.42円 × 0.95 = 272.099円
Money.fromMinorUnits(28_642)
  .applyRate(9500, 4);      // ① 27,209.9 銭 → trunc → 27,209 銭 = 272.09円
// majorUnits は 272.09（円未満切捨は別途呼ぶ）
```

## まとめ（暗記用）

- 整数演算自体に丸めは要らない。
- 丸めが要るのは **割り算を含む演算（比率適用）** をしたとき。
- 見積ドメインの丸めは **銭単位** と **円単位** の 2 段階。
- 「戻すとき」ではなく「割るとき」が境界線。
