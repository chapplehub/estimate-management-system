# ディレクトリ構造の比較: サブドメイン優先 vs レイヤー優先

## 概要

DDDでバックエンドを実装する際のディレクトリ構造について、「サブドメイン優先（縦割り）」と「レイヤー優先（横割り）」を比較検討した。

## 1. サブドメイン優先（縦割り）構造

```
src/
├── subdomains/
│   ├── estimate/
│   │   ├── domain/
│   │   ├── application/
│   │   ├── infrastructure/
│   │   └── presentation/
│   ├── employee/
│   │   ├── domain/
│   │   ├── application/
│   │   ├── infrastructure/
│   │   └── presentation/
│   └── shared-kernel/
└── shared/
```

### メリット

- **高凝集・低結合**: 1つのサブドメインに関する全てのコードが近接
- **チーム分割しやすい**: サブドメインごとに担当を分けやすい
- **マイクロサービス化への移行が容易**: サブドメインごと切り出せる
- **コンテキスト境界が明確**: 各サブドメインの独立性が視覚的にわかる
- **依存関係の管理**: サブドメイン間の依存を検知しやすい
- **スケールしやすい**: ドメインが増えても構造が崩れにくい

### デメリット

- **レイヤー間の依存関係が見えにくい**: domainがinfrastructureに依存していないか確認しづらい
- **共通コードの重複リスク**: 各サブドメインで似たようなコードが生まれがち
- **初期のオーバーヘッド**: サブドメインが1-2個の段階では過剰な構造
- **横断的な変更が面倒**: 全サブドメインのapplication層に影響する変更など
- **学習コスト**: レイヤー構造に慣れた開発者には直感的でない

## 2. レイヤー優先（横割り）構造

```
src/
├── domain/
│   ├── employee/
│   └── estimate/
├── application/
│   ├── employee/
│   └── estimate/
├── infrastructure/
│   ├── employee/
│   └── estimate/
├── presentation/
└── shared/
```

### メリット

- **レイヤーの責務が明確**: どこを見ればビジネスロジックか一目瞭然
- **レイヤー間依存の検証がしやすい**: domainフォルダにinfraのimportがあれば即座に気づく
- **初期開発が楽**: 小規模なうちはシンプルで理解しやすい
- **技術スタック単位の変更**: 例えばPrisma→別のORMへの移行がinfrastructureだけで完結
- **共通パターンの把握**: 全エンティティ、全リポジトリなどが一覧できる
- **学習しやすい**: レイヤードアーキテクチャとして理解しやすい

### デメリット

- **低凝集**: 1つの機能を実装するのに複数ディレクトリを行き来
- **サブドメイン間の境界が曖昧**: employeeとestimateの依存関係が見えづらい
- **スケールしにくい**: ドメインが増えると各レイヤー内が肥大化
- **マイクロサービス化困難**: サブドメインごとに切り出すのが大変
- **チーム分割しづらい**: レイヤーで分けるのは非効率、ドメインで分けると複数フォルダを触る

## 3. 推奨と移行タイミング

### 現在の推奨: レイヤー優先（横割り）

**理由:**
- サブドメインが少ない初期段階ではシンプルで開発しやすい
- DDD学習に適している（レイヤー間依存違反が視覚的にわかりやすい）
- 後でサブドメイン優先に変更することも可能（逆は難しい）

### サブドメイン優先への移行を検討すべきタイミング

以下のような状況になったら移行を検討:
- サブドメインが5個以上になった
- チームが複数に分かれた
- サブドメイン間の依存関係が複雑化してきた
- マイクロサービス化を視野に入れ始めた
