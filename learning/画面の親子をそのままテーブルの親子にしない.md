# 画面の親子をそのままテーブルの親子にしない（表現・ドメイン・永続化の三層分離）

作成日: 2026-06-13

## 概要

見積詳細画面（issue #326）の「セット明細（セット親子行）」をどうモデル化するかを検討する中で得た、スキーマ設計の価値観についての学び。

UI のデザイン（親行 → 子行の入れ子）を見て、それをそのまま DB の入れ子構造（基底テーブル + サブタイプ + 子テーブル）に写してしまった。だが「DB やバックエンドのこと（＝事実と不変条件）」に視点を移すと、明らかに別の形（価格付き末端行を1表に保ち、薄いセットヘッダ衛星 + 所属の交差表）の方が良かった。

**核心の誤り: 「画面に見えた形」で「テーブルの形」を決めてしまった。見た目の親子 ≠ 格納の親子。**

親子が「ある」と気づいたのは正しい。誤りは「親子がある → だから入れ子テーブル」と一足飛びに構造へ写したこと。親子という事実は本物でも、その正しい表現は層ごとに違う。

## 詳細

### いちばん大事な視点：同じ「親子」が3つの層で3つの違う形を取るのが正常

| 層 | 最適化する力 | セット親子の表現 |
|---|---|---|
| 表現（UI） | 操作しやすさ | 入れ子ツリー（親行の下に子行をインデント表示） |
| ドメイン | 不変条件・振る舞い | 不変条件を持つ集約（ネスト禁止・連続配置・親はメモ以外不変 をドメイン層で担保） |
| 永続化（DB） | NULL なき事実 | 正規化された関係（価格付き行・群ヘッダ・所属、を別々の事実として保存） |

1つの層の形を別の層に漏らさないのが設計の規律。**デザインツリーは UI の成果物であって、それに DDL を選ばせてはいけない。**

### 持つべきだった価値観（優先順）

1. **三層を別物として設計する（表現／ドメイン／永続化）**
   見たデザイン（入れ子）は「UI の形」。それは永続化の形の候補ですらない。

2. **「事実の種類」で考える（正規化の原点）**
   表は1種類の事実を記録する。「画面に何が映るか」ではなく「どんな事実を保存するか」を問う。
   セット明細での事実は3つ：(a) 価格付き行が在る (b) セット群が在る (c) ある行がある群に属す。
   視覚ツリーはこれを1つに融合していた。3つに割ること自体が正規化。

3. **列で割る、分類で割らない（既存フレーム `schema-table-decomposition-framework`）**
   武器は既に持っていたのに抜かなかった。「3種別（通常/親/子）」を見て3つの表役割へ反射した。
   列で見れば即座に出た：
   - セット親 = 価格列（単価・数量・掛率・値引・各金額）が全滅 → 割る（kawasima ルール2「適用不能 → 排他的サブタイプ表」）
   - 構成子 = 通常明細と列が完全一致 → 割らない（観点4「共通列の N 表複製は変更コスト・ドリフト ×N」）

4. **同一性を問う（フレーム観点3）**
   「構成子行は"新種"か、それとも"バッジを付けた同じ生き物"か？」
   振る舞い（自分で価格計算する）は通常明細と同一。同じ生き物 +「セット X に属す」というバッジ（＝関係）であって、新種（＝新テーブル）ではない。

5. **導出を構造にしない**
   「親の金額 = 子の合計」は計算であって、格納すべき包含関係ではない。合計を保持するために表の木を建てない（既存方針「導出可能な値は列で持たない」）。

6. **構造は対価を払うか問う（ADR-0043 の教訓）**
   表・FK・階層を1つ増やすたび、計測可能な対価（多態参照・型横断クエリ・制約）を要求する。
   取引先で CTI を「対価ゼロ」と剥がしたばかり。同じテストを明細に当てれば、基底 + サブタイプは対価ゼロ → 却下、と自分で出せた。
   さらに決定打：`RevisedEstimateItemDetail`（改訂明細の粗利スナップショット）は価格付き末端行にぶら下がる。通常明細と構成明細を別表に割ると、この衛星の FK 先が2つに割れて多態参照になる ―― ADR-0043 が「CTI を正当化できない」とした典型理由の逆を招く。→ **価格付き末端行は1表に保つべき強い根拠。**

### 具体例：見積セット明細の3案

| 案 | 形 | 評価 |
|---|---|---|
| Shape ① | 単一表 `estimate_items` + `kind` 判別子 + `parentItemId` | 実装は最単純だが、セット親に無意味な価格0列が並び kawasima ルール2 に反する。「別概念を kind で1表にまとめない」方針にも反 |
| Shape ② | 基底 `estimate_items` + `estimate_normal_items`/`estimate_set_items` + 子表（最初に飛びついた案） | **CTI 再導入（ADR-0043 と衝突）・価格列重複・RevisedDetail 多態参照**。NG |
| Shape ③-a | **採用**: `estimate_items`=通常+構成を同居（列不変・RevisedDetail 単一FK維持）／`estimate_set_items`=価格なし・金額/位置は導出の薄いヘッダ衛星／所属は交差表 `estimate_set_components`（行の存在で表現・NULLゼロ） | フレーム・kawasima・ADR-0043 と全整合。NULL ゼロ |

セット親の「金額=子合計」「位置=構成子の最小 sortOrder」はすべて導出。永続化する派生列がゼロなので ADR-0033 とも素直。ネスト禁止・連続配置などはドメイン層（Estimate 集約）が担保（観点5）。

### 次に「親子が見える」設計に出会ったときのチェックリスト

1. これは**どの層**の親子か？（UI / ドメイン / DB ―― 混ぜていないか）
2. **どんな事実**を保存する？（ものの存在／群の存在／所属、を分けたか）
3. 種別ごとに**どの列が適用不能**か？（列で割る、分類で割らない）
4. これは**新種か、バッジ付きの同種**か？（同一性）
5. それは**保存すべき関係か、導出か**？
6. この構造は**何を買っている**か？（対価ゼロなら却下）

要は **「画面を一度忘れて、事実だけを並べ直す」**。
