# 重複を許すか結合を入れるか — 4つの負債とドリフト危険

作成日: 2026-06-25

## 概要

「似ているコードを共有ヘルパに畳むべきか、重複のまま残すべきか」を判断する指針。
販売単価3層（共通・得意先別・納品先別）の時点解決 QueryService（#459）の設計で、
write 側ヘルパ（`appendPeriodRows`）の抽出は正しく、read 側（`@>` 一発引き）の抽出は
不要、と判断した過程から抽出した一般原則。

結論を一言で:
> **コードを減らすこと自体に価値を置かない。重い負債を減らすときだけ、軽い負債（コード重複）を支払う。**
> 重複を消す前に必ず問う —「**その重複は、コードより重い負債を隠し持っているか？**」

## 詳細

### 1. 4つの負債とその序列

参考記事（[Four Liabilities](https://medium.com/zus-health/state-coupling-complexity-code-four-liabilities-8283ef029943)）が示す、
技術的負債の重さの序列:

**State（状態） > Coupling（結合） > Complexity（複雑性） > Code（コード）**

運用ルール（curun1r のコメント）:
- 状態を減らすためなら、結合を増やしてよい
- 結合を減らすためなら、複雑性を増やしてよい
- 状態・結合・複雑性のいずれも増やさないときに限り、コード（重複）を減らしてよい

帰結: **dedup（重複解消）はタダではない。必ず結合か複雑性を対価に払う。**
Sandi Metz "duplication is far cheaper than the wrong abstraction" は、
間違った抽象＝最軽量の Code を消すために、より重い Coupling と Complexity を
背負い込み、負債の総量を増やした状態を指す。

### 2. 「畳むべき重複」の本当の十分条件 = ドリフト危険

rule-of-three は「抽出してよい必要条件」だが「抽出すべき十分条件」ではない。
本当の十分条件は **「ドリフトすると害をなす正しさ／危険を担っているか」**。

「ドリフト」= 同じであるべき複数の写しが、時間とともに食い違っていくこと。
危険を測る軸は、似ているかどうかではなく次の **3条件の AND**:

1. **同期必須性（同じ真実か）** ← 最重要
   その規則が変わったら、すべての写しが一緒に変わらねばならないか。
   1つの業務ルールを複数箇所が実装しているなら、片方だけ変えるのは「バグ」。
2. **不可視性（黙って食い違うか）**
   1つだけ変えたとき、型チェッカ／テストが気づくか。
   コンパイルエラー・テスト赤になるなら「うるさい」ので軽い。
   通ってしまうのに出力だけ静かに間違うなら「黙っている」ので重い。
3. **被害の重さ（食い違うと何が壊れるか）**
   誤った金額・セキュリティ穴・データ破損か、見た目のラベル程度か。

**3つ同時に成立するときだけ危険。** 1つでも欠ければ軽い Code に戻る。

### 3. よくある誤解の位置づけ

- **「業務的にクリティカル」= 条件3（被害の重さ）そのもの。増幅器であって引き金ではない。**
  参照が1箇所なら（＝重複していない）どれだけ重要でもドリフトしない。
  重要度はドリフト時の損失額を吊り上げるだけで、ドリフトを生まない。
- **「複雑で理解が難しい」= 条件2の確率を上げる係数。軸ではない。**
  単純なコードでも条件1+2が揃えば十分危険（境界1文字 `[)` がその例）。
  複雑さは Complexity という別の負債として独立に数える。

### 4. 真の重複 vs 偶然の重複（DRY の原典的理解）

本当に効く軸は条件1「**同じ真実か**」。これは DRY 原典（Hunt & Thomas）の
"every piece of **knowledge** has a single representation" の knowledge＝知識／規則。
**似ているコード ≠ 同じ知識。**

- **真の重複** = 1つの規則の写し（同期必須）→ 畳む
- **偶然の重複** = 今はたまたま同じ形だが、変わる理由が別々 → 畳んではいけない

偶然の重複を1つに畳むのが、まさに "the wrong abstraction"。
独立に変わりたかった2つを結合し、以後フラグ・分岐・パラメータが増殖して泥沼化する。

### 5. このコードベースでの実例

**write の range 生成 = 真の重複 → 畳む（#458 `appendPeriodRows`）**
- 3層が同じ業務概念「適用期間」を実装し、半開 `[)` という1規則を共有（条件1○）
- `[]` に書き間違えても型は通り、テスト不備なら黙って誤った重複判定（条件2○）
- 結果は誤った価格（条件3○）

**read の `@>` = 偶然の重複 → 畳まない（#459 は #448 と対称にインライン SQL）**
- 半開の正しさは SQL ではなく **列定義（schema の `daterange` + EXCLUDE）に single-source 済み**
- `@>` はどんな境界規則の daterange にも正しく動く演算子で、read 側にドリフトしうる規則が乗っていない（条件1が実質ゼロ）
- 似ていても、守るべき真実が既に1箇所にあるので畳む必要なし
- 「短いから／単純だから」ではなく「偶然の重複だから」が正しい理由

### 6. 現場で使う1行の問い

> **「この規則が変わったら、これらの写しのうち**いくつが一緒に変わらねばならず**、
> 1つ変え忘れたら**気づけるか**？」**

- 「全部が一緒に変わる必要があり、変え忘れても気づけない」→ 重い。畳む（3つ揃う前でも）
- 「独立に変わってよい」or「変え忘れたら即落ちる」→ 軽い。何コピーでも放置

### 7. 非対称性（畳む判断は慎重に）

**誤って畳んだ結合をほどく方が、重複を後から畳むより高くつく**
（全呼び出し元を探して分割し、紛れ込んだフラグを解きほぐす必要がある）。
だから迷ったら「規則が本当に同期必須だと証明されるまで待つ」のが賢い。
rule-of-three は回数ルールというより、
**「これは偶然の重複ではなく真の重複だ」という証拠が3回めで揃う**証拠集めの経験則。

### 8. 関連する小さな設計判断もこの序列で切れる

- **共有ヘルパは状態を持たせない**（自由関数）→ 最重量の負債 State を入口で断つ
- **引数を discriminated union にしない** → 型コスト(Complexity)は安いが、ヘルパに
  `switch` を強いて層知識を再集権化（Coupling 新設）。ADR-8tg がドメインで排除した
  判別子を infra に呼び戻す重い取引になる
- **read のキーを parallel-arrays にしない** → 型で守られない順序依存(Coupling)を
  新規導入する。それで消える重い負債が無いので割に合わない
