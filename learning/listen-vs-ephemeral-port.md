# LISTEN ポートとエフェメラルポートの違い

作成日: 2026-03-31

## 概要

通信は常に「ポート対ポート」で成り立つ。受信側（LISTEN）と送信側（エフェメラル）の役割と寿命の違い。

## 通信のペア構造

通信は必ず両方のポートのペアで識別される。

```
Next.js (クライアント側)          PostgreSQL (サーバ側)
┌─────────────────┐              ┌─────────────────┐
│ 送信元ポート:52431│ ──────────→ │ 受信ポート:5432  │
│ (OSが自動割当)   │ ←────────── │                 │
└─────────────────┘              └─────────────────┘
```

## LISTEN ポート（受信側）

- プロセスが指名 or 設定ファイルで指定
- プロセスが動いている間ずっと存在
- ポート番号は固定

## エフェメラルポート（送信側）

- OS が自動で割り当て（Linux では通常 32768〜60999）
- 通信中だけ存在し、終了後に OS に返却
- ポート番号は毎回ランダム

## 比較表

| | 受信側 (LISTEN) | 送信側 (エフェメラル) |
|--|--|--|
| 誰が決める | プロセスが指名 | OS が自動割当 |
| 寿命 | プロセスが動いている間ずっと | 通信中だけ |
| ポート番号 | 固定 | 毎回ランダム |
| `ss -tlnp` に出る | 出る | 出ない |

## ss コマンドのオプションと見える範囲

```bash
# LISTEN（受信待ち）だけ表示
ss -tlnp

# 全状態（送信側も含む、ESTAB 等）
ss -tnp
```

- `-t` → TCP のみ
- `-l` → LISTEN 状態のみ（このフィルタが送信側を除外する原因）
- `-n` → ポート番号を数字で表示
- `-p` → プロセス名も表示

## 補足: ポート枯渇

通信が大量に発生するとエフェメラルポートの範囲を使い切る「ポート枯渇」が起きることがある。
高負荷な本番サーバで稀に発生する障害の原因の一つ。
