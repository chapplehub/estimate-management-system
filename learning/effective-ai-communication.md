# Claude Codeとの効果的なコミュニケーション

## 概要

Claude Codeに質問する際、**質問のコンテクストによって回答の質が大きく変わる**ことがある。
このドキュメントでは、実際に発生した事例から、より良い質問の仕方を学ぶ。

## 実例：hidden inputにIDを含めるべきか？

### 背景

従業員更新フォームで、IDをどのように渡すべきか検討していた：

- **方法A**: `bind()`でクロージャからIDを渡す
- **方法B**: `<input type="hidden" name="id" value={...} />`でIDを渡す

### 1回目の質問（改善の余地あり）

```
いま、formタグにidを持っていないせいでupdateEmployeeにidも渡す仕様になってるんだけど
そもそもformタグにhiddenでidも持たせるべきかな？
```

**Claudeの回答:**
- **選択肢1（現状維持：bind()）を推奨**
- 理由: idがHTMLに露出しない（セキュリティ的に良い）

**問題点:**
- 公式ドキュメントへの言及がない
- セキュリティ観点だけで判断
- フレームワークの設計思想を考慮していない

---

### 2回目の質問（改善後）

```
hidden inputにidを含めてクライアント側で見えることにどんなセキュリティリスクがありますか？

https://ja.react.dev/reference/react-dom/components/form
https://nextjs.org/docs/app/guides/forms#passing-additional-arguments
```

**Claudeの回答:**
- **選択肢2（hidden input）を推奨に変更**
- 理由: React/Next.js公式の推奨パターン
- セキュリティリスクを正確に評価（認証がない現状ではどちらも同じ）
- フレームワークの思想に従うべきという結論

**改善点:**
- 公式ドキュメントを確認
- セキュリティと公式推奨の両面を考慮
- より包括的な回答

---

## なぜ回答が変わったか？

### 1回目の回答の問題

1. **公式ドキュメントを確認しなかった**
   - 「セキュリティ優先」という一般論で回答
   - フレームワークの設計思想を考慮せず

2. **前提条件が不明確**
   - 「認証機能がない現状」という前提を明示しなかった
   - 将来的な変更可能性を示唆しなかった

3. **トレードオフを十分に説明しなかった**
   - 複数の選択肢を示したが、判断基準が不明確

---

### 2回目の回答の改善点

1. **公式ドキュメントを確認**
   - Reactは「hidden input」を推奨していることを確認
   - Next.jsは「bind()」も紹介していることを確認
   - 両方のパターンが有効であることを理解

2. **セキュリティリスクを正確に評価**
   - 「認証がない現状では、どちらも同じリスク」と明示
   - 「認証機能実装時に権限チェックが必要」という将来の対応を示唆

3. **フレームワークの思想を考慮**
   - 「クライアントは信用しない」というReactの設計思想
   - 「サーバー側で必ず検証する」という前提
   - 公式推奨に従う方が、将来的にフレームワークの変更に追従しやすい

---

## より良い質問の仕方

### パターン1: 公式ドキュメントを最初から提示（推奨）

```
いま、formタグにidを持っていないせいでupdateEmployeeにidも渡す仕様になってるんだけど、
Reactの公式ドキュメント（https://ja.react.dev/reference/react-dom/components/form）を見ると
hidden inputにidを含めるパターンが推奨されているように見えます。

Next.jsの公式（https://nextjs.org/docs/app/guides/forms#passing-additional-arguments）だと
bind()でidを渡すパターンも紹介されています。

どちらが推奨ですか？このプロジェクトではどちらを選ぶべきですか？
```

**効果:**
- ✅ 公式ドキュメントを最初から確認できる
- ✅ 「あなたが調べた上で質問している」ことが伝わる
- ✅ AIも公式の意図を考慮した回答をする
- ✅ フレームワークの設計思想を考慮した回答が得られる

---

### パターン2: 「公式の推奨を知りたい」と明示

```
いま、formタグにidを持っていないせいでupdateEmployeeにidも渡す仕様になってるんだけど、
そもそもformタグにhiddenでidも持たせるべきかな？

**React/Next.jsの公式ではどう推奨されていますか？**
```

**効果:**
- ✅ 「公式の推奨を知りたい」という意図が明確
- ✅ AIも公式ドキュメントを確認する動機になる
- ✅ 一般論ではなく、フレームワーク固有の回答が得られる

---

### パターン3: 自分の仮説と懸念を先に述べる

```
いま、formタグにidを持っていないせいでupdateEmployeeにidも渡す仕様になってるんだけど、
hidden inputでidを渡す方がReactの公式パターンに近い気がします。

ただ、セキュリティ的にクライアント側にidを露出させるのは良くない気もします。

どう判断すべきですか？
```

**効果:**
- ✅ あなたの思考プロセスが見える
- ✅ 迷っているポイントが明確
- ✅ AIも「公式 vs セキュリティ」の両面を考慮した回答をしやすい
- ✅ トレードオフを説明しやすい

---

### パターン4: 判断基準を問う

```
いま、formタグにidを持っていないせいでupdateEmployeeにidも渡す仕様になってるんだけど、
hidden inputでidを渡すのと、bind()でidを渡すのでは、
どのような基準で判断すべきですか？
```

**効果:**
- ✅ 二択ではなく、判断基準を聞く
- ✅ 複数の観点（セキュリティ、保守性、公式推奨など）を考慮した回答が得られる
- ✅ 自分で判断するための材料が得られる

---

## 質問のベストプラクティス

### 良い質問の要素

1. **公式ドキュメントのリンクを含める**
   - AIが調べる手間が省ける
   - 議論の土台が明確になる
   - フレームワークの設計思想を考慮した回答が得られる

2. **自分の仮説や懸念を明示する**
   - 「セキュリティが心配」「公式に従いたい」など
   - AIがどの観点で回答すべきか分かる
   - トレードオフを説明しやすい

3. **「〜すべき？」より「どう判断すべき？」**
   - 二択で聞くと、どちらか一方に偏った回答になりやすい
   - 判断基準を聞くと、両面を考慮した回答が得られる
   - 自分で意思決定するための材料が得られる

4. **コンテキストを明示する**
   - プロジェクトの現状（認証機能がない、など）
   - 将来の予定（Auth.js導入予定、など）
   - チームの方針（一貫性を重視、など）

---

### 悪い質問の例

#### 例1: コンテキスト不足

```
idをどうやって渡せばいい？
```

**問題:**
- 何のidか不明
- どこで使うのか不明
- なぜ悩んでいるのか不明

---

#### 例2: 二択で限定しすぎ

```
hiddenとbind()、どっちがいい？
```

**問題:**
- 判断基準が不明
- 他の選択肢があるかもしれない
- AIが一方的に判断してしまう

---

#### 例3: 公式ドキュメントを確認していない

```
Reactでformにidを渡す方法を教えて
```

**問題:**
- 公式ドキュメントを確認すれば分かる
- AIの回答が一般論になりがち
- プロジェクト固有の判断材料が得られない

---

## AI側の改善点

### 今後AIが意識すべきこと

1. **公式ドキュメントを積極的に確認する**
   - 一般論だけで回答しない
   - フレームワークの設計思想を理解する
   - 「公式ではどう推奨されているか」を常に考慮

2. **複数の選択肢とトレードオフを示す**
   - 「これが正解」と断言しない
   - 判断材料を提供する
   - それぞれのメリット・デメリットを明示

3. **前提条件を確認する**
   - 「認証機能がない現状では」という前提を明示
   - 将来的な変更可能性を示唆
   - コンテキストに応じた回答をする

4. **質問の意図を確認する**
   - 「セキュリティが心配ですか？」
   - 「公式推奨に従いたいですか？」
   - 「一貫性を保ちたいですか？」

## まとめ：理想的な質問の例

```
いま、formタグにidを持っていないせいでupdateEmployeeにidも渡す仕様になってるんだけど、
Reactの公式（https://ja.react.dev/reference/react-dom/components/form）を見ると
hidden inputにidを含めるパターンが推奨されているように見えます。

ただ、クライアント側にidを露出させるのはセキュリティ的に懸念があります。

このプロジェクトでは、公式の推奨パターンに従うべきか、
セキュリティを優先してbind()でidを渡すべきか、どう判断すべきですか？

※ 現状は認証機能がありませんが、将来的にAuth.jsを導入する予定です。
```

**この質問の良い点:**
- ✅ 公式ドキュメントを提示
- ✅ 自分の懸念（セキュリティ）を明示
- ✅ 判断基準を問う形式
- ✅ プロジェクトのコンテキスト（認証機能の予定）を共有
- ✅ 「公式 vs セキュリティ」の両面を考慮した回答が得られる

## 実際の適用例

### このプロジェクトでの結論

最終的に**hidden inputにIDを含める（公式推奨パターン）**を採用した。

**理由:**
1. React/Next.jsの公式推奨パターン
2. 認証機能がない現状では、セキュリティリスクは同じ
3. Auth.js導入時に権限チェックを実装すれば安全
4. フレームワークの設計思想（クライアントは信用しない）に従う
5. コードがシンプル（ラッパー関数が不要）

**実装:**
```tsx
// Server Action
export async function updateEmployee(_prevState, formData: FormData) {
  const id = formData.get("id") as string;
  // TODO: Auth.js導入後に権限チェックを追加
}

// Client Component
<form action={updateEmployee}>
  <input type="hidden" name="id" value={employee.id} />
  <input name="name" defaultValue={employee.name} />
</form>
```
