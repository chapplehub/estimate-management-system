# 共通化の可否は「新たな構造を生むか」で決まる — 畳むコスト側の軸

作成日: 2026-06-26

## 概要

「似ているものを1つに畳むか」を決めるとき、便益（ドリフト危険を消せるか）だけ見ては
片手落ちになる。**畳むこと自体のコストは対象によって桁違いに変わる**からだ。

販売単価3層の時点解決（#459）の設計で、同じ会話の中で正反対の2判断をした:

- **read SQL ロジックは共通化しなかった** — 畳むと「共有ヘルパ」という新構造と、その引数で
  「層をどう判別するか（parallel-arrays か discriminated union か）」という**新たな判断点**が生まれるから。
- **解決結果 DTO は共通化した** — 畳んでも `{ sellingPrice: string }` という
  キーも分岐軸も持たない byte-identical 型を1つの名前に畳むだけで、**新しい構造も判断点も生まれない**から。

一言で:
> **共通化は「便益 − 自身が生む構造」の差し引きで決める。**
> 便益（消えるドリフト危険）がゼロでも、コスト（生む構造）もゼロなら畳んでよい。
> 便益があっても、コスト（新構造・新判断点・再集権化される知識）が上回るなら畳まない。

[[duplication-vs-coupling-four-liabilities]] が「畳む**便益**＝ドリフト危険があるか」を測る軸なら、
本メモは「畳む**コスト**＝自身が構造を生むか」を測る対の軸。両方を引き算して初めて判断が立つ。

## 詳細

### 1. 共通化のコストは「生まれる構造」で測る

DRY は「コードを減らす」操作に見えるが、実際には**コードを減らす代わりに別の何かを増やす**取引である
（[[duplication-vs-coupling-four-liabilities]] の負債序列 State > Coupling > Complexity > Code）。
増える「別の何か」が、畳むコストの正体:

| 共通化対象 | 生まれる構造 | 新たな判断点 | 再集権化される知識 | コスト |
|---|---|---|---|---|
| read SQL ヘルパ | ヘルパ関数1本＋設定 interface | キー渡しを parallel-arrays か `{column,value}` か DU か | 「得意先/納品先」という層の存在をヘルパが知る | **高** |
| 解決結果 DTO | なし（型名が1つ減るだけ） | なし | なし（DTO はキーを持たない） | **ほぼ0** |

ヘルパは、それ自体が「どう変化を吸収するか」を表現せねばならず、変化軸（層）を
**自分の中にエンコード**する。だから引数設計という判断点が芋づる式に生まれ、最悪は
discriminated union → `switch` → 層知識の再集権化（ADR-8tg がドメインで排除した判別子の infra 復活）まで滑り落ちる。

DTO は変化軸を持たない（キーも層も入っていない、単一フィールドの値の形）。
畳んでも「3つの同一 interface が1つの名前になる」以上のことは起きない。エンコードすべき変化が無いので、判断点も生まれない。

### 2. 「変化軸を持つか」が分水嶺

- **変化軸を持つ共通化**（ロジック・ヘルパ・基底クラス）= 畳むと、その軸をパラメータ／判別子／継承で
  表現する**新構造**が要る。これが Coupling・Complexity を新たに背負う本体。
- **変化軸を持たない共通化**（キーなし DTO・定数・純粋な値の形）= 畳んでも畳む先に表現すべき変化が無い。
  名前が1つに減るだけ。ほぼタダ。

集約を層ごとに分けた ADR-8tg の根拠が「identity（複合自然キー）が層ごとに異なる＝変化軸がある」だったのと表裏。
**同じ issue でも、変化軸を持つもの（集約・ロジック）は分け、持たないもの（DTO）は畳む**、で一貫する。

### 3. なぜ便益軸（four-liabilities）と両方要るか

2軸は独立に効く。4象限で考えると判断が機械的に決まる:

| | コスト低（構造生まない） | コスト高（構造生む） |
|---|---|---|
| **便益高（ドリフト危険あり）** | 迷わず畳む | 畳む（危険 > 構造コスト）例: write `appendPeriodRows` |
| **便益低（偶然の重複）** | 畳んでよい（タダなら名前を減らす）例: 本件 DTO | **畳まない**例: 本件 read SQL ヘルパ |

- read SQL ヘルパは「便益低（`@>` は偶然の重複・半開規則は schema に single-source 済み）× コスト高（新構造＋判別子）」→ 左下…ではなく右下、明確に不可。
- DTO は「便益低（同一型が3つあるだけ）× コスト低（構造を生まない）」→ 右下ではなく左下、畳んでよい。
  **決め手は便益（小さい）ではなくコストの不在だった。** タダなら、わずかな整い（#428 の `override ?? common` が単一型で書ける）でも取りに行ける。

便益軸だけ見ると「DTO も偶然の重複だから畳むな」と誤読しかねない。
コスト軸を併せて初めて「偶然の重複でも、畳むコストがゼロなら畳んでよい」が言える。

### 4. 現場で使う2行の問い

[[duplication-vs-coupling-four-liabilities]] の便益の問いに、コストの問いを足す:

> 便益:「この規則が変わったら、写しは**一緒に変わらねばならず**、変え忘れに**気づけないか**？」
> コスト:「畳んだ先に、**変化軸を吸収する新構造（パラメータ／判別子／継承）が要るか**？要らずに名前が減るだけか？」

- 便益高 → 構造を生んででも畳む
- 便益低 × コストほぼ0 → 畳んでよい（名前を減らす小さな整いのために）
- 便益低 × コスト高 → **絶対に畳まない**（典型的な the wrong abstraction の入口）
