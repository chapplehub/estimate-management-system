# 集約間で「一瞬だけ」関係する不変条件の守り方 — 強整合＝集約内に収める、ではない

作成日: 2026-06-26

## 概要

「強整合（集約内不変条件の強制）を求めるなら、関係するエンティティは*必ず*同じ集約に収めるべき」と考えていたが、これは正しくない。

集約 A・B にまたがる不変条件でも、両者が**ほんの一瞬（特定の状態遷移の瞬間）しか**関係しないなら、B を A に丸ごと取り込む（full-merge）必要はない。整合性境界に入れるべきは「**その不変条件が即時整合を要求する*最小の*データ**」だけ。残りは別アクター・別ライフサイクルとして別集約に置いてよい。

具体題材は issue #440：見積申請の version 関門に残る並行性の窓（別バリエーション二重前進の TOCTOU）。

## 詳細

### 問題の構造

- 不変条件「1見積1前進」（1つの見積が同時に持てる前進バリエーションは高々1つ）は、本来 **Estimate 集約の不変条件**。
- だが判定材料（前進状態）は ADR-0058 により**別集約**（EstimateApplication / EstimateApprovalExemption）の行の存在から導出される。
- 集約が自分の不変条件を、自分の境界の外にある状態で守らねばならない → 集約またぎ協調が必要 → 並行性の窓が生まれる。
- 具体的な窓：`estimateRepository.update()`（version bump）と `applicationRepository.insert()` が**別トランザクション**のため、「version が k+1 に進んだのに申請行がまだ可視でない」一瞬がある。その隙に別バリを Preview した後続が、兄弟チェック（A 未挿入で素通り）も version 関門（新版 k+1 を正当に読んだので通る）も抜けて二重前進する。

### 誤っていた思考と、正しい定石

誤：「強整合が要る ⇒ 必ず集約内に収める（full-merge）」。

正：Vernon『Effective Aggregate Design』の集約設計則が衝突したときの定石。
- **則1**：真の不変条件は整合性境界（集約）の中でモデル化せよ。
- **則2**：集約は小さく設計せよ。
- 則1と則2が衝突したら → 「その不変条件が**即時整合を要求する最小のデータ**だけ」を同居させる。それ以外は別集約にして ID 参照／結果整合で協調する。

### 今回への当てはめ

「1見積1前進」が即時整合を要求するのは「**どのバリが今、前進枠を占有しているか**」という1事実だけ。
承認ステップ・plan・承認者割当・承認/差戻/取下イベントまで同居させる必要はない。それらは**申請という別アクター（承認者）の、submit 後に独立して動くライフサイクル**に属する。

窓が噛むのは **submit の一瞬**だけ。だから取った解は full-merge ではなく **atomic submit（選択肢A）**：

> submit の瞬間だけ version bump と申請 insert を**単一トランザクション**で原子化する。

これで「version が進んだのに行が不可視」の継続時間がゼロになり、どの interleaving でも後続は *version 関門で死ぬ*（stale）か *兄弟チェックで弾かれる*（A が可視）かのどちらかになって窓が消える。

### full-merge を退けた理由（爆風半径と偽競合）

- **ADR の前提崩壊**：申請の分離は ADR-0053/0054/0056/0058/0061/0062/0063 の土台。特に ADR-0061 は進行ロックをアプリ層に置く理由を「判定材料は集約の外」と明記。full-merge は全部の前提を崩す。
- **偽競合（false contention）**：version は集約ルートに1つ（ADR-0039）。申請を見積集約へ入れると、承認者の「バリ A を approve」と作成者の「作成中バリ B を編集」（進行ロックはバリ単位なので B は編集可）が**同じ estimate.version を bump** して衝突する。別アクター・別ライフサイクルを1つのロックに押し込む代償。
- 一瞬の不変条件のために恒久的な結合を買う取引になる。

### 設計選択肢の比較（記録用）

| 案 | 整合性境界に入れるもの | 代償 |
|---|---|---|
| full-merge | 申請＋承認ステップ＋全イベント丸ごと | ADR 6本の前提崩壊・集約肥大・偽競合 |
| slot 同居 | 「前進枠＝advancingVariationId」1事実のみ | 「前進」がスロットと導出ステータスの二重表現に。withdraw/reject 時のスロット解放が別集約またぎの整合義務を生む。nullable 列＝NULL排除方針に逆行 |
| **atomic submit（採用）** | 何も移さない。submit の瞬間だけ bump+insert を1tx | コードベース初の集約またぎ tx。ただし窓は確実に閉じ、更新する ADR は ADR-0068 の tx スタンス1本のみ |
