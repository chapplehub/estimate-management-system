# インデックス設計: 選択性・部分インデックス vs 複合インデックス

作成日: 2026-06-18

> **追記（2026-06-18・同日）**: 本稿の実例だった「`estimate_approval_steps` に `status='AWAITING'` の部分インデックスを張る」案は、その後のグリルで**承認ステップの構造をごっそり変え、status を保存カラムではなく行の存在から導出する**方針に転換したため**前提が覆った**（→ `derive-status-from-event-rows-not-mutable-column.md`）。status カラム自体が無くなったので `WHERE status='AWAITING'` の部分インデックスは張れず、Inboxは反結合（NOT EXISTS）で引く。**ただし本稿の一般論（選択性 vs カーディナリティ、部分 vs 複合、使われないインデックスは負債）はそのまま有効**。具体例が陳腐化しただけで教訓は生きている。

## 概要

DBインデックスを張るかどうか・どう張るかの判断軸を整理した。きっかけは見積申請の承認ステップテーブル（`estimate_approval_steps`）の承認Inboxクエリ（`status='AWAITING'` かつ `role_id ∈ 自分の役割` のステップを引く）のインデックス設計。

当初は `role_id` 単独インデックスの予定だったが、「いつも引く値が固定された希少値」というユースケースから**部分インデックス**が最適という結論に至った。その過程で「カーディナリティ」と「選択性」の違い、部分インデックスと複合インデックスの使い分けを学んだ。

## 詳細

### メリットの本質は「カーディナリティ」ではなく「選択性(selectivity)」

- インデックスの効能は **全行スキャン(Seq Scan)を避けて目的の行へジャンプできる**こと。
- 効くかどうかは「そのクエリが全体の何割まで絞れるか = 選択性」で決まる。
  - 高カーディナリティ列（`id`, `email`）: 1値指定でほぼ1行 → 選択性が高い → 効く。
  - 低カーディナリティ列（`status` のlike 4値）: 1値指定でも数割が該当 → 選択性が低い → 普通は効かない（プランナがあえてSeq Scanを選ぶことも）。
- 「高カーディナリティの組み合わせなら速い」は結論としては当たることが多いが、本質は選択性。

### 低カーディナリティでも「特定の希少値だけを引く」なら部分インデックスが輝く

`status` は4値（NOT_STARTED / AWAITING / APPROVED / REJECTED）で低カーディナリティだが、`AWAITING` という**特定の値だけは希少**（各申請で常に高々1ステップ、残りが大多数）。

- 複合インデックス `(role_id, status)`: キーの組み合わせで絞る。**全行が載る**。
- 部分インデックス `(role_id) WHERE status='AWAITING'`: WHEREが「どの行をインデックスに載せるか」を決める。**AWAITINGの行だけが載る**。

前者は「キーで絞る」、後者は「載せる行を絞る」。希少値しか引かないなら後者がコンパクト＆高速。

```sql
CREATE INDEX idx_approval_step_awaiting
  ON estimate_approval_steps (role_id) WHERE status = 'AWAITING';
```

具体例: 承認ステップ累計10万行・うちAWAITING常時500行の場合
- `(role_id, status)` 複合 → インデックスに10万エントリ
- `(role_id) WHERE status='AWAITING'` 部分 → インデックスに500エントリだけ。Inboxはこの500件の中だけを見る

### デメリット（インデックス共通）と部分インデックスによる緩和

1. **書き込みコスト**: INSERT/UPDATE/DELETE で対象列が変わるとB-treeの張り替えが発生（書き込みが遅くなる）
2. **ストレージ**: インデックスは実体を持つので容量を食う
3. **プランナのオーバーヘッド**: 候補が増えると実行計画選択が少し重くなる（通常は誤差）

部分インデックスはこれらを縮小する。AWAITING行しか載らないので小さく、`AWAITING → APPROVED` 遷移の瞬間にエントリが**自動的にインデックスから外れる**（WHEN条件を外れるため）。「現在承認待ち」集合だけを常に小さく保てる。

### 部分インデックスを複数 vs 複合インデックス1本

「NOT_STARTED用も追加するか?」という問いに対する実務判断:

- **1つの希少値だけ引く** → 部分インデックス
- **複数の status 値を満遍なく引く** → 複合インデックス `(role_id, status)` 1本
  - 複合なら `role_id` 単独（左端プレフィックス）でも `role_id + 任意のstatus` でも1本で賄える
  - 部分インデックスは1値ごとに1本必要。3値引くなら3本…と増え、書き込みコストもストレージも嵩む

さらに「そもそもそのクエリが存在するか」を問うことが重要:
- NOT_STARTED = 前ステップ待ちでアクション不要 → Inboxに出さない
- APPROVED/REJECTED履歴 = 申請を開いて経緯を見る = `application_id` で引く。roleで横断検索しない

→ 使われないインデックスは「死蔵インデックス」になり書き込みを遅くするだけの負債。**ユースケースが無いインデックスは作らない**。

## 教訓

- インデックスは「カーディナリティ」で機械的に決めず、「**実際のクエリの選択性**」と「**そのクエリが本当に存在するか**」で判断する
- 低カーディナリティ列でも「いつも引く値が1つの希少値に固定」なら部分インデックスが最適
- 「1つの希少値」=部分インデックス / 「複数値を満遍なく」=複合インデックス
- 使われないインデックスは負債。ホットパスのクエリにだけ張る
