# スキーマ分解の判断フレーム（分類数でテーブルを割らない）

作成日: 2026-05-19

## 概要

「提出先区分(2) × 見積区分(3) = 6 分類あるなら、見積・見積バリエーション・見積明細も 6 つずつ作るべきか？」という問いから導かれた、テーブル分解の判断フレーム。

結論は **No**。分類の数でテーブルの数を決めてはいけない。テーブルを割る単位は「分類」ではなく「列」であり、その判断には大前提＋5 つの観点を用いる。`docs/business/estimate/システム設計書(見積).md` §11 の NULL 排除作業の中で、`EstimateItem.deliveryPrice` をどう扱うか（子テーブル化 vs サブタイプ分割 vs 6分割）を俯瞰して検討する過程で確立した。

## 詳細

### 大前提（土台）

> テーブルを割る単位は「分類」ではなく「列」。分類が N 個あっても表が N 個になるわけではない。kawasima のデシジョンツリー（NULL 排除方針）は *分類ごと* ではなく *NULL になりうる列ごと* に当てる。
>
> 問いは「この区分は表を分けるべきか？」ではなく
> **「この列は、全組合せのうちどの部分集合で適用不能になるか？」**

この設計書自体がその証拠になっている：見積区分は 3 値あるが `Estimate` を 3 表にはしていない。`scheduledRepairDate`（REPAIR のみ）/ `actualRepairDate`・`emergencyReason`（AFTER_REPAIR のみ）という固有列が分岐する所だけ衛星表（`RepairEstimateDetail` / `AfterRepairEstimateDetail`）に出している。判断は常に *列* 単位で行われていて、*分類* 単位ではない。

### 5 つの観点

| # | 観点 | 問い | 出力（どう構造化するか） |
|---|---|---|---|
| 1 | **構造差 vs 値・ルール差** | その区分で「列構成」が変わるか、「取りうる値・制約・遷移」だけが変わるか | 列構成が変わる → サブタイプ/衛星表。値・ルールだけ → 1 表＋discriminator＋ドメイン層 |
| 2 | **直交軸を掛け算しない**（最重要） | 2 つの区分は独立した軸か | 独立なら、各軸の衛星を共通核に *独立に* ぶら下げる。N×M の積表は軸が絡まり変更波及が爆発する |
| 3 | **アイデンティティ同一性** | 別の *物* か、同じ物の別 *フレーバー* か | 別物 → 並行表階層。同じ物＋付帯情報 → 共通核＋衛星 |
| 4 | **薄い共通核／小さな衛星** | 共通列を N 表に複製していないか | 複製は変更コスト・スキーマドリフト risk が ×N。**「DB 構造変更の波及」はテーブルを増やす方が最大化し、薄い核＋衛星が最小化する**（直感と逆向き。ここが本質） |
| 5 | **ルールはスキーマでなくドメイン層** | 区分駆動の不変条件・排他を表トポロジで表現していないか | ルール → ドメイン層で担保（ADR-0019）。テーブルは構造のみを表す |

### 適用手順

1. NULL になりうる列を全部列挙する
2. 各列に観点 1：構造差か、値・ルール差か
3. 構造差の列だけ、観点 2〜3：どの軸の衛星か／別アイデンティティか
4. 観点 4：共通列は核に一枚、固有列だけを小さな衛星に
5. 残った区分駆動の制約は観点 5：ドメイン層へ

### この設計書での実適用結果

6 分類空間（提出先 2 × 見積区分 3）に上記を当てた結果、**構造的に未決だったのは `EstimateItem.deliveryPrice` ただ 1 列だけ**だった。残りはすべて「共通核＋的を絞った衛星」で既に正しく解けていた（修理 → `RepairEstimateDetail` 等、系譜 → `EstimateVariationCopy/Revision`、受注確定/取消 → `OrderConfirmation/Cancellation`）。

判断軸が「6 をどう割るか」から「`deliveryPrice` 1 列が将来独りか／仲間が増えるか」に縮み、最終的に役割で命名したサブタイプ衛星 `RevisedEstimateItemDetail` に着地した（価格専用名 `RevisionItemPrice` は将来拡張で名前と内容が乖離するため不採用）。

### 補足: なぜ観点 2 が最重要か

このフレームの本質は「**分類は *データ* の性質、テーブルは *構造* の決定であり、両者は別レイヤー**」という分離にある。6 という数字は提出区分×見積区分という *データの組合せ* の話で、構造（テーブル）に直結させてはいけない。観点 2（直交軸を掛け算しない）が最重要なのはこのためで、残り 4 観点は「では構造側はどう決めるか」の具体化になっている。

## サブ判定: 親抽出 (CTI) vs 重複 (TPC)

「テーブルを分ける」と決まった後の **一段下のサブ問題**: 共通項目を親テーブルにくくり出すか、親無しで各サブタイプに重複させるか。`RepairEstimateDetail` と `AfterRepairEstimateDetail` が共通列（`estimateId`, `targetProductId`, `faultDescription`）を共有しているのに親を抽出せず重複させている判断の根拠になる。

### まず 3 パターンを区別する

ORM 文献のサブタイプ継承マッピングには 3 つの型がある：

- **STI (単一テーブル継承)**: 1 表＋discriminator 列＋サブタイプ固有列を nullable で同居。→ **NULL を生む**ので本プロジェクトの NULL 排除方針では原則却下。
- **CTI (親子分離)**: 親表＝共通列のみ／子表＝固有列のみ、親 ⇄ 子は 1:1 FK。→ **NULL は生まない**。共通くくり出し型。
- **TPC (各サブタイプ独立／重複)**: 親無し、各サブタイプが共通列含めて自前で持つ。→ **NULL は生まない**が共通列が複製される。`RepairEstimateDetail` 等はこちら。

**重要な精緻化**: §11.1.1 の設計判断は「親に同居すると NULL が残る」と書いているが、これは厳密には STI 却下の論理であって CTI まで否定する強い理由にはなっていない。CTI なら NULL は出ない。修理を TPC にした本当の決め手は「NULL」ではなく、以下の基準群の総和。

### 判断基準（TPC vs CTI）

| # | 基準 | TPC（重複）に倒す側 | CTI（親抽出）に倒す側 |
|---|---|---|---|
| 1 | **共通列の数** | 少ない（〜3-5列） | 多い（5列以上、特に重い関係列を含む） |
| 2 | **共通列の安定性** | 共通列はもう増えない | 共通列が今後も育つ見込み |
| 3 | **共通エンティティとしての意味** | サブタイプは業務上「別物」。共通の名前で扱う場面が無い | 共通の「物」として他テーブルから参照したい／共通名でクエリしたい |
| 4 | **サブタイプ数の安定性** | 固定（2-3 で打ち止め） | 増える見込み（新サブタイプを軽く追加したい） |
| 5 | **共通振る舞い・共通制約** | 個別に書いて支障無い | 共通バリデーション・共通インデックス・共通ライフサイクルを一箇所に集約したい |
| 6 | **挿入/更新の単純さ** | 1 INSERT で完結したい | 2 段 INSERT・カスケードを許容できる |
| 7 | **読みやすさ** | スキーマをフラットに見せたい | 多態構造として整理した方が読みやすい |
| 8 | **DB レベル整合性** | 排他はドメイン層任せでよい | 親 `@unique` で「1 親に最大 1 件」を DB で言いたい |

### シンプル版の意思決定

- **デフォルトは TPC（重複）**: 共通列が少なく、サブタイプが業務上別物として扱われ、共通クエリ・共通参照ニーズが薄いなら迷わず重複。フラットで INSERT もドメイン層も読みやすい。
- **CTI に倒すのは**: 基準 1・3・4・5 の複数が当てはまるとき（特に「共通列が多い」「共通エンティティを外から参照したい」「サブタイプが今後増える」のいずれか）。
- **STI は本プロジェクトでは選ばない**: NULL 排除方針と矛盾。

### 修理サブタイプへの当てはめ（TPC が正解だった理由）

| 基準 | 修理の実態 | 押す方向 |
|---|---|---|
| 1 共通列数 | `estimateId, targetProductId, faultDescription` の 3 列のみ | TPC |
| 2 安定性 | 共通列は変わる見込み無し | TPC |
| 3 共通エンティティ意味 | REPAIR と AFTER_REPAIR は採番接頭辞・タイトル・必須項目セット（§6）で業務上明確に別物。「RepairInfo」として外から参照する場面は皆無 | TPC |
| 4 サブタイプ数 | `EstimateType` 3 値で固定 | TPC |
| 5 共通振る舞い | `targetProductId` index は両表に書けば良い。共通バリデーションも単純 | TPC |
| 6 INSERT 単純さ | 1 INSERT で済む | TPC |
| 7 読みやすさ | 「これが REPAIR 詳細・これが AFTER_REPAIR 詳細」と完結 | TPC |
| 8 DB 整合 | 子側 `estimateId @unique` で同等を表現可能。両表跨ぎの排他は ADR-0019 でドメイン層担保 — TPC/CTI 同等 | 中立 |

**8 項目中 7 項目が TPC を支持**。重複は 3 列だけで、その代わり業務上の別物性とフラットな読みやすさを得ている。「分化させた方が見やすい」という直感は基準 7・3 が効いた結果で、根拠がある。

### CTI に倒したくなる仮想シナリオ

修理詳細が将来こう育ったら CTI を再検討する：

- 共通列が 10 列規模に増える（修理対象機器の付帯情報、保証区分、サポートチケットID、修理オーダー No…）
- 「修理見積詳細」を別エンティティ（修理ワークフロー、部品手配など）が外部から参照する
- サブタイプが 4-5 種に増える（緊急修理／予防保全／オンサイト／センドバック…）

このいずれも今は無いので TPC のままで OK。
