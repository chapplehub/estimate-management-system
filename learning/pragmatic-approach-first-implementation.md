# プラグマティックアプローチで初実装

## 概要

アーキテクチャ議論に時間を取られ開発が進まない状態から、「まず動くものを作る」プラグマティックアプローチに切り替えて、20分で従業員管理画面を実装した記録。

## 背景：陥っていた罠

### 問題

- DDDの勉強のためにプロジェクトを開始
- バックエンド（Domain/Application/Infrastructure層）は95点の完成度
- **しかしUIは0%**で、実際に使える機能が何もない
- ディレクトリ構造やアーキテクチャばかり気になって開発が進まない
- 「完璧な設計」を追求しすぎて実装に入れない

### 典型的な「DDD沼」「アーキテクチャ沼」

学習プロジェクトでよくある失敗パターン：
- 設計に時間をかけすぎる
- 動くものを作る前に完璧を目指す
- アーキテクチャ議論が目的化する
- ユーザー価値を生まない

## 解決策：プラグマティックアプローチ

### 方針転換

```
❌ 理想主義アプローチ
「完璧なアーキテクチャを設計してから実装しよう」
→ アーキテクチャ議論が永遠に終わらない

✅ プラグマティックアプローチ
「最小限のアーキテクチャで動かして、必要に応じてリファクタリング」
→ 学びながら進化させる
```

### 実践したルール

1. **アーキテクチャ議論を一時凍結**（1ヶ月間）
2. **まず動くものを作る**（2週間スプリント）
3. **既存のsubdomains/を活用**（せっかく作ったので捨てない）
4. **認証は後回し**（まずCRUD機能を完成させる）
5. **UIは最低限**（機能優先、デザインは後）

## 実装内容（20分で完成）

### 作ったもの

**ファイル:** `web/src/app/employee/page.tsx`

**機能:**
- ✅ 従業員登録フォーム（Server Actions使用）
- ✅ 従業員一覧表示（テーブル）
- ✅ subdomains/のビジネスロジック活用
- ✅ パスワードハッシュ化（bcrypt）
- ✅ バリデーション（HTML5 + ドメイン層）

### アーキテクチャ

```
app/employee/page.tsx (Presentation)
  ↓ Server Actions
subdomains/employee/commands/CreateEmployeeCommand (Application)
  ↓
subdomains/employee/entities/Employee (Domain)
  ↓
subdomains/employee/infra/prisma/PrismaEmployeeRepository (Infrastructure)
```

**重要な判断:**
- `app/actions/`を作らず、page.tsx内にServer Actionを直接定義（シンプル）
- subdomains/のコマンド・クエリを直接呼び出し（BFFパターン）
- 型安全性は保ちつつ、過度な抽象化は避けた

### 使った技術

- **Server Actions**: Next.js 16のネイティブ機能
- **CQRS**: 既存のCommand/Query構造を活用
- **bcrypt**: パスワードハッシュ化
- **TailwindCSS**: 最小限のスタイリング

## 学んだこと

### 1. 「完璧な設計」より「動くコード」

DDD学習プロジェクトであっても、**動くものがないと学びが少ない**：
- 設計書だけでは実際の問題に気づけない
- 動かしてみて初めてわかることが多い
- ユーザー価値を生むことでモチベーションが保てる

### 2. 既存のアーキテクチャを活用

すでに95点のバックエンド基盤があるなら、**それを活用する**：
- subdomains/のコマンド・クエリをそのまま使う
- アーキテクチャを再設計しない
- Presentation層を薄く保つ

### 3. プラグマティックでもDDDは守れる

今回の実装でも以下は守られている：
- ✅ ビジネスロジックはDomain層（Employee.create()）
- ✅ データ取得はQuery Service（CQRS）
- ✅ バリデーションはValue Object（MailAddress, EmployeeCd）
- ✅ 依存性逆転の原則（IEmployeeRepository）

**つまり：シンプルに書いても、DDDの本質は保てる**

### 4. 80%の品質で100%動くものを作る

完璧主義を捨てる：
- UIデザイン: 最低限（後で改善できる）
- エラーハンドリング: console.error（後で改善できる）
- 認証: なし（後で追加できる）

**まず動かして、使ってみて、改善する**

**まず完成させる。議論はその後。**

## まとめ

### Before（アーキテクチャ沼）

- バックエンド: 95点
- UI: 0%
- 動くもの: なし
- モチベーション: 低下

### After（プラグマティックアプローチ）

- バックエンド: 95点（維持）
- UI: 40%（CRUD画面の基礎）
- 動くもの: あり！
- モチベーション: 回復

### 重要な教訓

> 「完璧な設計」を追求するより、「動くもの」を作って改善する方が、結果的に良い設計にたどり着く

**プラグマティズム = 実用主義**
- 理想と現実のバランスを取る
- 目的（DDDを学ぶ、システムを完成させる）を見失わない
- 小さく作って、早く学ぶ
