# Issue #474 実装の計画からの逸脱記録

## 逸脱1: E2E の検証対象を「一覧の現在有効単価が新価格に更新」から「詳細画面の旧行終了＋新行連続」へ変更

- **元の計画内容**（price-revision-wizard.md Step 5）:
  「改定実行→旧行が改定日で終了＋改定日開始の新行が連続（重複エラーなし）＋**一覧の現在有効単価が新価格に更新**、を検証」
- **実際の実装内容**:
  一覧の現在有効単価の更新は検証せず、詳細画面で「旧行（現在有効）が改定日で終了し単価据え置き ¥4,000」「改定日開始の新行（将来）が ¥5,000・無期限で連続」「2期間が並ぶ」ことを検証した。
- **逸脱の理由**:
  改定日は `endDatePeriod` の温度ガード（`endDate > today`）を継承するため必ず**明日以降**であり、当日改定は構造的に不可能。したがって改定直後の一覧「現在有効単価」は依然として旧価格（改定日が来るまで旧行が現在有効）であり、新価格は改定日に発効する。計画の「一覧の現在有効単価が新価格に更新」は当日発効を暗黙の前提とした不正確な記述だった。実際に検証すべき不変は「旧行が改定日で打ち切られ、新行が接触境界で連続する（重複エラーが出ない）」ことであり、これは詳細画面で直接観測できる。新行（¥5,000）の存在自体が接触境界で重複が起きなかった証拠になる。

## 補足: 計画どおりに実装した主要点（逸脱なし）

- 集約クエリ `currentValidPeriod`、アプリ層単一コマンド `ReviseCommonSellingPricePeriodCommand`（適用終了＋新規追加の合成・単一version）、`revisePeriodSchema`/`revisePeriodAction`、専用 `ReviseForm`、`authorityFor` への `revisable` 追加、active 行への「改定」ボタンはいずれも計画どおり。
- 据え置き（新単価＝現単価）はフォーム・サーバーいずれでも拒否せず許容（計画どおり）。方向ラベルは現単価 props からの client 算出・表示専用（計画どおり）。
