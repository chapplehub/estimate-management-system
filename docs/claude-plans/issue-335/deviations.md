# Issue #335 実装の計画からの逸脱記録

計画ファイル: `activate-deactivate-variation.md`

## 1. 進行ロック拡張点の「no-op であること」のテストを書かなかった

- **元の計画（Step 1 Red）**: 「`DeactivateVariationCommand` の進行ロック拡張点が現状 no-op であること（後日接続時に書き換わる足場）」をテストする。
- **実際の実装**: 専用テストを書かず、代わりに「既に無効なバリエーションを無効化しても成功する（冪等・進行ロック未接続）」という観測可能な振る舞いのテストに置き換えた。
- **逸脱の理由**: private no-op メソッドは観測可能な副作用を持たないため、それ自体を検証するテストは実装詳細への結合になり、`/tdd` スキルが戒める anti-pattern（実装の形だけを assert する crap test）に該当する。S7 時点では申請テーブルが無く「申請中なら無効化拒否」という振る舞いを構築できないため、ガードが効かない（無効化が表示ステータスにゲートされていない）ことを冪等テストで間接的に担保した。拡張点の存在・契約はコード内コメントと ADR-0061 で記録済み。

## 2. リダイレクト理由の命名

- **元の計画（Step 2）**: 「ACTIVATED / DEACTIVATED を既存 toast 機構に合わせて定義」。
- **実際の実装**: `ESTIMATE_VARIATION_ACTIVATED` / `ESTIMATE_VARIATION_DEACTIVATED` と命名。
- **逸脱の理由**: 既存の `REDIRECT_REASON` は対象エンティティを接頭辞に持つ規約（`PRODUCT_ACTIVATED` 等）。バリエーションは見積の子なので `ESTIMATE_VARIATION_` 接頭辞に揃えた。意味は計画どおり。

## 3. E2E 専用 seed 見積 N9905007 を追加

- **元の計画（Step 4）**: E2E を 1 本追加する（seed 追加の明記なし）。
- **実際の実装**: `seed-estimates.ts` に専用見積 `s7Toggle: "N9905007"`（V1・V2 とも ACTIVE）を新設し、これに対してトグルの往復を検証した。
- **逸脱の理由**: 状態変更 E2E は ADR-0020（状態管理は独立 chain・新規/専用データ）に従い、他ファイルが参照する共有 seed（N9905001 等）を書き換えて干渉させないため専用データに隔離した。chain 終端で両バリを再有効化し seed 状態へ復元するため、再シードなしでも繰り返し実行できる。
