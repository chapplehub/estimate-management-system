# Issue #320 計画からの逸脱記録

Issue 本文および実装計画（move-submission-type-to-variation.md）からの逸脱を記録する。
1〜4 はグリルセッション（/grill-with-docs）で実装前に合意済みの差分、5 は実装中に確定した差分。

## 1. C2 UpdateEstimate は「入力組み替え」ではなく単純削除

- **元の計画（Issue 本文）**: 「C1 / C2 / C3: 入力組み替え（バリエーション記述子ごとに提出区分を持たせる）」
- **実際の実装**: C2 の入力から `submissionType` を削除しただけで、移動先はない
- **逸脱の理由**: C2 はヘッダ専用コマンドでバリエーション記述子を受け取らないため、「記述子ごとに持たせる」が適用不能。提出区分は不変属性（ADR-0045）となり、変更経路自体が存在しなくなった

## 2. C4 UpdateVariation を対象に追加（型レベル除外）

- **元の計画（Issue 本文）**: 対象リストに C4 なし
- **実際の実装**: `VariationContentDescriptor` を `Omit<EstimateVariationDescriptor, "variationNumber" | "submissionType">` に変更し、C4 で提出区分を指定できないことを型で保証
- **逸脱の理由**: 提出区分がバリエーション単位になると C4 が自然な変更地点となるため、「変更できない」ことの明示が必要になった。ミューテータ非提供＋型レベル除外の二重の構造的ガードを採用

## 3. index は移動ではなく廃止

- **元の計画（Issue 本文）**: 「`estimates.submission_type` 列＋index を `estimate_variations` へ移動」
- **実際の実装**: 列のみ移動し、`estimates_submission_type_idx` は DROP（新 index は張らない）
- **逸脱の理由**: 2値 enum の単独 index は選択率が低く実用せず、利用クエリも現存しない。バリエーション絞り込みは常に `estimateId` 起点で既存複合 index で十分（ADR-0045 帰結）

## 4. 設計書の改訂範囲を §7.5 から拡張

- **元の計画（Issue 本文）**: 「設計書 §7.5（データ構造）の記述改訂」のみ
- **実際の実装**: §7.5 に加え §5.3（引き継ぎ表の行移動）・§7.2（矢印表記の是正・用語修正）・§3.5（項目一覧への追加）を改訂
- **逸脱の理由**: §5.3/§7.2 に「見積単位の提出区分」前提の記述が残ると設計書が新モデルと矛盾する。特に §7.2 の「提出区分: 納品先 → 得意先」は不変属性モデルと真っ向から矛盾し、「複製」は CONTEXT.md の Avoid 用語だった

## 5. 実装計画 Step 2〜6 を 1 コミットに統合

- **元の計画（実装計画 Step 2〜6）**: ドメイン層 → アプリ層 → C6 → インフラ層 → docstring を個別コミット
- **実際の実装**: 全レイヤーの組み替えを 1 コミット（99b84d3）で実施
- **逸脱の理由**: pre-commit フックがプロジェクト全体の `tsc --noEmit` を実行するため、必須フィールド追加の型カスケード（記述子 → コマンド → マッパー）を分割するとどの中間コミットもコンパイルが通らない。expand-contract による段階化も検討したが、squash マージされる PR 内では過渡コードの価値がないため「型カスケード = 1 つの意味単位」と判断した
