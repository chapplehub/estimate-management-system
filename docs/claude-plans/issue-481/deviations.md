# Issue #481 実装の逸脱記録

## 1. サーバ側 `verifyAdmin` リダイレクトを E2E 対象外とした（ADR-0020 判断5 からの逸脱）

### 元の計画 / 関連 ADR

ADR-0020（E2E 構成）の権限テスト方針は、認可の検証を「権限のないユーザーが保護操作へ
到達したときの拒否」まで含めて E2E で確認することを基本とする。本 Issue の計画でも当初は
権限テストの一形態として、一般ユーザーがミューテーション系の保護経路へ到達した際の
サーバ側拒否（`verifyAdmin()` → `/signin?reason=forbidden` リダイレクト）を検討対象にしていた。

### 実際の実装

E2E では以下のみを検証し、サーバ側 `verifyAdmin` リダイレクトそのものはテストしなかった。

- 一般ユーザーで一覧・詳細が **閲覧できる**こと。
- 一般ユーザーの詳細で「操作」列ヘッダー（`<th>操作</th>`）と新規追加・各行操作ボタンが
  **一切描画されない**こと（#482 / PR #484）。

（`common-selling-prices-crud.e2e.ts` の `共通販売単価（一般ユーザー・#482）` describe）

### 逸脱の理由

- **UI から不可達**: #482（PR #484）後、本画面は admin 専用ページを持たず、ページ自体は
  全ユーザーが閲覧可能。ミューテーション系 UI（新規追加・編集・適用終了・削除）は
  一般ユーザーには描画されないため、「一般ユーザーが保護操作へ到達する」状態を
  UI 操作で再現できない。テストするには Server Action を直接 POST する必要がある。
- **Server Action 直接 POST は脆い**: アクションの内部呼び出し規約（エンコーディング・
  action id・bind 引数の並び）は Next.js の内部仕様に密結合で、フレームワーク更新で
  容易に壊れる。回帰の安全網としての費用対効果が低く、E2E の趣旨（ユーザー可視挙動の固定）
  からも外れる。
- **正本は別レイヤーで担保**: 認可（`verifyAdmin`）と状態別不変条件（将来行のみ編集削除可、
  現在有効行のみ適用終了可、重複禁止、開始日 ≥ 今日）の検証は、ドメイン集約
  （`CommonSellingPrice`）とアプリ層コマンドの単体テストが正本として担う。E2E は
  「権限に応じた UI 出し分け」と「正常系フロー＋代表的ドメインエラー（重複拒否）の
  ユーザー可視挙動」に責務を限定する。

## 2. 楽観ロック競合・状態別サーバ拒否を E2E 対象外とした

### 元の計画

計画（シードフィクスチャ節）で「楽観ロック競合・状態別サーバ拒否は E2E 対象外」と
既に明記済み。本項は逸脱というより、その判断の根拠を完了時点で確定記録するもの。

### 実際の実装

`common-selling-prices-crud.e2e.ts` 冒頭コメントに対象外である旨と理由を明記し、
テストは作成しなかった。

### 理由

- **楽観ロック競合**: version 不一致を誘発するには同一行に対する同時 2 操作が要り、
  単一ブラウザの直列操作では決定的に再現できない。
- **状態別サーバ拒否**: #482 後は失効/現在有効行の編集・削除ボタン自体が描画されず
  （上記 1 と同根）、UI から不可達。いずれもドメイン集約／コマンドの単体テストが正本。
