# Issue #301: 実装計画からの逸脱記録

## 1. Step 5（クエリ側 DTO への version 追加）・Step 6（フォーム往復）は対象なし

- **元の計画内容**: 見積詳細・編集画面が使うクエリ／DTO に version を含め、編集フォームに hidden input・Zod スキーマ・Server Action の受け渡しを実装する
- **実際の実装内容**: 何も実装しなかった
- **逸脱の理由**: 見積サブドメインにはクエリ（`application/queries/`）もプレゼンテーション層（`src/app` 配下の見積画面）も存在しなかった。C2/C3/C4 は #302 でアプリ層コマンドのみ実装されており、UI は未着手。フォーム往復（ADR-0039 決定3）は見積編集 UI を実装するイシューで、本イシューが確立したコマンド Input の `version` フィールドに対して実装する。計画立案時に UI の存在を確認しなかったことが原因

## 2. 計画外の追加作業: 見積コマンドテストの flaky 解消（e56637d）

- **元の計画内容**: なし（計画外）
- **実際の実装内容**: C1/C2/C3/C4 の4テストファイルが共有していた採番年度 2097 をファイルごとに分離（C1=2097 / C2=2096 / C3=2095 / C4=2094）
- **逸脱の理由**: Step 1 のコミット時に pre-commit フックの全テスト実行で不規則な失敗が顕在化。vitest のファイル並列実行で (1) 互いの beforeEach クリーンアップが相手のデータを削除 (2) 同年度の MAX+1 採番が衝突していた。#302 由来の既存バグであり、本イシューのコミットを安定して通すための前提修正として対応した

## 3. リポジトリ分割を2コミットに分けた（save の一時併存）

- **元の計画内容**: Step 2 でインターフェース分割（save 廃止）、Step 3 で実装、Step 4 でコマンド移行
- **実際の実装内容**: 第1コミット（0d00977）で insert/update を**追加**し save は @deprecated で併存、第2コミット（a1fa79f）でコマンド移行と同時に save を削除
- **逸脱の理由**: pre-commit フックが全テスト GREEN を要求するため、「インターフェースから save を消す」と「全呼び出し元の移行」は原子的に行うしかない。TDD のサイクル（リポジトリの振る舞い確立 → コマンドへの貫通）をコミット境界に対応させるため、移行期 API として save を1コミットだけ残した。最終状態は計画どおり（save は存在しない）
