# Issue #331 実装の逸脱記録

> CLAUDE.md「実装中に計画と異なる対応をした場合、作業完了時に記録」に従う。
> 各項目は {元の計画／前提} / {実際の実装} / {逸脱の理由} で記す。

## 1. 税率（taxRate）を編集不可にした（設計指示書からの逸脱）

- **元の計画／前提**: 設計指示書 `02_差分一覧と本実装方針.md` 129 行は「税率＝編集可」。
  当初の grill でも税率・端数とも編集可で検討していた。
- **実際の実装**: `taxRate` は **常に read-only 表示**とし、C2 入力（`UpdateEstimateInput`）からも
  削除した。フォームでも自由入力欄を設けず「税率は見積年月日から自動決定」と注記する。
  編集可能なのは税端数区分（`taxRoundingType`）のみ。
- **逸脱の理由**: 税率は master（TaxRateMaster）由来で、§8.7 の整合チェックも見積年月日・
  締切日から master 解決した税率で行い、利用者の自由入力値は使わない。自由入力の意味が
  薄く、誤入力が §8.7 と矛盾する余地を生むため。ユーザー判断で編集不可に確定。

## 2. C2（UpdateEstimate）が修理情報の編集も担う（issue 前提の訂正）

- **元の計画／前提**: issue #331 は「C2 実装済み・UI 配線が主」を前提にしていた。
- **実際の実装**: C2 は修理情報に未対応で、集約ルートにも修理情報の一括編集入口が無かった。
  集約ルートに `changeRepairDetail` / `changeAfterRepairDetail` を新設（子 detail へ委譲）し、
  C2 入力へ修理フィールドを追加して estimateType に応じて片方のみ更新するようにした。
- **逸脱の理由**: 修理詳細は Estimate 集約の一部であり、同一画面・同一保存・単一 version
  往復で扱うのが自然（2 コマンドの version 競合を避ける）。「UI 配線が主」という前提が
  実コードと食い違っていたため、必要なバックエンド補修を本 issue に取り込んだ。

## 3. 改訂凍結下の保存欠陥を冪等セッターで修正（ADR-0049）

- **元の計画／前提**: 当初は改訂済み見積の編集可否を UI 都合で扱う案も検討。
- **実際の実装**: C2 は全ヘッダーセッターを無条件に呼ぶため、改訂済み見積では同値でも
  `assertHeaderMutable` で必ず throw し、変更可能なはずの締切日・部署すら保存できない欠陥が
  あった。各ロックセッターを「新値＝現在値なら no-op（assert より前で早期 return）」化した
  （ADR-0049）。DTO に `hasRevision` を足し、UI はロック項目を disabled 表示（二重防御）。
- **逸脱の理由**: 同値設定は「変更」ではないため no-op が不変条件の意図に忠実。フォームが
  全項目を送る単純な配線を保てる。副次効果として同値税率の無駄な全バリ再計算も消える。

## 4. E2E 用 seed に消費税率マスタ投入を追加（実装中に判明した前提）

- **元の計画／前提**: 計画 Step 7 は seed に通常／改訂済み／REPAIR の見積を保証するとした。
- **実際の実装**: `seed-e2e.ts` に TaxRateMaster（8%/10%）の投入を追加した。
- **逸脱の理由**: e2e 実行で C2 更新が「適用税率なし」で throw することが判明。§8.7 の保存時
  税率一致チェックが TaxRateMaster を参照するが、e2e seed（dev seed と別系統）はこれを
  投入していなかった。見積編集 E2E の前提として必須のため追加した。

## 5. セット群編集は本 issue のスコープ外（先送りの明示）

- **元の計画／前提**: ④〜⑨ 明細・バリ操作はスコープ外（計画 §10）。
- **実際の実装**: 本 issue はヘッダー領域（②③）の編集に限定。④〜⑨（VariationPanel）は
  RSC のまま据え置いた。
- **逸脱の理由**: スコープを 1 画面相当（ヘッダー編集）に絞るため。明細・セット群編集は
  後続スライス（S5 等）で扱う。
