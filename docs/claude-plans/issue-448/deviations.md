# Issue #448 実装計画からの逸脱記録

## 1. コミット粒度を水平分割から TDD 垂直スライスへ変更

- **元の計画内容**: ステップを Step 1（IF＋DTO）／Step 2（Prisma 実装）／Step 3（ラッパ＋factory）／Step 4（統合テスト）の4コミットに水平分割し、テストは最後にまとめて足す想定だった。
- **実際の実装内容**: `/tdd` の垂直スライス方針に従い、以下の3コミットに再編した。
  - `feat: 共通販売単価 時点解決 QueryService（@> で時点直引き・happy-path）` — IF＋DTO＋Prisma 実装＋happy-path テストを1つの動く縦スライスとして commit。
  - `test: 共通販売単価 時点解決の境界日・null 回帰テストを追加` — 半開 `[)` 境界・null・無期限上端の回帰テスト。
  - `feat: 共通販売単価 時点解決のユースケースラッパと factory を追加` — ラッパ＋factory＋ラッパテスト。
- **逸脱の理由**: TDD は「全テスト先書き→全実装」の水平分割を明確なアンチパターンとする。各コミットが常にビルド・テスト可能な状態を保つ垂直スライスの方が、各時点でグリーンを担保でき、想像でなく実挙動に基づいてテストを書ける。実際、`DECIMAL(12,2)` 由来の `::text` 結果（下記2）は実装を走らせて初めて判明した。

## 2. DTO が運ぶ単価文字列は銭2桁付き（"1000.00"）

- **元の計画内容**: 計画の Step 2 例 SQL は `selling_price::text` で `"1000"` のような整数文字列を想定する書き方だった。
- **実際の実装内容**: `selling_price` は `DECIMAL(12,2)`（銭精度）のため `::text` は常に銭2桁を伴う（例: `"1000.00"`, `"1234.56"`）。テスト期待値はこれに合わせた。
- **逸脱の理由**: 設計判断（string 素通し＋消費側 `Money.fromDecimalString`）は不変。DTO は文字列をそのまま運ぶ契約なので scale 2 でも `Money.fromDecimalString` が正しく解釈し、契約は壊れていない。実 DB の列定義に追従しただけで、方針変更ではない。

## 3. factory ファイル名を `pricingQueryFactory.ts` に確定

- **元の計画内容**: Step 3 で「命名は既存規約に合わせる」と保留していた。
- **実際の実装内容**: 既存の `<subdomain>QueryFactory.ts` 規約（`customerQueryFactory.ts` 等）に合わせ `pricingQueryFactory.ts` とした。
- **逸脱の理由**: 計画が規約委譲を明示しており、規約適用の確定であって逸脱ではない（記録のため明記）。
