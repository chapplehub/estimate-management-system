# 共通売単価 保守画面 — ユースケース定義（#429 第一段階 / claude.ai design 受け渡し用）

> 本書は **claude.ai design に画面実装を依頼するための入力**。画面が満たすべき目的・対象・ユースケース・ルールを定義する。視覚デザインと細かなインタラクションは設計者に委ねるが、**ドメインルール（重複禁止・適用終了の意味・最低1期間前提・無効化という語を使わない等）は確定事項**として遵守すること。
>
> 用語は本リポジトリ `CONTEXT.md`「価格」節を正準とする。特に「**無効化(Deactivation)**」「**無効(INACTIVE)**」は Product/バリエーションのフラグ専用の予約語であり、**販売単価の期間には使わない**。

## 1. 目的・背景

価格決定機構（ADR-0064）が参照する販売単価3層（共通／得意先別／納品先別）のうち、**共通売単価**を人が登録・改定・整理するための保守画面。共通売単価が無い商品は価格決定が「解決不能」で見積を作れないため、これをメンテナンスする導線が必須。

本画面は **3層のうち共通層のみ**を対象とする第一段階スライス。原価・上書き2層（得意先別／納品先別）は本画面では扱わない（→ 8章 将来拡張）。

## 2. スコープ

**含む**

- 共通売単価の一覧・登録・編集（適用終了を含む）・削除（未適用期間の訂正のみ）。
- 商品ごとの適用期間（開始／終了）と単価の入力。

**含まない**

- **原価の表示・編集**（原価は別集約・別ライフサイクル。原価移設は #465）。第一段階では原価を画面に一切出さない。
- 得意先別／納品先別販売単価（上書き2層）の保守。
- 価格決定（見積側）の単価固定化・再解決。
- 複数適用期間を扱う原価保守画面。

## 3. 対象データモデル（画面が扱う概念）

共通売単価は **商品ごとに1つの集約**を成し、その中に複数の「適用期間行」を持つ。

| 概念 | 説明 | 制約 |
|---|---|---|
| 商品 | 共通売単価のキー。商品コード・商品名で識別 | 1商品＝1集約 |
| 適用開始日 | 期間の下端（含む） | 必須・`YYYY-MM-DD`・実在日 |
| 適用終了日 | 期間の上端（含まない）。`null` = 無期限 | 任意・開始日より後 |
| 共通売単価 | その期間に適用する売単価（円） | **0円以上**（`SellingUnitPrice` は非負） |

- 適用期間は半開区間 `[開始, 終了)`。「終了日」はその日を**含まない**（例: 終了日 `2026-07-01` なら `2026-06-30` まで有効）。
- ある参照日（見積年月日／現在日）に対し、その日を含む期間行は **高々1件**（重複禁止の帰結）。

## 4. ユースケース

### UC-1 商品横断の一覧
保守担当者が、商品ごとの共通売単価の状況を俯瞰する。

- 1行＝1商品（集約境界に一致）。商品コード・商品名・**現在有効な単価**（今日を含む期間の単価。無ければ「未設定／失効中」）を表示。
- 共通売単価が**未設定の商品**を見つけられること（→ UC-5 / 7章の推奨1）。商品検索・未設定での絞り込みを想定。
- 行を選ぶと UC-2（その商品の期間明細）へ。

### UC-2 商品の適用期間明細（ドリルダウン）
1商品の適用期間行を時系列で並べる。

- 各行＝適用開始日・適用終了日（無期限は「無期限」表示）・単価。開始日昇順。
- 各行の時点状態を区別して見せる: **現在有効**（今日 ∈ 期間）／**失効**（今日 ≥ 終了）／**将来**（今日 < 開始）。※「失効」は派生状態でありフラグではない。
- ここから UC-3（登録）・UC-4（編集）・UC-5（削除）へ。

### UC-3 適用期間の登録
新しい適用期間行を追加する（＝新単価の発効や将来改定の予約）。

- 入力: 適用開始日・適用終了日（任意・無期限可）・単価。
- **既存期間と重複したら拒否**（ドメインが `overlaps` で弾く。サーバ側で必ず検証。クライアント側の事前バリデーションは UX 改善として任意）。
- 典型: 現行の無期限期間に終了日を入れて締め（UC-4）、続けて新期間を発効開始日から登録 ＝ **改定**。

### UC-4 適用期間の編集（適用終了 = end-dating を含む）
適用期間行を変更する。ただし**履歴・時点解決の真実性を守るため、行の時点状態によって編集できる範囲を制限する**（過去を書き換えない）。

- **将来行（今日 < 開始）**: 開始日・終了日・単価の**全項目を編集可**。まだ一度も時点解決の対象になっていないため自由に訂正できる。
- **現在有効行（開始 ≤ 今日 < 終了）**: **終了日の設定（＝適用終了 end-dating）のみ可**。開始日・単価は変更不可。現在の単価を改めたい場合は、今日以降で適用終了し新しい期間を追加する（＝改定）。
- **失効行（今日 ≥ 終了）**: **編集不可**。過去に解決済みの単価・期間は書き換えない。
- **適用終了（end-dating）**: 現在有効行に終了日を設定し、以後の時点解決から外す操作。**独立した「無効化」操作ではなく、編集（終了日設定）の一形態**。過去・現行の行は消さず履歴として残す。
- 編集後も重複禁止は維持（サーバ検証）。
- 楽観ロック: 編集対象の version を読み出し、保存時に突き合わせる（ADR-0039）。競合時は再読込を促す。

> **プロトタイプとの差異**: 現行プロトタイプ（v1.1）は全行を無制限に編集可能だが、これは暫定イメージ。実装（#429/#466）では上記の状態別制限を**ドメイン操作の不変条件**として強制する。

### UC-5 適用期間の削除（訂正のみ）
入力ミスの取り消し。

- **まだ適用が始まっていない未来開始（今日 < 開始）の期間行に限り物理削除**を許す。
- 適用済み・進行中・失効済みの行は**削除しない**（履歴・時点解決の真実性を守る）。これらを止めたい場合は UC-4 の適用終了で行う。

### UC-6（前提の充足支援）最低1期間の担保
「有効商品は共通売単価を最低1期間持つ」前提（#422 論点7）を満たすための導線。詳細は 7章 推奨1。

## 5. ルール・不変条件（確定事項）

1. 同一商品内で適用期間は**重複しない**（ドメイン実装済み。サーバで必ず弾く）。
2. 適用期間は半開区間 `[開始, 終了)`、終了 `null` = 無期限。番兵日付・NULLフラグは使わない（ADR-0067）。
3. 単価は **0円以上**（`SellingUnitPrice` は非負）。マスタ入力では **0円を許容する** ── #422 論点7「0円なし」は価格決定が0円フォールバックを持たない方針であって、マスタの正規入力としての0円を禁じるものではない。
4. **「無効化」という語・操作を本画面に作らない**。価格を止めるのは「適用終了（編集）」、誤入力の取り消しは「削除（未適用限定）」。
5. 有効商品から共通売単価の**全期間を失わせる操作は提供しない**（最低1期間前提）。
6. 原価は本画面に一切表示しない。
7. **行の時点状態で編集・削除の権限を分ける**（過去を書き換えない＝時点解決の真実性保護）。ドメイン操作の不変条件として強制する。

   | 行の状態 | 編集 | 削除 |
   |---|---|---|
   | 将来（今日 < 開始） | 全項目可 | 可（誤入力訂正） |
   | 現在有効（開始 ≤ 今日 < 終了） | 終了日設定（適用終了）のみ可 | 不可 |
   | 失効（今日 ≥ 終了） | 不可 | 不可 |

## 6. 用語（CONTEXT.md 準拠）

- **適用終了 (End-dating)**: 終了日を設定し以後適用しなくする編集。独立コマンドではない。
- **失効 (Lapsed)**: 参照日が `[開始, 終了)` に含まれない**派生状態**（フラグではない）。
- **無効化 (Deactivation) / 無効 (INACTIVE)**: Product・バリエーション専用の予約語。**本画面では使用しない**。

## 7. 確定した設計方針

1. **最低1期間前提(#422論点7)の担保方法** ─ 第一段階では跨集約のハード強制（商品有効化時チェック等）を作らず、価格決定側の既存「解決不能で操作拒否」の安全弁に委ねる。画面は **未設定商品を可視化する導線**（UC-1 の未設定表示・絞り込み）で担保する。ハード強制は別スライスで検討。
2. **作成と編集の画面統一(#351)** ─ 登録（UC-3）と編集（UC-4）を**単一フォーム部品**で統一する（既存マスタ画面の統一方針を踏襲）。
3. **一覧の粒度** ─ UC-1 は**商品単位で1行**（集約境界に一致）、期間明細は UC-2 のドリルダウンで展開。フラットな期間一覧にはしない。

## 8. 将来拡張への配慮

第一段階は共通売単価のみだが、今後のユースケース次第で **共通売単価＋原価の併記**や **共通＋得意先別の並置**を行う可能性がある。第一段階の画面構造が、後からこれらの併記ビューを足すことを**構造的に妨げない**よう、商品単位の枠組み・期間明細コンポーネントを独立性高く保つこと（原価・上書き層を別コンポーネントとして後付けできる余地）。
