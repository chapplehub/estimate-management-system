# Issue #294 実装の計画逸脱・既知の制約

## 1. §4.8「受注作成後は編集不可」ガードの先送り

- **元の計画**: イシュー本文は C2 の不変条件として §4.8 を挙げていた。
- **実際の実装**: C2/C3/C4 に §4.8 の受注ロックガードを実装していない。
- **理由**: `Estimate` 集約に `Order` への参照が無く（スキーマ schema.prisma の Estimate↔Order
  逆リレーションは別イシューへ明示的に先送り済み）、集約が「受注作成済みか」を観測できない。
  今強制するには受注存在フラグの捏造か Order 一式の引き込み（C5 を除外した理由と同型のスコープ
  膨張）が必要になる。Estimate↔Order リンク実装イシューにブロックされるため先送りとし、リンク
  実装時に C2/C3/C4 へガードを追加する。

## 2. granular な per-item ミューテータは当面コマンド未使用（温存）

- **元の計画**: C4 は明細を宣言的フルセット置換（Path 1）。
- **実際の実装**: `EstimateVariation` の `addItem`/`removeItem`/`changeItem*`/`changeOverallDiscount`
  と、ルートの対応委譲メソッド群は、本イシューのどのコマンドからも呼ばれない。C4 は専用の
  `replaceContent`（単発 recalc）で全置換する。
- **理由**: 編集画面のフォーム全体保存という実態に full-replace が合致し、クエリ層（安定 itemId の
  供給元）がスコープ外のため。granular メソッドは将来の「サーバ権威の編集ごと永続化」
  （autosave／共同編集。確定金額をクライアント計算とズレなく即時表示する要件）で活きるため
  削除せず温存する。削除可否は当該要件の確定時に別途判断する。

## 3. 楽観ロック未対応（lost update の既知制約）

- **元の計画**: 並行更新はこのイシューでは扱わない。
- **実際の実装**: C2/C3/C4 は「findById → メモリ上で変更 → save」型で、version 列も保存時の
  一致チェックも持たない。2 人が同じ見積を同時編集すると後勝ちで片方の変更が静かに消える。
- **理由**: version 列が無く既存 Update コマンド群も未対応で、横断対応が必要。スキーマ移行と
  全 Update コマンドへの波及を伴うため別イシュー **#301** に切り出した。

## 4. `checkTaxRateThenSave` の単独ユニットテストを置かない

- **元の計画**: 計画 Step 3 は共通機構の単体テスト（consistent=save / mismatch=未保存）を想定。
- **実際の実装**: `checkTaxRateThenSave` 専用テストは作らず、その分岐（saved / taxRateMismatch
  未保存）を C2/C3/C4 の実 Prisma 統合テストで網羅した。
- **理由**: 本プロジェクトはテストダブル（vi.fn/Fake/InMemory）を一切使わず、実 DB 統合テストで
  検証する方針（mocking.md「自分のクラスはモックしない／DB はテスト DB を使う」）。薄い
  オーケストレーション関数を fake で単独テストするとこの方針に反するため、コマンド統合テストで
  実質的に同じ振る舞いを検証する形に寄せた。
