# Issue #332 実装計画からの逸脱記録

計画ファイル: `variation-content-edit-c4.md`（10ステップ）。TDD（`/tdd`）で実装した際の逸脱。

## 1. ステップ順を TDD 用に並べ替え

- **元の計画**: Step 2（dnd-kit 依存追加）→ Step 3（スキーマ）→ … の番号順。
- **実際の実装**: 純粋ロジック（スキーマ → 金額プレビュー → 作業コピー操作 → 写像 → Server Action）を先に RED→GREEN で固定し、UI を積んだ後、D&D 実装の直前（Step 9 相当）で dnd-kit を追加した。
- **理由**: TDD の縦スライス原則。純粋ロジックは dnd-kit を必要としないため、テスト可能な単位を先に通すのが無駄がない。

## 2. Step 5/6/7 を1コミットに統合

- **元の計画**: 編集トグル（5）／インライン編集・プレビュー（6）／追加・削除・行アクティブ（7）を別ステップ・別コミット。
- **実際の実装**: 「編集可能なバリパネル」一式として1コミット（`87879f5`）にまとめた。
- **理由**: 編集テーブルの無いトグルは単体で意味を成さず密結合。1つの成果物として扱う方が自然。

## 3. 純粋モジュールを追加抽出（計画に明記なし）

- **元の計画**: スキーマ（Step 3）・Server Action（Step 4）が主な新規ファイル。
- **実際の実装**: `previewAmounts.ts`（金額プレビュー）・`variationLines.ts`（作業コピー操作）・`variationContentMapping.ts`（form値→C4入力の写像）をテスト可能な「縫い目」として切り出し、それぞれ単体テストした。
- **理由**: TDD のテスト容易性。特に写像は `actions.ts`（"use server"）から同期 export できないため別モジュール必須。

## 4. 編集テーブルから改訂価格列を省略

- **元の計画**: 明記なし（閲覧テーブルには改訂価格列あり）。
- **実際の実装**: 編集 variant（`LineEditTable`）では改訂価格列を出さない。
- **理由**: S4 編集対象は非改訂バリのみで改訂価格は常に null（「—」）・編集不可。列幅の無駄を省いた（§10 と整合）。

## 5. D&D を E2E から除外

- **元の計画**: Step 10 で「並び替え」を含む受け入れ条件を E2E 化。
- **実際の実装**: D&D 並べ替えは E2E に含めず、並べ替えロジックは `reorderLines` のユニットテストで担保した。
- **理由**: dnd-kit のポインタ操作は Playwright で不安定。ヘッダー編集 E2E が税率不一致をユニットに委ねるのと同方針。

## 6. conform の空メモ検証バグを修正（計画外）

- **元の計画**: なし（実装中に E2E で発覚）。
- **実際の実装**: conform の `parseWithZod` が空フォームフィールドを `undefined` に変換するため、メモ（トップレベル＋各明細の顧客/社内）を `z.string().optional()` にした。`safeParse` では再現しない conform 経路の回帰テスト（`variationSchema.conform.test.ts`）を追加。
- **理由**: 空メモ送信が「expected string, received undefined」で検証失敗し保存できなかった。空＝未入力＝Null Object（ADR-0034）に対応するので optional が意味的にも正しい。

## 7. サジェスト E2E 用のシードデータ追加

- **元の計画**: なし。
- **実際の実装**: `seed-e2e.ts` に周辺商品サジェスト E2E 専用の商品 PRD810（本体）→PRD811（周辺）と関連を追加。
- **理由**: 既存シードに ProductRelation が無く、サジェストの受け入れ条件を E2E 化できないため。専用商品で他テストから隔離した。
