# ADR-20260627-86b: 共通売単価の保守は「過去不変」を芯に、行の時点状態を集約の不変条件として強制し、参照日は実行時サーバー生成で注入する

| ステータス | 採用 |

## 改訂履歴

- 2026-06-27 初版（過去不変を芯にした時点ガード・参照日の実行時注入・適用終了の独立コマンド化）
- 2026-06-29 一覧読みモデルの単価表現を「値/null の2値」から **三状態 `priceStatus`（active/lapsed/unset）** へ更新（#473）。当初は「未設定の内訳は持たず null に畳む」としていたが、保守画面の業務要件として「失効中（期間はあるが現在有効が無い）」と「未設定（期間が皆無）」の区別が必要になったため、BE 一覧読みモデルが状態を直接返すよう変更した。詳細は「## 改訂: 一覧読みモデルの三状態化（#473）」節を参照。

## コンテキスト

#466 で `CommonSellingPrice` 集約に保守系のドメイン操作（登録・編集・適用終了・削除）と編集/一覧の読みモデルを実装する。集約は現在 `create` / `addPeriod` の追加専用で、編集・適用終了・削除のドメイン操作を持たない（#426/#448 で集約・Repository・時点解決 QueryService までは実装済み）。

共通売単価は「適用期間 `[開始, 終了)` × 単価」の行を商品単位で束ねる。価格決定（ADR-0064）は見積年月日時点で有効な行を解決し、解決した見積単価は明細上で固定される（マスタ改訂に追従しない）。つまり **過去のある日に有効だった行は、既に発行された見積がその値で時点解決した「履歴の真実」** であり、後から書き換えると時点解決の再現性が壊れる。

ここで「保守画面が行をどこまで編集・削除できるか」「その可否を判定する『今日』をどう供給するか」「可否ガードをどの層で守るか」を決める必要がある。判断は相互に依存する（ガードは『今日』に依存し、『今日』の供給時点が安全性を左右する）。

なお #429（保守画面）向けの `docs/claude-plans/issue-429/use-cases.md` は **UI 設計者（claude.ai design）への入力メモ**であり、ドメイン/実装の正本ではない。同書が「適用終了は独立コマンドではない」と記すのは UI 上の見せ方（無効化という別操作を作らない）の主張であって、実装のコマンド構造を拘束しない。**本 ADR をドメインルールの正本とする。**

典拠: ADR-0066（共通売単価集約）, ADR-0067（適用期間 VO / daterange+EXCLUDE）, ADR-0039（楽観ロック）, ADR-0030（横断コンテキストは引数）, ADR-0025 / 20260624-95f（JST 暦日文字列）, ADR-0018（状態遷移は専用コマンド）, ADR-0038（失敗は throw）, CONTEXT.md「価格」節。

## 検討した選択肢

判断軸は4つあり、いずれも「過去不変」という単一の芯から導出される。

### 軸1: 行の時点状態による編集・削除の権限

| 行の状態 | 編集 | 削除 |
|---|---|---|
| 将来（今日 < 開始） | 全項目（開始日・終了日・単価）可 | 可（誤入力訂正） |
| 現在有効（開始 ≤ 今日 < 終了） | 終了日設定（＝適用終了 end-dating）のみ可 | 不可 |
| 失効（今日 ≥ 終了） | 不可 | 不可 |

現在有効行の単価を直接書き換えさせない。単価改定は「適用終了（終了日を未来方向に確定）＋新期間追加」で表現し、各行を不変の履歴として残す。将来行はまだどの見積も時点解決していないため全項目訂正可。失効行は過去そのものなので不可。

### 軸2: 可否ガードの置き場 — A. 集約の不変条件（採用） / B. command 側のガード（不採用）

A を採用。状態別権限を集約のドメイン操作の不変条件として強制し、違反は `BusinessRuleViolationError` を throw する。command 側だけのガードにすると、別経路（将来の別 command・テスト・バッチ）が集約を直接操作したときに過去を書き換えられる。集約に置けば経路を問わず守れる（ADR-0027/0029 と同じ構造保証の思想）。

ただし永続化からの復元 `reconstruct` は不変条件を再検証しない（既存設計を踏襲）。これにより seed・移行（バックフィル）で過去日データを投入する道は残り、業務操作だけを厳格化できる。

### 軸3: 「今日」（参照日）の供給 — A. 保存実行時にサーバー生成（採用） / B. 画面表示時の値をフォーム往復（不採用）

A を採用。参照日が守るのは「過去を書き換えない」であり、過去か否かは **保存が実行されたその瞬間の今日**で判定しないと、日付をまたいだ保存で「表示時は将来→実行時は現在有効」になった行の単価・開始日を書き換えられてしまう（過去保護の破れ）。

これは楽観ロックの `version`（画面表示時のトークンをフォーム往復させ編集ウィンドウを守る・ADR-0039）と **逆方向**である点に注意。version は過去から運んでくる、参照日は実行時に生む。

供給経路: Server Action（アプリ境界）で `new Date()` → `toJstCalendarDay()`（ADR-0025 の JST 固定純関数）で `"YYYY-MM-DD"` に変換し、command 入力に `referenceDate` として詰め、ドメインメソッドの**最後の引数**で渡す（ADR-0030 の横断コンテキスト引数渡し）。`new Date()` は Server Action 一箇所に隔離され、command/集約テストは固定文字列で決定的になる。読み側（一覧 SQL `daterange @> 参照日::date`・編集 DTO の行 status）も同じ参照日を入力に取り、`CURRENT_DATE`（DB サーバー TZ 依存）は使わない。

### 軸4: 適用終了のドメイン表現 — A. 独立メソッド/独立コマンド（採用） / B. 編集に内包（不採用）

A を採用。適用終了は入力が `{ periodId, endDate }` のみで、将来行の全項目編集とは入力の形が異なる。独立させれば `endDatePeriod` のシグネチャ自体が「終了日しか受けない＝現在有効行への唯一の操作」を型で表現でき、ADR-0018（Activate/Deactivate を Update から分離）の思想と一貫する。各メソッドが単一の行状態に責任を持つ（edit＝将来行・endDate＝現在有効行・delete＝未来開始行）。

加えて軸1の表に無い**登録（新規追加）の時点制約**として、`addPeriod` は **開始日 ≥ 今日**を強制する（過去にさかのぼった行の後付け登録を禁止）。過去日データの投入は軸2の `reconstruct` 経路に限定し、業務操作からは閉じる。

## 決定

「過去を不変に保つ」を芯に、以下を採用する。

1. **行の時点状態（将来/現在有効/失効）による編集・削除の権限**（軸1の表）を `CommonSellingPrice` 集約の不変条件として強制する。
2. ドメイン操作: `addPeriod(period, price, referenceDate)`（開始日 ≥ 今日＋重複禁止）／`editPeriod(periodId, changes, referenceDate)`（将来行のみ）／`endDatePeriod(periodId, endDate, referenceDate)`（現在有効行のみ・終了日のみ・独立メソッド）／`deletePeriod(periodId, referenceDate)`（未来開始行のみ）。`create` は参照日を持たない。
3. 参照日は保存実行時にサーバーが `toJstCalendarDay(new Date())` で生成し、フォーム往復させず、各メソッドの最後の引数で注入する。
4. 適用終了は独立コマンドとして実装する（UI 上は編集フォーム内の終了日設定として提示してよいが、実装は専用コマンドにディスパッチする）。
5. command は判別共用体を持たず、すべての失敗を throw する（重複・過去開始・状態違反＝`BusinessRuleViolationError`、VO 検証＝`ValidationError`、楽観ロック競合＝`ConflictError`・ADR-0038/0039）。

## 根拠

- **過去不変が単一の芯**: end-dating（消さず区切る）・現在有効行の単価ロック・登録の過去禁止・削除の未来行限定・参照日の実行時生成は、すべて「既発行見積が時点解決した値を後から変えない」の別表現。個別ルールの暗記でなく芯から再導出できる。
- **集約に置くから経路を問わず守れる**: ガードを command に置くと別経路で抜ける。不変条件にすれば構造で守られ、`reconstruct` を例外とすることで移行の柔軟性と業務の厳格性を両立する。
- **参照日は実行時でなければ穴が開く**: 日跨ぎ保存で過去保護が破れる。version と逆方向であることを明示しないと、後続実装が「参照日もフォームに載せる」誤りを犯しやすい（本 ADR が surprising として記録する所以）。
- **読み書きで判定基準を揃える**: 集約の `ApplicablePeriod.contains` と SQL の `daterange @>` が一致するのは、適用期間 VO が daterange と 1:1（ADR-0067）かつ参照日が単一の JST 暦日変換点（ADR-0025）を通るから。この楔が write ガード・一覧 SQL・編集 DTO status の3経路の判定を分裂させない。

## 影響

- `CommonSellingPrice` 集約に `editPeriod` / `endDatePeriod` / `deletePeriod` を追加し、`addPeriod` に `referenceDate` 引数を追加（既存シグネチャ変更・集約テスト修正を伴う）。
- pricing サブドメイン初の `application/commands`（登録/編集/適用終了/削除）を新設。Repository の `update(aggregate, expectedVersion)` を実装（差分 upsert・ADR-0032/0039、スキーマの version 列・EXCLUDE は既設）。
- 読みモデル: 一覧 QueryService（全商品母集合・現在有効単価は値/null の2値・参照日アプリ注入）と編集 QueryService（親に version・期間行配列・各行 status を read 側算出）を新設。DTO の Decimal は文字列で運ぶ。**※「値/null の2値」は #473 で三状態 `priceStatus` に更新（下記「## 改訂」節）。**
- 状態別（将来/現在有効/失効）の編集・削除可否の境界テスト、参照日素通しの command テストを必須とする。
- use-cases.md（#429）の「適用終了は独立コマンドではない」等の記述は UI メモであり本 ADR と競合しない（本 ADR が実装の正本）。

## 改訂: 一覧読みモデルの三状態化（#473）

### 背景

初版は一覧 DTO の `currentSellingPrice` を「参照日に有効な単価の**値 or null**」の2値とし、未設定（期間が皆無）・将来のみ・失効のみを一様に null に畳んでいた（「未設定の内訳は持たず、null＝『今日有効な単価なし』として UI が可視化する」）。畳んだ理由は、当時は「現在有効単価の有無」だけが一覧の関心だと見立てていたため。

しかし #429 保守画面のモック境界を実 BE へ接続する #473 で、保守運用上は **「失効中（過去/将来に期間はあるが現在有効が無い＝要改定）」と「未設定（一度も単価を設定していない＝要初期登録）」を一覧で区別したい**という業務要件が確定した。null の2値ではこの内訳を判別できない（FE 側で商品ごとに期間有無を再取得すると N+1 的な重い処理になる）。

### 決定（初版からの差分）

一覧読みモデル（`CommonSellingPriceListItemDTO`）に **`priceStatus: "active" | "lapsed" | "unset"`** を追加し、BE が1クエリで状態を直接返す。

- `active`: 参照日を覆う期間行がある（`currentSellingPrice` は値）
- `lapsed`: 期間行は在るが参照日を覆う行が無い（将来のみ／失効のみ・`currentSellingPrice` は null）
- `unset`: 期間行が皆無（`currentSellingPrice` は null）

`currentSellingPrice`（値/null）は後方互換のため維持し、`priceStatus` がその null の内訳を区別する。算出は一覧 SQL の派生列（参照日カバー行の JOIN 有無＝active、期間行の `EXISTS`＝lapsed、皆無＝unset）で行い、読み書きの判定基準を揃える芯（軸3・根拠の「読み書きで判定基準を揃える」）はそのまま踏襲する。あわせて検索条件（商品コード/商品名の部分一致・`priceStatus` 絞り込み）も BE 1クエリに寄せ、FE での全件取得→絞り込みを避ける。

### 不変な部分

本改訂は**読みモデルの表現のみ**を変える。過去不変を芯とする時点ガード（軸1）・可否ガードの集約配置（軸2）・参照日の実行時サーバー生成（軸3）・適用終了の独立コマンド化（軸4）は一切変更しない。編集読みモデルの行 `status`（future/active/expired）も従来どおり。
