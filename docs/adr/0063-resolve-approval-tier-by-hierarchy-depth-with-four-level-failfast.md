# ADR-0063: 役職の承認段階は最上位役職からの階層深さで算出し、4 段不変条件を申請時に fail-fast で守る

| ステータス | 採用 |

## コンテキスト

ADR-0062 で、承認要否ポリシーは抽象的な**承認段階**（`ApprovalGoalTier`: 課長段階／部長段階／本部長段階／社長段階）を返し、`ApprovalChainBuilder` が組織スナップショットの `positionTier` を使って具体役職（`finalApprovalPositionId`）を解決する、と決めた。`ApprovalChainBuilder` の入力 `ApprovalChainOrgRole.positionTier: ApprovalGoalTier` は**既に解決済みの段階値**を要求しており、ビルダー自身は `positionTier.isAtLeast(goalTier)` で比較するだけで「どの役職が課長か」を知らない。

しかし ADR-0062 は「アプリ層が組織サブドメインからスナップショットを組み立てて `positionTier` を詰める」とだけ述べ、**`PositionId` をどう承認段階に翻訳して `positionTier` を作るか**という解決方式を意図的に未記載のまま残した。本 ADR（issue #408）はその空白を埋める。

前提となる組織モデルの制約:

- `Position` は `superiorPositionId` の自己参照（線形の連結リスト）で、明示的な `level` 列を持たない（`schema.prisma`）。
- `PositionName` は自由文字列（`VarChar(50)` unique）で、「どの役職が課長／部長か」をシステムが文字列から構造的に判定する根拠にはできない。
- seed では `課長 → 部長 → 本部長 → 社長`（`POS001..POS004`）の 4 段が、`ApprovalGoalTier` の 4 ランク（rank 1..4）と 1:1 で対応する。
- 業務はこの「**4 役職 = 4 承認段階**」を不変条件として保証・強制する前提（#408）。ただし現状コードにこの不変条件を強制する制約は無く、`Position` 階層は schema 上は任意深さ・枝分かれを許す。

## 検討した選択肢

### A. `PositionName` 文字列マッチで段階を決める（不採用）

`PositionName` を「課長／部長／本部長／社長」にマッチさせて承認段階を割り当てる。ADR-0062 は「`PositionName` は自由文字列ゆえポリシーが依存できない」ことを選択肢 B 採用の根拠にした。同じ脆さをアプリ層へ移すだけで、役職改名（例: 「課長」→「営業課長」）で静かに壊れる。

### B. 最上位役職からの階層深さで算出する（採用）

最上位役職（`superiorPositionId = null` ＝社長）を構造的アンカーとし、根からの距離で段階を割り当てる。

```
distance 0 (社長 / superiorPositionId=null) → 社長段階 (PRESIDENT・rank 4)
distance 1 (本部長)                          → 本部長段階 (DIVISION_MANAGER・rank 3)
distance 2 (部長)                            → 部長段階 (DEPARTMENT_MANAGER・rank 2)
distance 3 (課長)                            → 課長段階 (SECTION_MANAGER・rank 1)
```

マスタ追加ゼロ。「4 役職 = 4 承認段階」という業務の不変条件を、構造（深さ）からそのまま導出する。

### C. `PositionId → 承認段階` の専用マッピングを持つ（不採用）

明示的なマッピング（テーブル / 設定 / カラム）で `PositionId` ごとに段階を宣言する。改名・構造変更に強いが、業務が 4 対 4 を不変条件として強制する前提では冗長で、新役職追加のたびに段階宣言を維持する運用コストが恒常的に乗る。

## 決定

**B を採用する。** 役職の承認段階は、最上位役職（`superiorPositionId = null`）を起点とした階層深さで算出する（距離 0→社長段階 … 3→課長段階）。

ただし深さ算出の唯一の弱点（前提が破れたとき段がずれて**黙って**誤った承認ルーティングを生む）を打ち消すため、解決時に「**最上位を起点に正確に 4 段の単一鎖**」であることを検証し、破れていたら**申請時に fail-fast で例外を投げる**。

解決責務は estimate サブドメインの**アプリ層**（`EstimateApplication` を生成する申請ユースケースの純粋 resolver）に置く。`Estimate` 集約・`position` サブドメインには持たせない。

## 根拠

- **不変条件に忠実で最小コスト**: 「4 役職 = 4 承認段階」を業務が強制する以上、構造（深さ）から導出するのが最も素直で、マスタ追加・運用負荷がゼロ。専用マッピング（C）は同じ事実を二重管理する冗長。
- **fail-fast で唯一の弱点を無害化**: 深さ算出が静かに誤る唯一のケース（係長・専務などの段追加で前提が破れる）を、申請時の例外として顕在化させる。専用マッピングが持つ「未割当なら止まる」という明示性を、実行時検証で代替する。
- **アンカーは根（null superior）**: 葉側は schema 上 `subordinates` が 1 対多で枝分かれしうるが、根の `null` は構造的に一意。各ノードの距離は単一親（`superiorPositionId`）を辿るだけで曖昧さなく定まる。
- **翻訳の所有権は estimate 側（依存の向き）**: `承認段階`（`ApprovalGoalTier`）は estimate ドメインが承認のために定義した語彙で、`position` サブドメインには存在させない。依存は `estimate → 組織`の一方向に保ち、汎用の組織サブドメインへ承認都合の段階概念を逆流させない。ADR-0062 の「アプリ層が snapshot を組み立てる」に忠実。
- **純関数でテスト容易**: resolver は取得済みの役職グラフを引数で受ける純関数（ADR-0030 / ADR-0052 / ADR-0025 の引数渡し・ドメイン純関数の流儀）。コマンド・リポジトリ・DB なしに 4 段検証と段階割当を単体テストできる。

不採用理由:

- **A**: 自由文字列依存（ADR-0062 が拒否したのと同じ脆さ）。改名で破綻する。
- **C**: 4 対 4 が不変条件である前提では冗長で、恒常的な運用コストが過剰。

## 影響

- resolver は、最上位からの距離が `[0, 3]` の範囲外、または「根から 4 段の単一鎖」を成さない場合（鎖長 ≠ 4 / 同一距離に複数役職 / 根が複数または不在）に例外を投げる。これにより係長・専務などの段追加は「黙って誤る」のではなく**申請時エラーとして顕在化**する。新たな段を承認に組み込むには、本 ADR と `ApprovalGoalTier` の再設計が必要。
- `positionTier` を供給する責務は `EstimateApplication` を生成する申請ユースケース（別アプリ層 issue で実装）にある。`Estimate` 集約は申請ロジックを持たず、申請の入力（`finalTotal` / `leafCategories` / `estimateType` / 対象バリエーション）を読まれる側に徹する（ADR-0027 集約境界）。`position` サブドメインは素の役職グラフ供給のみ。
- issue #408 の成果物は、本 ADR・CONTEXT.md の用語（`承認段階`）・純粋 resolver（単体テスト付き）まで。申請コマンドと組織スナップショット組立ては別のアプリ層 issue、リポジトリの Prisma 実装は #407（インフラ層）に切り分ける。
