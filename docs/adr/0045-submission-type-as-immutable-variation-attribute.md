# ADR-0045: 提出区分はバリエーション単位の不変保存属性とする

| 項目       | 値         |
| ---------- | ---------- |
| ステータス | 採用       |
| 起票日     | 2026-06-12 |
| 最終更新日 | 2026-06-12 |

## コンテキスト

提出区分（`submissionType`: 得意先宛 / 納品先宛）は当初 `Estimate` 本体の属性として実装された（`estimates.submission_type` 列＋ `Estimate.changeSubmissionType()`）。しかし C7 ReviseForCustomer（#296）の設計グリルで、これがモデリングの誤りであることが確定した。

- 得意先改訂は「納品先宛バリエーションから得意先宛バリエーションを派生させる」**集約内操作**であり、見積全体の性格変更ではない
- したがって同一見積内に納品先宛と得意先宛のバリエーションが**共存しうる**。提出区分が見積単位だとこの状態を表現できない

移動先をバリエーション単位と決めた上で、さらに 2 つの設計判断が必要だった。

1. **保存属性か、導出属性か**: 改訂系譜（EstimateVariationRevision・§11.2.1）が存在するなら「改訂先 = 得意先宛」と系譜から導出できるのではないか
2. **可変か、不変か**: 従来は C2 UpdateEstimate で見積単位の提出区分を変更できた。バリエーション単位に移した後も変更操作を残すか

## 決定

提出区分を `EstimateVariation` の**不変の保存属性**とする（#320）。

1. **バリエーション単位**: `estimate_variations.submission_type` 列（NOT NULL・デフォルトなし）。`estimates` からは撤去
2. **保存属性（系譜から導出しない）**: 得意先宛バリエーションは得意先改訂を経ずに C1/C3 で**直接作成もできる**（改訂は経路のひとつにすぎない）。直接作成されたバリエーションには系譜が存在しないため、導出は原理的に不可能
3. **不変（ミューテータ非提供）**: 宛先を後から切り替える業務操作は存在しない。`changeSubmissionType()` は `Estimate` から撤去し、`EstimateVariation` にも移さない。不変性はバリデーションではなく「セッターが存在しない」構造で担保する。提出区分を指定できるのは作成系記述子（C1/C3 の `EstimateVariationDescriptor`、C6 の継承、C7 の改訂生成）のみで、C4 UpdateVariation の `VariationContentDescriptor` からは型レベルで除外する（`Omit`）。宛先を誤って作成した場合の訂正経路は「無効化（`VariationStatus.INACTIVE`）して作り直す」

## 検討した選択肢

### A. 系譜からの導出（不採用）

「改訂系譜を持つバリエーション = 得意先宛」と導出すれば列を持たずに済む。だが直接作成された得意先宛バリエーション（系譜なし）と納品先宛バリエーションが区別できず破綻する。

### B. 可変の保存属性（不採用）

C4 UpdateVariation の入力に `submissionType` を含め、バリエーション単位で変更可能にする。入力ミス訂正には便利だが、「納品先宛の明細をラベルだけ得意先宛に張り替える」操作が得意先改訂（価格を再計算して派生させる正規経路・§7.2）の脱法ルートになる。業務上「宛先の切り替え」という操作自体が存在しないことを確認し棄却。

### C. 不変の保存属性（採用）

上記「決定」の通り。将来セッター追加の提案が出た場合は、まず本 ADR の前提（切り替え操作は業務に存在しない）が変わったかを確認すること。

## 帰結

- §7.2「納品先宛は申請・受注作成不可」の判定はバリエーション単位の検査になる（受注が `variationId` に 1:1 で紐づく構造と整合）
- C6 複製の「提出区分は複製元から継承（§5.3）」は、見積単位→**複製元バリエーション単位**の継承になる
- 旧 `estimates.submission_type` の単独 index は移設せず**廃止**する。2 値 enum の単独 index は選択率が低く実用しないうえ、バリエーションへの絞り込みは常に `estimateId` 起点（既存の複合 index で十分）。横断検索の画面要件が実際に生まれた時点で、そのクエリ形に合わせて再導入する
