# ADR-0057: 見積複製(C6)の UI は作成画面統合ではなく詳細画面モーダル＋DuplicateEstimate 経路で実装する

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-18 |
| 最終更新日 | 2026-06-18 |

## コンテキスト

issue #360 本文は、見積複製(C6)の UI を「#351 の C1 新規作成画面に『ヘッダ引き継ぎ＋バリ選択プリフィルの新規作成』として統合する」前提で起票されている。その根拠として ADR-0042 の「『見積のみ複製』はプレゼン層のヘッダ引き継ぎ新規作成 UX として実現する」を挙げる。

しかし配線設計の調査で前提のズレが判明した。

1. **コマンドが別物**。C1 `CreateEstimate` はヘッダー全項目＋初期バリ1件をインライン編集で組み立てる純粋な組立器。C6 `DuplicateEstimate` は複製元をロードし、見積区分・得意先・納品先・税端数区分・提出区分・修理詳細を**複製元から継承**し（`EstimateDuplicationService`）、選択バリの内容を**単価クリア**して複写し、**複製系譜(`EstimateVariationCopy`)を記録**する。利用者入力面は `selectedVariationIds[]`＋見積日／締切／部署のみと遥かに小さい。
2. **AC が CreateEstimate 経路では満たせない**。#360 の受け入れ条件（≥1選択・系譜記録・単価クリア）は `DuplicateEstimate` でしか実現しない。`CreateEstimate` は系譜も単価クリアも持たない。
3. **ADR-0042 の「ヘッダ引き継ぎ新規作成」は別ケース**。それは §5.2「選択なし可（見積のみ複製）」＝**バリ0件**のケースを指し、系譜なしの `CreateEstimate` に流す将来 UX である。≥1選択・系譜ありの C6 とは別物で、issue 本文はこの2つを混同していた。

加えてこのアプリは**編集専用ルートを持たず**、詳細画面(`[estimateNumber]`)が閲覧と編集(C2/C4 インライントグル)を兼ねる。よって「複製後に編集画面へ」は新ルート不要で、新見積の詳細画面に着地するだけで単価再入力の導線が既存編集パイプラインに委譲できる。

## 検討した選択肢

### A. C1 作成画面に複製モードを足す（不採用）

`CreateEstimateForm` を `mode:create|duplicate` でパラメータ化し1フォームに畳む。入力形状（全ヘッダー入力＋1バリのインライン編集 vs 継承＋複数バリ選択）と駆動コマンド（CreateEstimate vs DuplicateEstimate）が大きく異なり、神コンポーネント化する。系譜・単価クリアは CreateEstimate に無く、結局別経路を内包することになる。

### B. 専用の複製画面（独立フルページルート）を新設（不採用）

C6 専用ルートにフォームを置く。だが C6 の入力面は小さく（バリ選択＋3項目）、かつバリ内容の編集はここで行わない（単価は遷移先で再入力）ため、フルページは過剰。バリ選択には複製元のバリ一覧が要るが、それは詳細画面が既に保持している。

### C. 複製元詳細画面のヘッダー操作エリアに「見積複製」ボタン → モーダル → DuplicateEstimate → 新見積詳細画面へ遷移（採用）

複製元詳細画面に「見積複製」ボタン（`VariationPanel` 内の C3「複製」とはラベル・階層で区別）を置き、モーダルでバリ複数選択(≥1・複製元の並び順で採番・`isVariationDuplicatable` で適格性判定し不適格バリは表示＋チェック不可)・見積日・締切・部署を収集する。税率は見積日からライブ導出し read-only。確定で `DuplicateEstimate` を駆動し、新見積の詳細画面へ閲覧モードで遷移、単価は既存 C4 インライン編集で再入力する。

## 決定

C を採用する。見積複製(C6)の UI は C1 作成画面への統合でも専用フルページでもなく、複製元詳細画面のヘッダー操作エリアから開くモーダルとして実装し、`DuplicateEstimateCommand` を駆動して新見積の詳細画面へ遷移させる。

## 根拠

C6 は CreateEstimate とは駆動コマンド・入力形状・成果物（系譜・単価クリア）が異なり、作成画面への統合(A)は神コンポーネント化する。C6 の入力面は小さくバリ内容編集を伴わないため専用フルページ(B)は過剰で、バリ選択に必要な複製元バリ一覧は詳細画面が既に持つ。編集専用ルートが無く詳細画面が編集を兼ねる構造上、単価再入力は遷移先詳細画面の既存編集パイプラインへ委譲でき、新規実装は「モーダル部品＋Server Action＋遷移」に縮小する。ADR-0042 の「ヘッダ引き継ぎ新規作成」はバリ0件の別 UX であり、C6 と同一視しない。

## 影響

- #360 の実装は #351 作成画面へ手を入れず、複製元詳細画面に「見積複製」モーダルと専用 Server Action を新設する。#351 への依存は共有部品（`resolveEffectiveTaxRate`・日付ユーティリティ・`REDIRECT_REASON` パターン・部署 select 等）の再利用に縮小する。
- 税率導出＋§8.7 整合チェックは、ADR-0056 と対称な app-shared `checkTaxRateThenDuplicate` が担う（コマンドは `taxRate` を生値で受け取るのみで §8.7 を保証しないため）。
- 複製元の適格性（改訂明細を含むバリは複製元にできない）は C3 と同じく UI 抑止に依存する（`isVariationDuplicatable` 再利用）。ドメイン側未ガードという非対称は据え置き、強化は別 issue 候補。
- issue 本文の「#351 統合」前提からの逸脱として `docs/claude-plans/issue-360/deviations.md` に記録する。
- ADR-0042 の「見積のみ複製＝ヘッダ引き継ぎ新規作成 UX」はバリ0件ケースの将来 issue として依然有効で、本 ADR とは独立。
