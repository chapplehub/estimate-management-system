# ADR-0037: アプリケーション層コマンドは予測可能な業務結果を判別共用体(Result)で返し、想定外のみ throw する

| 項目 | 値 |
|---|---|
| ステータス | 置き換え済み（→ [ADR-0038](0038-unify-failures-as-exceptions-and-limit-unions-to-multiple-normal-outcomes.md)） |
| 起票日 | 2026-06-09 |
| 最終更新日 | 2026-06-10 |

> **注**: 本 ADR の判定基準「同一リクエスト内でユーザーが入力を直して解決可能 → Result」は曖昧さがあったため、ADR-0038 で「失敗はすべて throw、判別共用体は複数の正常な結末を持つコマンドの戻り値設計に限定」へ再定義された。実装上の結論（`TaxCheckedSaveResult`・汎用 Result 不導入）は 0038 でも存続している。

## コンテキスト

Issue #294（C2/C3/C4 既存見積の基本編集クラスタ）で、コマンドが「保存時の税率チェック」（設計書 §8.6 / §8.7）を扱う必要が生じた。

`TaxRateConsistencyCheckDomainService` は事実返し型ドメインサービス（ADR-0011）で、税率不一致でも throw せず `{ kind: "consistent" } | { kind: "mismatch" }` を返す。設計書 §8.7 は、保存時の税率不一致を「**その場で修正**」（不整合のまま確定保存させず、ユーザーが入力を直す）と定めている。

ここで「不一致をコマンドがどう扱うか」を決める必要があった。これは C2/C3/C4 に限らず、今後の申請・受注コマンド（§8.7 は受注でも税率チェックを要求）や、楽観ロックの競合（#301）など、**「失敗ではないが処理を分岐させたい予測可能な業務結果」を持つコマンド全般**に共通する横断的判断である。

それまでのコマンド（CreateEstimate ほか各サブドメインの Create/Update/Delete）は、成功時に集約または void を返し、異常時は例外（`BusinessRuleViolationError` / `NotFoundEntityError` / `ConflictError`）を throw するのみで、「予測可能な業務分岐を戻り値で表現する」前例が無かった。

## 検討した選択肢

### A. 予測可能な業務結果は判別共用体(Result)で返し、想定外のみ throw する（採用）

成功とビジネス上の分岐を `kind` 判別共用体で返し、呼び出し側に分岐を強制する。想定外（プログラムの前提崩れ・データ不整合）は従来どおり例外で抜ける。

```typescript
export type TaxCheckedSaveResult =
  | { kind: "saved"; estimate: Estimate }
  | { kind: "taxRateMismatch"; estimateDateRate: TaxRate; deadlineRate: TaxRate };

// 税率不一致なら保存せず Result を返す（§8.7）。不在・採番衝突・適用税率なしは throw。
async execute(input): Promise<TaxCheckedSaveResult> { ... }
```

汎用 `Result<T, E>` 型は新設せず、ドメインの `TaxRateCheckResult`（ADR-0011 で定着した `kind` 共用体）に倣い、コマンドごとに名前付きの結果型を定義する。

### B. すべて throw で表現する（不採用）

税率不一致も `BusinessRuleViolationError` 等で throw する。

### C. 不一致でも保存し、警告を戻り値に添える（不採用）

`{ estimate, warning }` を返して保存も行う。

## 決定

アプリケーション層コマンドは、**予測可能な業務上の結果分岐を `kind` 判別共用体(Result)で返し、想定外の状態のみ例外を throw する**。汎用 Result 型は導入せず、コマンドごとに名前付き結果型を定義する。

## 根拠

- **例外を制御フローに使わない**: 税率不一致は「正常に起こりうる業務分岐」であり、エラーではない。throw（選択肢 B）は呼び出し側で try/catch を強い、正常系と異常系の区別を曖昧にする。Result なら型で分岐を強制でき、プレゼンテーション層が「その場で修正」UI を素直に実装できる。
- **§8.7「保存時＝その場で修正」との整合**: 不一致時は**保存せず** Result を返す（選択肢 C は不整合の確定保存を許し §8.7 に反する）。永続化前にチェックすることで DB を汚さない。
- **ADR-0011 との一貫性**: ドメインサービスが事実を返し、エラー化を呼び出し側の責務とする方針の自然な延長。コマンドはその事実を業務結果へ翻訳する。
- **想定外は throw**: 見積不在（`NotFoundEntityError`）・採番衝突（`ConflictError`）・適用税率なし（`BusinessRuleViolationError`）・VO 検証失敗は、呼び出し側が同一リクエストで分岐して回復できる性質ではないため例外が適切。§3.4「無効状態は編集不可」も、救済が別操作（無効→有効の状態遷移）であり同一リクエストの入力修正で解決しないため throw 側に置く。
- **汎用 Result 型を作らない**: 既存に `Result<T,E>`/`Either` は無く、ドメインは `kind` 共用体で統一されている。汎用型の導入はジェネリクスの複雑さを増やすだけで、命名付き共用体の方が結果の意味が型名から読み取れる。

## 影響

- 「同一リクエスト内でユーザーが入力を直せば解決する予測可能な分岐＝Result」「別アクションや前提崩れ＝throw」という線引きを、今後の全コマンドが踏襲する。判断に迷う場合はこの基準で切り分ける。
- 申請・受注コマンド（§8.7 の受注パターン＝新バリエーション作成必須）や、楽観ロックの競合（#301）も本方針に整合させる。#301 は本 ADR の線引きを前提に競合の返し方を決める。
- プレゼンテーション層は Result の各 `kind` を網羅して扱う必要がある（網羅漏れは型で検出可能）。
- コマンドごとに結果型が増える。横断で共有できる結果（例: 税率チェック→保存の `TaxCheckedSaveResult`）は `application/shared/` に集約して重複を防ぐ。
- 既存コマンド（成功時に集約/void を返す Create/Update/Delete 群）は本 ADR で一律変更しない。予測可能な業務分岐を持たないコマンドは従来どおり throw のみでよい。本方針は「分岐がある場合に Result を選ぶ」指針であり、全コマンドの Result 化を強制するものではない。
