# ADR-0000: ADRの管理運用方針

| 項目 | 値 |
|---|---|
| ステータス | 採用 |

| 最終更新日 | 2026-06-24 |

## 改訂履歴

- 2026-04-03 初版（4桁連番フラット ＋ INDEX 手編集）
- 2026-06-24 ID採番を「連番」→「日付＋短ランダム接尾辞」へ変更（連番の衝突多発のため）

## コンテキスト

設計に関わる重要な判断を ADR（Architecture Decision Records）として記録する。

初版は4桁連番（`ADR-0066`）で ADR を一意識別していた。しかし連番は**生成時に、調整なしで、全 worktree 共有の単調増加カウンタから採番する**構造のため、並行ブランチ開発で同じ番号を独立採番し、マージ時に衝突する。

実例:
- #344 / #346 が両方 `0050` を採番（解消 #352）
- #437 / #449 が両方 `0066` を採番

参照トークン `ADR-NNNN` は本番コードのコメントに多数直書きされるため、衝突後のリネームは参照張替えコストが高い。「生成時に確定し、二度と変わらない ID」が必要になった。

## 検討した選択肢

### A. 4桁連番（不採用・初版方式）

共有カウンタを無調整で採番するため、並行ブランチで必ず衝突する。本改訂の動機そのもの。連番＋CIガード（重複をマージ時に検出）も検討したが、衝突を「サイレントなマージバグ」から「PR時のエラー」に変えるだけで、生成時に共有カウンタを引く構造は温存され、開発中に直書きした参照の張替えも残るため不採用。

### B. 日付のみ `ADR-YYYYMMDD`（不採用）

1日に複数 ADR が上がると衝突する。粒度が粗すぎる。

### C. UUIDv7 フル形式（不採用）

時系列ソート可能・無調整一意で技術的性質は理想的。だが36文字・不透明で、**コメントに直書きする参照トークンとしては最悪**（手打ち不可・口頭不可・grep 困難）。今回の痛点（参照）を悪化させる。128bit は数百件規模の母数に対し過剰。

### D. 日付＋短ランダム接尾辞（採用）

`ADR-YYYYMMDD-sss`。UUIDv7 が狙う「時系列＋無調整一意」を人間スケールに縮約したもの。

### INDEX 自動生成（不採用）

INDEX を frontmatter から機械生成すればマージ衝突は消える。だが INDEX のカテゴリ分類（承認フロー／価格決定 等の見出し）は日付・ID から導出できない**人間の編集知識**であり、素朴に生成すると失われる。この分類の価値（ADR-0000 がディレクトリ分類方式を退けた理由そのもの）を優先し、INDEX は人手キュレーションのまま残す。

## 決定

### ファイルの配置場所と命名規則

配置場所: `docs/adr/`

ファイル名: `YYYYMMDD-sss-{slug}.md`
- 例: `20260624-a3f-common-selling-price-aggregate.md`
- `sss`: base36（0-9a-z）3桁のランダム接尾辞（46,656通り）。`openssl rand -hex 2 | cut -c1-3` 等で生成する。
- `slug`: 設計変更のタイトル（3〜6語程度をハイフンで連結）

### 参照トークン

本文・コード・コミットでは `ADR-YYYYMMDD-sss` で参照する（例: `ADR-20260624-a3f`）。

### 採番時の衝突確認

新規作成前に `ls docs/adr/` で同一 `YYYYMMDD-sss` が存在しないことを確認する。万一かぶった場合は接尾辞を採り直す。

### INDEX.md の役割

ADR 全体を見渡す目次として人手で維持する。各 ADR をカテゴリ見出し配下に1行追記する。1つの ADR が複数カテゴリに属する場合は複数見出しに記載してよい。

### TEMPLATE.md の使い方

タイトル、ステータス、起票日、最終更新日を更新し、各セクションを記載していく。

### どのような判断を ADR として記録するか

- その決定を覆すと、複数のコンポーネント（モジュール・サービス・テーブル等）に影響する
- その決定の理由を、一言で表現できず調査するとなると時間がかかりそうなもの
- 2つ以上の選択肢を検討した

### ステータスの運用（採用→廃止/差替の流れ）

新規追加時はステータス：採用。採用した ADR を廃止する場合は廃止。採用した ADR を別の ADR の決定に差し替える場合は差替。

### 移行方針（前進適用のみ）

- 既存 `0000`〜`0067` の連番 ADR はリネームしない（参照 churn 回避）。
- 数字 ID（`ADR-0066`）と日付 ID（`ADR-20260624-a3f`）はどちらも不変の文字列トークンで、共存可能。
- 本 ADR-0000 はメタ方針のため例外的に `0000` を維持し、現地改訂する。
- 既存の `0066` 衝突は「最後の数字リネーム」として別 issue で解消する。

## 根拠

### 無調整での一意性（衝突の根治）

ID を共有カウンタではなく「生成時の日付＋ランダム」から導出するため、worktree 間で会話せずに一意になる（UUID／git ハッシュと同じ coordination-free 一意性）。

### identity と reference の分離

人間が打つ参照は読みやすい日付 ID、正確な日付が要る場面はヘッダ表の起票日で足りる。UUIDv7 フル形式を退けたのは、機械的 identity と人間向け reference を1トークンに押し込めると参照が劣化するため。

### 日内順序の扱い（意図的トレードオフ）

接尾辞はランダムで、日内の正確な順序を持たない。決定ログにとっては「いつ頃の判断か」の可読性がミリ秒順序より高価値、という判断。

### 初版から保持する決定

1 ADR = 1 ファイル（diff 可読性）、ステータス運用（採用/廃止/差替）、記録基準、INDEX による柔軟なカテゴリ分類は初版のまま。連番方式（選択肢 A）を初版で採った「グローバル一意な短い参照」という狙いは、日付 ID でも維持される。

## 影響

- 番号衝突と、それに伴う参照張替えが原理的に消える。
- INDEX.md は引き続き人手で維持する（カテゴリ分類を保つため）。更新を忘れると一覧から漏れるリスクは初版同様残るが、ファイル自体は `docs/adr/` に存在するため `ls` で発見できる。
- TEMPLATE.md のタイトル例を日付 ID 形式へ更新する。
- TEMPLATE.md の全セクションが必須ではない柔軟性は維持する。
