# ADR-20260629-3x5: E2E の時点依存テストは today 相対シードで扱う（クロック固定しない）

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-29 |
| 最終更新日 | 2026-06-29 |

## コンテキスト

共通販売単価 保守画面（#429 / #481）は、適用期間行の時点状態（`future` / `active` / `expired`）を **サーバ側の `new Date()` から導出**する（`page.tsx` → `toJstCalendarDay(new Date())` を参照日に渡す）。状態はストアされたフラグではなく、参照日（今日）と適用期間 `[開始, 終了)` からの**派生**である（CONTEXT.md「失効」、ADR-0067）。

この画面の E2E を作成するにあたり、時点依存に起因する以下の問題に直面した。

- 基準日（参照日）を外部から固定する機構が**存在しない**。`actions.ts` も各アクションで `new Date()` を直接生成し「参照日は実行時にサーバー生成して注入する」方針（ADR-20260627-86b）。
- 状態の腐り方が一様でない。`expired`（終了が過去）と無期限 `active`（開始が過去・終了 null）は壁時計が進んでも安定だが、`future`（今日 < 開始）と有界 `active` は時間経過で `future → active → expired` と遷移し、**固定日付のシードはいずれ状態が変わってテストが壊れる**。
- かつ UC-4（将来行の全項目編集）・UC-5（将来行削除）は集約の不変条件上、**将来行が必須**（`CommonSellingPrice.editPeriod` / `deletePeriod` は `isFuture` 行のみ許可）。将来行を避けられない以上、固定日付では成立しない。

E2E を「いつ実行しても決定的」にするための設計判断が必要になった。

## 検討した選択肢

### A. 基準日を env 等で注入する機構を本番に追加（不採用）

`toJstCalendarDay(new Date())` の参照日生成点に、テスト時のみ固定日を注入できる仕組み（環境変数等）を設ける。シードを固定日付にしても状態を決定的に検証できる。

しかし本番コードには `new Date()` 直書きが各 Server Action / ページに定着しており、注入点を後付けすると**テスト都合が本番コードに漏れる**（ADR-0017 が「テストのためにプロダクションコードへ `data-column` を足す」案を退けたのと同じ精神）。注入機構そのものが新たな不具合面・設定面を生む。

### B. today 相対シード＋JST 起点ヘルパ（採用）

シード（`seed-e2e.ts`）とテストコードの双方で「実行時の今日（JST 暦日）」を起点に適用期間の日付を算出する。例：`future = today+30`、有界 `active = [today-30, today+30)`、`expired = [today-60, today-30)`。十分なマージン（数十日）でシード投入（`pnpm e2e:seed`）からテスト実行までの時間差を吸収する。

### C. 固定日付で単調な状態のみ検証（不採用）

固定の過去日で `expired`、固定の過去開始＋無期限で `active` のみを検証し、`future` を検証対象から外す。固定日付で安定する範囲に限れば決定的になる。

しかし将来行が必須の UC-4 / UC-5 を検証できず、本画面の回帰検知という #481 の目的を満たせない。

## 決定

**B（today 相対シード＋JST 起点ヘルパ）** を採用する。基準日固定の注入機構は本番に導入しない。

## 根拠

- **本番非侵入**: 注入機構を持たないため、テスト都合が本番コードに漏れない（ADR-0017 と同じ判断基準）。
- **決定性**: `pnpm e2e:seed` のたびに today 起点で期間を作り直すため、実行日に依らず `future` / `active` / `expired` が安定して再現する。
- **網羅性**: 将来行を必要とする UC-4 / UC-5 を含め、全状態を検証できる。
- **暦日の一致**: 「今日」はサーバの `toJstCalendarDay`（UTC ミリ秒に +9h して `getUTC*`）と**同じ JST ロジック**で算出する。ローカル TZ で計算すると深夜帯で 1 日ずれうるため、シード／テスト側にも JST 起点の日付ヘルパを用意して突き合わせる。

## 影響

- `seed-e2e.ts` の共通販売単価フィクスチャ（PRD901 等）は、期間を today 相対の `daterange` で raw insert する。過去開始（`expired`）は集約の `assertStartNotPast` を通せないため、業務操作経路ではなく raw insert で投入する。
- E2E テストコードは、登録・編集の入力日付を today 相対で生成する（新規期間は不変条件上 開始 ≥ 今日）。
- マージン（本画面では ±30〜60 日）はシード投入からテスト実行までの猶予。CI でも `e2e:seed` 直後に実行するため十分。
- 本 ADR は「サーバ側の参照日から状態を導出する機能」一般に適用する方針とする（今後 得意先別／納品先別販売単価・原価の保守画面 E2E でも同様）。
