# ADR-0060: 改訂先バリエーションは数量も固定する（粗利スナップショット保全）

| 項目 | 値 |
|---|---|
| ステータス | 採用（単価の調整可否は ADR-0064 で改訂） |
| 起票日 | 2026-06-19 |
| 最終更新日 | 2026-06-22 |

## コンテキスト

#390 で改訂先バリエーション（得意先宛・`revisedFrom` あり）の部分編集（単価・掛率・明細値引・全体値引・メモ）を可能にするにあたり、編集可能フィールドの確定で要件と既存ドメインの食い違いが表面化した。

- **要件（#390）**: 改訂先の数量は固定。商品・数量・改訂価格・行構成は変更不可、価格系とメモのみ可変。
- **既存ドメイン**: 行構成固定ガード `assertLineStructureMutable`（`_revisedFrom !== null` で行追加・削除・全置換を拒否）は**数量変更を弾かない**。`changeItemQuantity` は `editableVariationOrThrow`（凍結拒否）を通る経路だが、凍結されるのは改訂*元*であって改訂*先*ではないため、**改訂先でも数量変更が素通りする**。
- **ADR-0046 / CONTEXT.md（旧）**: いずれも「単価・掛率・値引・**数量**・メモの調整は可能」と明記しており、数量を可変として扱っていた。

ここで改訂価格スナップショットの意味論が決定的になる。`reviseForCustomer` は改訂元各明細の **`finalAmount`（行金額＝数量 × 単価 − 値引）** を `RevisedEstimateItemDetail.deliveryPrice` として凍結する（粗利の真実の源・§8.4）。明細単位の粗利は「`deliveryPrice`（改訂元の行金額）− 改訂先の行 `finalAmount`」という**行金額同士の差**である。改訂先で数量を変えると、改訂先 `finalAmount` は新数量で再計算される一方 `deliveryPrice` は旧数量の行金額を凍結したままとなり、**粗利が別物同士の引き算になって意味を失う**。

つまり「数量可変」は型上は正しい入力で粗利の基準を黙って壊す種類のバグであり、旧ドキュメントの「数量調整可」は改訂価格が行金額スナップショットである事実と整合していなかった。

## 検討した選択肢

### A. ドメインに数量固定ガードを新設して二重防御で固定（採用）

`EstimateVariation` に数量専用ガード `assertQuantityImmutable`（`_revisedFrom !== null` なら拒否）を新設し、`changeItemQuantity` 先頭で呼ぶ。エラーメッセージは数量固有（粗利スナップショット保全を明示）とし、行構成固定（行の増減）とは別の名前付き業務ルールとして表現する。UI（調整フォーム）は数量を read-only 表示にし、調整コマンドの入力に数量を含めない。

### B. 行構成固定ガード `assertLineStructureMutable` を `changeItemQuantity` から流用（不採用）

実装1行で済むが、「行構成（追加/削除）」の名前・メッセージで数量変更を弾くことになり、固定の*根拠*（粗利スナップショット保全 ≠ 1:1 行対応保全）が異なる2つのルールを1つの名前に潰す。運用時に数量拒否の理由が伝わらない。

### C. UI 抑止のみ（ドメインは現状維持・不採用）

調整フォームで数量を read-only にするだけでドメインガードを足さない。プレゼンテーション層を経由しない経路（将来のバッチ・API・テスト）から数量変更が通り、粗利スナップショットが壊れうる。本コードベースの二重防御方針（ADR-0044/0046 の凍結・行構成固定はすべてドメイン強制）と不整合。

## 決定

改訂先バリエーションは行構成固定に加えて数量も固定する。固定はドメインの専用ガード `assertQuantityImmutable` で強制し、UI 抑止はその外側の写しとする（A を採用）。

## 根拠

- 改訂価格 `deliveryPrice` が改訂元の**行金額**スナップショットである以上、粗利（§8.4）が行金額同士の差として意味を持つには数量の固定が必須。数量可変は粗利の基準を黙って壊す。
- 数量可変は型上正しい入力で起きるため検証では防げない。ADR-0046 が全置換拒否で採った「操作自体を構造的に閉じる」思想と同型で、操作を入口で閉じるのが確実。
- 固定の根拠が異なる（行構成固定＝1:1 行対応保全、数量固定＝粗利スナップショット保全）ため、名前付きガードを分ける方がテスト・エラーメッセージの意図が読める。凍結と行構成固定を峻別する既存文化（ADR-0044/0046）に倣う。

## 影響

- 本 ADR は ADR-0046（選択肢A説明・影響欄）および旧 CONTEXT.md「行構成固定」定義の「数量の調整は可能」という記述を**改める**。CONTEXT.md には「数量固定」を新語として追加した。
- 改訂先の編集可能集合は「単価・掛率・明細値引・全体値引・メモ（バリ＋明細）」に確定。固定集合は「商品・数量・改訂価格・行構成」。（**ADR-0064・2026-06-22 で改訂**: 単価も固定へ移り、編集可能集合は「掛率・明細値引・全体値引・メモ」に縮小。固定集合に単価が加わる。）
- 改訂先には強度・根拠の異なる固定ガードが3つ併存する: 行構成固定（行の増減）・数量固定（粗利スナップショット保全）・改訂価格固定。いずれも `_revisedFrom !== null` から導出され、凍結（改訂元・メモ以外不可）とは別系統。
- 改訂先の部分編集を行うアプリ層コマンド（`AdjustRevisedVariationCommand`）は数量を入力に取らない。これは ADR-0046 が「将来 issue」とした選択肢C（明細 ID 明示の一括調整）の実装にあたる。
- 将来、改訂価格スナップショットを行金額ではなく単価単位で持つ設計に変えるなら、数量固定の根拠が変わるため本決定の見直しが必要になる。
