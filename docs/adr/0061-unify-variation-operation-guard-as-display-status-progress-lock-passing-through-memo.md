# ADR-0061: バリエーション操作ガードを表示ステータス由来の進行ロックに統一しメモを貫通させる

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-19 |
| 最終更新日 | 2026-06-19 |

## コンテキスト

#335（見積詳細画面 S7・C5 無効化/有効化）で、`ActivateVariationCommand` / `DeactivateVariationCommand`（ADR-0018）とトグル UI を実装する。ドメイン層は既に `EstimateVariation.activate()/deactivate()`・`Estimate.activateVariation()/deactivateVariation()`・`VariationStatus`（ACTIVE/INACTIVE）を備える。残るアプリ層・UI 層の実装に際し、issue が「申請ガードはアプリ層の空の拡張点として置き、申請テーブル（ADR-0001→0053）実装時に接続する」と定めた**ガードの正体・配置・スコープ**を確定する必要がある。

論点は「無効化を禁じるべき条件は何か」。当初想定の「申請中のバリは無効化不可」だけでは不足で、承認済・承認不要（免除）・受注作成済・受注確定済・受注取消済まで含める判断になった（#335 グリル）。この対象集合を棚卸しすると、無効化を許す状態 {作成中, 差戻, 取下} は「表示ステータス上まだ編集できる段階」と完全に一致し、無効化ガードが将来の編集ガード（申請中・承認済バリは単価・明細を編集不可）と**同一の述語**を問うことが判明した。

判定材料（申請・免除・受注の各イベント行）は Estimate 集約の外にあり（ADR-0053 で申請テーブルは見積専用に分離、ADR-0058 で状態は行の存在から導出）、表示ステータスは read 側の導出値である。よってこのガードを集約内不変条件として表現できず、配置の判断が要る。

## 検討した選択肢

### A. 表示ステータス由来の単一述語「進行ロック」をアプリ層に置き、無効化・編集で共有（採用）

`isCommitted(variation)`（＝表示ステータスがロック集合に入るか）を 1 つの述語としてアプリ層に置き、無効化コマンドと将来の編集コマンドの双方が呼ぶ。ロック集合は表示ステータス上の **{申請中, 承認済, 承認不要, 受注作成済, 受注確定済, 受注取消済}**、補集合（無効化・編集を許す自由集合）は **{作成中, 差戻, 取下}**。判定材料は `EstimateApplicationStatusQueryService`（仮）から引く（ADR-0056 の app-shared ラッパ型）。

S7 時点では `DeactivateVariationCommand` 内に**空の拡張点**（現状 no-op、後日この述語へ接続）を置く。編集側ガードの配線は将来 issue、有効化はガードなし。

### B. 無効化ガードをドメイン不変条件にする（不採用）

`Estimate.deactivateVariation` 内で申請・受注状態を見て弾く。だが判定材料が集約外（申請・免除・受注テーブル）にあり、ドメイン層を申請集約に依存させることになる。ADR-0053（テーブル分離・承認ロジックはドメインで共通化するがデータは持ち込まない）と衝突する。

### C. 無効化専用の no-op ガードに閉じ、編集ガードは別述語として将来作る（不採用）

S7 の責務は最小になるが、{作成中, 差戻, 取下} という同一の自由集合を編集ガードでも再実装することになり、ロック集合の二重定義・将来の食い違いリスクを抱える。

## 決定

バリエーションの無効化ガードと将来の編集ガードを、表示ステータス由来の単一述語 `isCommitted`（＝**進行ロック**）に統一する（A を採用）。

- **ロック集合** = 表示ステータス ∈ {申請中, 承認済, 承認不要, 受注作成済, 受注確定済, 受注取消済}。真なら無効化・内容編集を拒否
- **メモは貫通**。バリ単位・明細単位のメモは進行ロックに関わらず常に編集可（ADR-0059 の「メモは金額に効かない注記」を進行ガード一般へ拡張）
- **有効化はガードなし**。無効バリの再有効化は申請・受注を壊さない
- **配置はアプリ層**。判定材料が集約外のため、表示ステータスを QueryService で引く ADR-0056 型のラッパで包む。ドメインの状態遷移メソッドは現状どおり構造的可否のみ担う
- **#335 のスコープ**は無効化側のみ。`DeactivateVariationCommand` に空の拡張点を置き、後日 `isCommitted` へ接続する

## 根拠

- **自由集合 {作成中, 差戻, 取下} ＝ 編集可能段階**。承認済・免除後は編集不可（CONTEXT.md）、申請中は in-flight、受注済は下流が参照、で全てロック側。差戻・取下は作成者の手元に戻り（再申請は新申請）編集可能。よって「無効化できる ⟺ 編集できる」が表示ステータス上で成立し、両ガードが同じ述語を問う。
- **受注取消済もロックに含める**。受注は消えても「一度でも承認到達・受注到達したバリは隠せない」という監査的一貫性を優先。無効化は物理削除でなく一覧には残る（CONTEXT.md「バリエーション状態」）ため、隠蔽ではなく履歴保全として整合する。
- **メモ貫通**は ADR-0059 が凍結に対して張った原則の自然な一般化。凍結（系譜由来）と進行ロック（表示ステータス由来）は直交する別ガードだが、「内容編集と状態変更は止めるがメモは通す」振る舞いを共有する。
- **アプリ層配置**は ADR-0053/0058 と整合。ドメインを申請・受注集約に依存させず、表示ステータスという read 側の導出値を書き込み前に参照する形は ADR-0056（税率整合チェックの app-shared ラッパ）と同型。
- **述語共有（A）**は、無効化専用 no-op（C）が抱える自由集合の二重定義を避ける。issue 文言も「申請ガード」と単数で、単一述語への統一と整合する。

## 影響

- 集約まわりに 2 系統の進行ガードが並ぶ: **凍結**（系譜由来・改訂元・ADR-0044/0059）と **進行ロック**（表示ステータス由来・本 ADR）。両者は独立（直交）だが「メモは貫通」を共有する。CONTEXT.md に「進行ロック」を追加した。
- **#335 では `DeactivateVariationCommand` の拡張点は no-op**。空実装であること・接続先（`isCommitted`）・参照する表示ステータスをコメントで明示し、申請/受注の read model が揃った時点で `EstimateApplicationStatusQueryService` を DI して中身を埋める。
- **将来の編集ガード**接続時、価格と메モを混在編集する経路（`AdjustRevisedVariationCommand`＝改訂先の単価・掛率・値引・メモ、#390）は、価格部分のみ進行ロックの対象とし、メモ部分は貫通させる切り分けが要る。
- ドメインの `activate()/deactivate()`・`activateVariation()/deactivateVariation()` は構造的可否のみを担い続け（最終強制の二重防御）、進行ロックの判定はアプリ層に置く。
- 本決定は表示ステータスの導出（ADR-0058）と申請テーブルの見積専用分離（ADR-0053）に依存する。これらが変われば `isCommitted` の判定材料の取得経路を見直す。
