# ADR-0020: E2Eテストの構成・実行戦略

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-04-17 |
| 最終更新日 | 2026-04-17 |

## コンテキスト

ADR-0012 で E2Eテストの DB 環境と CRUD テストの直列化（`test.describe.serial`）方針を決定した。その後 roles / products / customers / delivery-locations の実装を経て、以下の暗黙パターンが定着したが明文化されていない。

1. serial chain の粒度（何をひとつの chain にまとめるか）
2. テスト種別ごとの実行方式（serial vs 並列）と使用データ（新規 vs シード）
3. 一覧画面における検索条件テストの粒度
4. 機能ごとのファイル分割基準
5. 権限差異がある機能のテスト構成

実装者ごとに判断がブレると、テスト実行時間・保守性・失敗時の blast radius が悪化する。本 ADR でこれらの設計判断を明文化し、スキル `/create-e2e-test` で実装者にルールを提示できるようにする。

## 検討した選択肢

### 判断1: serial chain の粒度

#### A. 全CRUDを1つのserialにまとめる（不採用）

1つの `test.describe.serial` に create → 詳細確認 → 更新 → ステータス切替 → 削除をすべて並べる。一見シンプルだが、chain が長くなるほど中間のテストが失敗した際の blast radius（後続のスキップ範囲）が大きい。また「状態管理の関心事」と「ライフサイクルの関心事」が1つの chain に混在するため、chain 全体の意味が捉えにくくなる。

#### B. 粒度基準「1 chain = 1 ライフサイクル × 1 関心事 × 1 データ区分」（採用）

chain の分割基準を以下とする:

**同じchainに入れる条件（すべて満たす）**:
1. 同じテストデータ（同じCD）を使い回せる
2. 同じ関心事カテゴリ（ライフサイクル / ステータス / サブリソース / リレーション）
3. ユーザーの典型的な利用フローに沿った順序

**別chainに分離する条件（いずれか該当）**:
1. 別のテストデータが必要（区分違い、権限違い）
2. 別の関心事カテゴリ（例: ステータス管理 vs ライフサイクル）
3. chain サイズが 5 を超える（blast radius 抑制）

**具体ルール**:
- ステータス管理（有効化/無効化）は独立 chain
- データ区分ごとに独立 chain（個別商品 vs セット商品）
- 権限ごとに独立 `test.describe`（管理者 vs 一般ユーザー）
- サブリソース操作は独立ファイル（例: `products-relations.e2e.ts`）

### 判断2: テスト種別ごとの実行方式と使用データ

#### A. すべて serial で実行、シードデータ優先（不採用）

全テストを直列化すれば DB 整合性は保たれるが、テスト実行時間が極端に長くなる。また、シードデータを前提にした成功系 CRUD テストは「テストがシードを汚染しないよう毎回 rollback」という仕組みがない現環境では実現困難。

#### B. 副作用あり=serial+新規データ / 副作用なし=並列+シードデータ（不採用）

一見妥当に見えるが、「副作用」の語彙が曖昧で「削除試行（失敗期待）」の扱いがブレる。削除試行は意図としては副作用を起こそうとするが、DB は不変。分類基準として信頼性が低い。

#### C. DB状態変化の有無で4分類（採用）

判断基準を「**副作用**」ではなく「**DB 状態変化の有無**」に置き換え、以下の4分類で戦略を決める。

| テストタイプ | DB状態変化 | 実行方式 | データ戦略 |
|---|---|---|---|
| 成功系CRUD（create/update/delete/activate/deactivate） | あり | `test.describe.serial` | 新規データをテスト内で構築（xxx901/xxx902 帯） |
| ドメインエラー（FK制約 / 使用中 / 下位レコードあり等） | なし | serial 外で並列 | **独自シード**を `seed.e2e.ts` に用意（xxx9NN 帯） |
| 簡易エラー（validation / duplicate / permission） | なし | serial 外で並列 | 共通シード |
| 検索・閲覧 | なし | serial 外で並列 | 共通シード |

**簡易エラー vs ドメインエラーの境界**: 「**関連データの読み込みが必要か**」で判別する。

- 簡易エラー: UI / zod / 権限ミドルウェアで検出（DB 問い合わせなし、または単一レコードの unique チェックのみ）
- ドメインエラー: ドメイン層のビジネスルール検証で検出（関連データを読み込んで判定）

**独自シードの命名規則**:

- **CD（識別子）**: `<prefix>9NN` 帯のみ。汎用シードは 001-0XX 帯、テスト専用は 9NN 帯で予約。CD にシナリオ名は埋め込まない（エンティティ別桁数制約があるため）。
- **シナリオの意味**: `seed.e2e.ts` のコメントと `name/description` フィールド（`E2E専用_<用途>` プレフィックス）で表現する。
- **テストコード側**: `const ROLE_HAS_CHILDREN = "ROLE901"` のように定数化して意図を明示する。

### 判断3: 一覧検索の粒度

#### A. 全条件を個別テスト（不採用）

網羅的だが、同じロジック（例: 完全一致）を共有する複数カラムで似たテストが並び、保守コストが高い。

#### B. parametrize で一括（不採用）

簡潔だが、検索ロジックが異なる条件（完全一致 vs 部分一致 vs ドロップダウン vs ブール）を同じ parametrize に混ぜるとアサーション分岐が複雑化する。複合条件の AND/OR バグも検出しづらい。

#### C. 主要条件は個別 + parametrize 併用（採用）

- 検索ロジックが異なる条件（完全一致 / 部分一致 / ドロップダウン / ブール）の代表例を **1つずつ個別テスト**
- 同ロジックを共有する条件群は **parametrize**（`for...of` でループ）
- **複合条件テストは必ず個別**

### 判断4: ファイル分割の基準

#### A. 常に 1 ファイル（不採用）

機能ごとに1ファイルだと肥大化しやすく、関心事の混在で可読性が落ちる。

#### B. 2ファイル基本 + 必要時に分離（採用）

- **2ファイル基準（デフォルト）**: `xxx-list.e2e.ts` + `xxx-crud.e2e.ts`
- **3ファイルに拡張**: 詳細画面にナビゲーション系（パンくず / 親子リンク / 404）がある場合は `xxx-detail.e2e.ts` を分離
- **独立ファイル追加**: サブリソース操作（例: 周辺商品モーダル、セット構成）は独立（`xxx-relations.e2e.ts` など）

### 判断5: 権限テストの扱い

#### A. 全機能で両ユーザーテスト（不採用）

網羅的だが、権限差異がない機能（例: 得意先・納品先は全ユーザーCRUD可）で管理者テストと一般ユーザーテストがほぼ完全に重複する。無駄なテスト実行時間を生む。

#### B. 権限差異ある機能のみ両ユーザーテスト（採用）

- **権限差異がある機能**（例: 役割・商品は管理者のみ編集可能）→ 管理者 / 一般ユーザー両方で `test.describe` を作成
- **権限差異がない機能**（例: 得意先・納品先は全ユーザーCRUD可）→ 管理者テストのみ + 一般ユーザー簡易 chain（作成 → 削除 の2ステップのみ）

**権限エラーテスト（必須）**: 権限差異がある機能では以下を必須とする。
- 一般ユーザーが管理者専用画面（例: `/xxx/new`）へ直接アクセス → `/signin?reason=forbidden` リダイレクト
- 一般ユーザー用の閲覧画面でも編集系ボタン（更新・削除）が表示されない

フロント側ナビ制御の回帰だけでなく、サーバ側の認可チェックの回帰検知にもなるため必須とする。

**ファイル分割ルール（権限観点）**:
- 原則: `xxx-crud.e2e.ts` 同一ファイルに両ユーザーの `test.describe`
- 300 行超で分離検討（数値は"きっかけ"であり、凝集度で最終判断）
- 分離順序: (1) 権限テスト分離 `xxx-auth.e2e.ts` → (2) chain種別分離 `xxx-status.e2e.ts` → (3) データ次元分離（例: `products-individual-crud.e2e.ts` / `products-set-crud.e2e.ts`）

## 決定

判断1〜5 を採用する。本ADRは既存テストに遡及適用せず、新規作成される E2E テストから適用する。

## 根拠

- **blast radius の最小化**: 判断1 で chain を小さく分割することで、失敗時の影響範囲を抑える
- **シードデータの保護**: 判断2 で成功系 CRUD を新規データに限定することで、汎用シードが改変されるリスクをなくす
- **テスト速度の確保**: 判断2 で DB 状態変化のないテストを並列実行させ、実行時間を抑える
- **ドメインエラーの隔離**: 判断2 の独自シード（9NN 帯）により、ドメインエラーテストが他テストの変更の影響を受けない
- **保守性**: 判断3 で検索ロジック種別ごとに個別テストすることで、バグ原因の特定が容易になる
- **凝集度**: 判断4 でユーザー体験の単位（一覧 / 詳細 / サブリソース）にファイルを分けることで、関連テストが近接する
- **認可回帰の検知**: 判断5 の権限エラーテスト必須化により、サーバ側認可チェックの劣化を検出できる

## 影響

- 新規 E2E テストは本 ADR に従う
- スキル `/create-e2e-test` で本 ADR のパターンを自動適用
- ドメインエラーテスト用に `seed.e2e.ts` へ独自シード（xxx9NN 帯）を追加する運用
- 既存シードデータ（001-0XX 帯）は変更しない
- 既存テスト（roles-crud / products-crud / customers-crud / delivery-locations-crud）への遡及適用は行わない
