# ADR-0029: ドメイン層は構造的不変条件のみ、業務イベント制約はユースケース層に置く

| 項目         | 値         |
| ------------ | ---------- |
| ステータス   | 採用       |
| 起票日       | 2026-06-01 |
| 最終更新日   | 2026-06-01 |

## コンテキスト

Issue #284 の Estimate 集約実装中、業務文書から導出される 2 種類の制約をドメイン層で強制するかどうかの判断が複数回発生した:

| 制約 | 性質 |
|---|---|
| §C1 空見積不可 | 最低 1 バリエーション必須 |
| §3.4 申請バリエーション 1 つ | 申請できるバリエーションは 1 つだけ |
| ADR-0019 サブタイプ排他整合 | estimateType ↔ 詳細テーブル |
| 最低 1 明細制約（暗黙の要請） | バリエーションには明細が必要 |
| 申請時の明細必須 | 申請するには明細が 0 件であってはいけない |

これらを全て「ドメインの不変条件」として集約ルートで強制すべきか、それともユースケース層に分担すべきかが論点だった。

DDD の原則的には「集約は不変条件を守る」が、**「不変条件」と「業務イベント時の制約」を同一視すると過剰制約**になる。Issue #284 の実装中、複数の場面で「ドメインで構造的に禁止すべきか、ユースケースで判定すべきか」を判断する必要があり、その判断軸を ADR として明文化しないと将来の集約実装（Order / Invoice / 申請ユースケース #6）で同じ議論を繰り返すことになる。

特に §3.4 については、当初の計画では集約ルートに「`activateVariation()` 時に他の ACTIVE が無いことを確認」する構造的禁止を入れる予定だったが、**「複数のバリエーションを ACTIVE のまま保持して比較する」**という業務ワークフロー（顧客提案・社内検討）が成立する必要があり、ドメイン層で同時に複数 ACTIVE を禁止すると業務実態と乖離する問題が浮上した。

## 検討した選択肢

### A. すべての制約を集約ルートで構造的に禁止（不採用）

```typescript
class Estimate {
  activateVariation(variationId: EstimateVariationId): void {
    if (this._variations.some((v) => v.isActive())) {
      throw new BusinessRuleViolationError("既に ACTIVE なバリエーションが存在します（§3.4）");
    }
    // ...
  }

  static create(input: { variations: EstimateVariation[]; ... }): Estimate {
    if (input.variations.some((v) => v.items.length === 0)) {
      throw new BusinessRuleViolationError("明細が必要");
    }
    // ...
  }
}
```

DDD の教科書通り。集約が全制約を守る。

**問題**:
- §3.4「**申請できる**バリエーションは 1 つ」を「**ACTIVE であるバリエーションは 1 つ**」と同一視している。実は「申請可能性」と「ACTIVE 状態」は別概念
- 業務上の正当なワークフロー（複数案を ACTIVE のまま顧客に提示する、社内で比較検討する）が成立しない
- 「明細を空にしてから一括入れ替える」「ヘッダだけ先に作って明細は後追加」のような業務ワークフローも禁止される
- 結局アプリ層で「ACTIVE 状態をいったん外して操作してから戻す」のような擬似トランザクションを書く羽目になり、本質的に複雑度がドメインから漏れているだけ

### B. すべての制約をユースケース層で判定（不採用）

```typescript
class SubmitEstimateUseCase {
  execute(estimateId: EstimateId): void {
    const estimate = await this.repo.findById(estimateId);
    if (estimate.variations.filter((v) => v.isActive()).length !== 1) {
      throw new Error("申請するには 1 つの ACTIVE バリエーションが必要");
    }
    // ...
  }
}
```

ドメインは緩く、業務制約は全てユースケースで判定する。

**問題**:
- ADR-0019 のサブタイプ排他整合や「最低 1 バリエーション」のような**構造的に常に成り立つべき制約**までユースケースに散らばる
- 同じ制約を複数ユースケースで重複実装するリスク
- ドメイン層が貧血モデル（Anemic Domain Model）に近づく

### C. 構造的不変条件と業務イベント制約を区別して分担（採用）

| 制約の性質 | 配置 | 理由 |
|---|---|---|
| **常に成り立つべき構造的不変条件** | 集約ルート（コンストラクタ・mutator） | 集約が壊れた状態を一瞬たりとも作らせない |
| **特定の業務イベント発生時のみ成り立つべき制約** | ユースケース（または Specification パターン） | 業務イベント発生時にチェック。それ以外の状態は許容する |

Issue #284 での適用例:

| 制約 | 性質 | 配置 |
|---|---|---|
| 最低 1 バリエーション (§C1) | 構造的（集約は空になれない） | **集約ルート** `Estimate.create()` で必須 |
| variationNumber 重複禁止 | 構造的（集約は一貫している） | **集約ルート** で全変更時にチェック |
| サブタイプ排他整合 (ADR-0019) | 構造的（型と詳細の整合） | **集約ルート** で attach/detach 時にチェック |
| 集計値が常に最新 | 構造的（不変条件） | **集約ルート + Variation** で自動再計算 (ADR-0028) |
| §3.4 申請バリエーション 1 つ | 業務イベント（申請時） | **ユースケース #6** で判定 |
| 申請時の明細必須 | 業務イベント（申請時） | **ユースケース #6** で判定 |
| 最低 1 明細 | 業務イベント（用途による） | **ユースケース** または **Specification**（ドメインでは items=[] を許容） |

## 決定

不変条件を以下の 2 種類に分類し、配置を分ける:

1. **構造的不変条件**: 集約が**常に**満たすべき条件。コンストラクタ・全 mutator・`reconstruct()` でチェックする。違反は `BusinessRuleViolationError` を即 throw。
2. **業務イベント制約**: 特定の業務行為（申請、確定、出荷等）の発生時にのみ満たすべき条件。ユースケース層または Specification パターンで判定する。違反は「**実行不可な業務操作**」として扱う。

判断軸は以下の質問:

> **「この制約が一時的に破れている状態は、業務上の正当なワークフローとして許容されるか？」**
>
> - YES（許容される）→ 業務イベント制約。ユースケース層
> - NO（許容されない）→ 構造的不変条件。集約ルート

## 根拠

### 業務ワークフローを Domain で阻害しない

§3.4 を集約ルートで禁止すると、「顧客提案中の 3 案を ACTIVE のまま並べて比較する」のような業務ワークフローが書けない。**業務実態に存在するワークフローは Domain で禁止すべきではない**。「申請」は業務イベントであり、その時点で 1 つに絞ればよい。

### 「最低 1 明細」も同様

「ヘッダだけ先に作る」「明細を空にしてからまとめて入れ替える」は実務でよくある操作。集約ルートで「常に明細が必要」と禁止すると、これらのワークフローを擬似トランザクションで偽装する必要が生じる。「**明細が必要**」なのは申請・確定・印刷といった**特定の業務イベント発生時**であり、その時点で Specification を当てれば十分。

### ドメイン層に過剰な責務を持たせない

ドメイン層は「**業務概念のモデル**」であって「**業務ワークフローのオーケストレーション**」ではない。後者はユースケース層の責務。両者を混ぜると:

- ドメインがユースケースごとに肥大化する
- 業務イベントが増えるたびにドメインの制約セットが増減する
- 「いつどの制約が効くか」がドメインから読み取れなくなる

### Specification パターンとの連携

「申請可能性チェック」「確定可能性チェック」のような業務イベント制約は、Specification パターンで実装できる:

```typescript
class CanSubmitSpecification {
  static check(estimate: Estimate): void {
    const activeVariations = estimate.variations.filter((v) => v.isActive());
    if (activeVariations.length !== 1) throw new BusinessRuleViolationError(...);
    if (activeVariations[0].items.length === 0) throw new BusinessRuleViolationError(...);
    // ...
  }
}
```

Specification は Domain Service（ADR-0011）または Policy（ADR-0023）のどちらに配置するかは別途判断（呼び出し前提・依存関係で決まる）。

### Issue #284 deviations §8, §9 との整合

deviations.md §8 (§3.4 を Domain 層で構造的に禁止しなかった) および §9 (集約レベルで「最低 1 明細」を強制しなかった) は、本 ADR の判断軸に基づく。これらを ADR として一般化することで、申請ユースケース実装（着手順序 #6）の方針が明確になる。

### 不採用理由まとめ

- **A（全部 Domain）**: 業務ワークフローを阻害し、結果的にアプリ層で擬似トランザクションを書くことになり複雑度がそちらに移るだけ
- **B（全部 UseCase）**: 構造的に常に成り立つべき制約までユースケースに散らばり、貧血ドメインに近づく

## 影響

### 申請ユースケース（着手順序 #6）の実装方針

§3.4 と「申請時の明細必須」は申請ユースケースで判定する。`SubmitEstimateUseCase` 内、または `CanSubmitEstimateSpecification` クラスで実装する。Domain 層からは「申請可能か？」のクエリメソッドが呼べてもよいが、その判定は Specification に委譲する。

### 既存・将来の集約への適用

- **Order 集約**: 「出荷時の制約」「確定時の制約」など、業務イベント時のみの制約はユースケースに置く
- **Invoice 集約**: 「請求時に項目 0 件不可」も同じ判断軸で UseCase 側

### 集約のテストとユースケースのテストの責務分担

- **集約のテスト**: 構造的不変条件のみを検証する（最低 1 バリエーション、サブタイプ整合、集計の自動再計算、等）
- **ユースケースのテスト**: 業務イベント制約を検証する（申請時に複数 ACTIVE で失敗する、申請時に明細 0 件で失敗する、等）

### ドキュメント上の責任明示

新しい業務制約が業務文書に追加されたとき、「**これは構造的不変条件か、業務イベント制約か**」を判断し、配置を決める。判断に迷う場合は本 ADR の判断軸（業務上の正当なワークフローとして一時的に破れている状態が許容されるか）に立ち返る。

### 例外: 集約境界をまたぐ制約

集約をまたぐ制約（例: 「顧客ごとに 1 つしか進行中の見積を持てない」）はそもそも集約ルートでは判定不能。これは常にユースケース層（または Domain Service）の責務。本 ADR の判断軸の前提として、「集約内で完結する制約」を対象としている。
