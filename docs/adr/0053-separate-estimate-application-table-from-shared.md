# ADR-0053: 申請テーブルを見積・価格共通から見積専用に分離する

| 項目 | 値 |
|---|---|
| ステータス | 採用（ADR-0001 を差替） |
| 起票日 | 2026-06-17 |
| 最終更新日 | 2026-06-17 |

## コンテキスト

ADR-0001 では、承認フローを持つ「見積」と「価格」の申請を **共通の1テーブル**（申請種別＋ポリモーフィックな対象ID）で管理する判断をした。動機は「承認フローのロジック（承認ステップの管理・ステータス遷移・差戻処理）の重複を避ける（DRY）」ことにあった。

issue #361 で見積申請の詳細設計に着手するにあたり、この前提を再評価した。再評価で明らかになった論点は以下である。

1. **「承認ロジックのDRY」と「テーブルの物理共通化」は別問題である。** 本プロジェクトは DDD であり、承認進行ロジック（チェーン構築・1人承認でクリア・全ステップ事前生成・差戻やり直し）は **ドメインサービス／ポリシー**（ADR-0011 / ADR-0023）として共通化できる。テーブルを共通化しなくても DRY は達成できる。
2. **共通テーブルはプロジェクトのスキーマ哲学と逆行する。** 対象IDがポリモーフィック（見積=バリエーションID、価格=価格申請グループID）になるため **DB外部キー制約を張れず**、整合性検証がアプリケーション層に降格する。また見積・価格の種別固有属性が **nullable 化**する。これは NULL 徹底排除方針・CHECK制約による整合性担保（ADR-0021）・サブタイプ平坦化で「集約＝1テーブル」（ADR-0043）と矛盾する。
3. **価格申請のフローは未確定である。** 価格の承認フローは「申請理由区分（値上げ／値下げ／新規追加）でルートが変わる」と方向性のみ示され、詳細は未確定（`docs/business/business-definition.md`）。見積（金額でゴール役職が決まる）とゴール決定ロジックが本質的に異なる。未確定の価格を前提に共通テーブルを今確定すると、後で価格固有要件が判明したときに共通テーブルが歪むリスクがある。

## 検討した選択肢

### A. 共通1テーブル維持（ADR-0001のまま）（不採用）

`applications` / `approval_steps` を見積・価格共通とする。対象IDはポリモーフィックで FK を張れず、種別固有属性は nullable。種別追加は行追加だけで済む反面、上記2・3の欠点を抱える。

### B. ハイブリッド（申請ヘッダ分離＋承認ステップ共通）（不採用）

申請ヘッダは種別ごとに分離し、承認ステップのみ共通テーブルとする。ヘッダの nullable は解消するが、承認ステップ→申請の参照がポリモーフィックに残り FK を張れない。価格未確定リスクも承認ステップ部に残る。

### C. 完全分離・見積専用（採用）

申請・承認ステップを **見積専用テーブル**（`estimate_applications` / `estimate_approval_steps`）として持ち、`variation_id` に通常の DB外部キーを張る。承認進行ロジックはドメイン層のサービス／ポリシーで共通化する。価格申請は将来、価格専用テーブルとして別途設計し、共通化が必要ならその時点でドメインサービスの抽出で対応する。

## 決定

**C を採用する。** 見積申請を `estimate_applications` ＋ `estimate_approval_steps` の見積専用テーブルで表現し、対象（バリエーション）への参照は `variation_id` の DB外部キーで担保する。承認の進行ロジックはドメインサービス／ポリシー（`ApprovalChainBuilder` / `ApprovalRequirementPolicy` など）として実装し、将来の価格申請と共有できる中立な位置に置く。

本 ADR により **ADR-0001 を差替** とする。

価格申請の設計時に、見積・価格の承認進行ロジックの共通部分をドメインサービスとして抽出・共有する。テーブルは種別ごとに分離したまま維持する。

## 根拠

- **DB整合性・NULL排除哲学との整合**: `variation_id` に FK を張れ、種別固有属性に nullable を持ち込まない。アプリ層でのポリモーフィック対象ID検証が不要になる。
- **DRY はコードで達成**: 承認ロジックの共通化はドメイン層で行える。テーブル共通化は DRY の必要条件ではない。
- **不変条件が強くなる**: 見積専用テーブルでは「申請種別」カラムが不要になり、ADR-0005 の `(対象ID, 申請種別, 申請回数)` ユニークは `(variation_id, attempt)` に簡約される。
- **価格未確定リスクの回避（YAGNI）**: 未確定の価格仕様に引きずられず、確定済みの見積仕様だけで最適なテーブルを設計できる。

不採用理由:
- **A**: スキーマ哲学（FK・NULL排除）と逆行し、価格未確定リスクを今抱え込む。
- **B**: nullable は解消するが承認ステップ参照のポリモーフィズムが残り、中途半端。

## 影響

- 申請テーブルに「申請種別」カラムは不要になる（見積専用のため）。
- 対象IDのアプリケーション層検証が不要になり、FK で整合性を担保できる。
- 価格申請の設計時に、承認進行ロジックの共通化（ドメインサービス抽出）を改めて行う必要がある。テーブルは分離のまま。
- 「承認免除」の表現は別 ADR（ADR-0054）で扱う。
- 見積側システム設計書 §1.3 のステータス導出は、見積専用申請テーブルを参照する形に確定する。
