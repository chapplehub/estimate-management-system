# ADR-0038: 失敗はすべて例外に統一し、判別共用体は「複数の正常な結末」の戻り値設計に限定する

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-10 |
| 最終更新日 | 2026-06-10 |
| 置き換え対象 | [ADR-0037](0037-application-command-result-union-for-predictable-outcomes.md) |

## コンテキスト

ADR-0037 は「予測可能な業務結果は判別共用体(Result)で返し、想定外のみ throw」と定めたが、線引きの文言「**同一リクエスト内でユーザーが入力を直して解決可能 → Result**」には曖昧さがあった。

この文言を字義通り適用すると、社員コード重複（`ConflictError`）や空見積不可（`BusinessRuleViolationError`）も「ユーザーが入力を直せば解決可能」なので Result になるはずだが、実装はすべて throw であり、**文言と実装が起票当初から乖離していた**。また「例外は技術的例外にのみ使い、ビジネス例外には使うべきでない」という一般論と、値オブジェクトが `ValidationError` を throw する現実装との関係も未整理だった。

そこで ADR-0037 の結論をいったん白紙とし、「そもそも Result を導入すべきか、Result に統一すべきか、例外に統一すべきか」から再検討した。

## 検討した選択肢

### A. 失敗はすべて例外。判別共用体は「複数の正常な結末を持つコマンド」の戻り値設計に限定（採用）

エラー処理の道具は例外のみ。判別共用体（`TaxCheckedSaveResult` 等）は**エラー処理機構ではなく、複数の正常な結末を持つコマンドの戻り値の設計**と再定義する。

```typescript
// 「保存された」「税率確認が必要」はどちらも失敗ではなく、業務上ありうる結末
export type TaxCheckedSaveResult =
  | { kind: "saved"; estimate: Estimate }
  | { kind: "taxRateMismatch"; estimateDateRate: TaxRate; deadlineRate: TaxRate };
```

### B. ビジネス的失敗をすべて Result 化する（技術 vs ビジネス軸）（不採用）

「例外は技術的例外のみ」の原則に従い、VO 検証失敗・ビジネスルール違反・NotFound 等のドメイン語彙で説明できる失敗をすべて Result で返す。

### C. 文字通りの全例外統一（不採用）

税率不一致も `TaxRateMismatchError` として throw し、判別共用体を廃止する。

## 決定

**失敗はすべてカスタム例外を throw する。業務上ありうる複数の正常な結末を持つコマンドは、命名付き判別共用体を戻り値で返す。汎用 `Result<T, E>` 型は導入しない（ADR-0037 を踏襲）。**

判定質問はただ一つ：

> **「それは失敗か、それとも業務上ありうるもう一つの結末か」**

- **失敗 → throw**。事象が予測可能か、技術的かビジネス的か、ユーザーが入力を直せば解決するか——これらは**判定に使わない**。
- **もう一つの結末 → 命名付き判別共用体**。

迷う場合の補助判定：

| 観察 | 分類 |
|---|---|
| コマンドが**意図的に効果を保留**し、呼び出し側に表示以外の**継続処理**（確認フロー・代替案提示など）がある | 結末（共用体） |
| コマンドが**中断**し、呼び出し側はメッセージを表示して終わるだけ | 失敗（throw） |

### 付随する取り決め

1. **値オブジェクトの throw は契約違反検出である**: プレゼンテーション層の Zod + Conform が形式検証の一次防衛線であり、VO の `ValidationError` 発火は「Zod を通ったのに不正な値が来た」＝プログラミングエラーの検出。assertion に近く、例外の正当な使いどころ。「ビジネス例外に Error を使っている」状態ではない。
2. **ドメイン例外メッセージはユーザー向け品質で書く**: 業務的失敗は `handleCommandError()` がメッセージをそのまま UI に表示するため、ドメイン層はユーザーが読める日本語メッセージの責務を負う。仕様書参照（`§C1` 等）は問い合わせ時にルールを特定できる実益があるため残してよい。エラーコード→文言の変換辞書は導入しない。
3. **分類は UX の進化で昇格しうる**: 現在 throw の失敗（例: 採番衝突）に専用 UI フロー（例: 空き番号の提案）ができたら、その時点で共用体の結末へ昇格させる。基準は不変のまま、個々の分類だけが変わる。
4. **`〜Result` サフィックスは継続**: `TaxCheckedSaveResult` 等の命名は「業務結末の共用体」を意味する規約として維持する（汎用 Result 型の意味ではない）。

## 根拠

- **判別共用体の対価が報われる条件で限定する**: 共用体の唯一の対価は「全呼び出し元に型レベルで分岐を強制する」こと。これが報われるのは**分岐先に書くべきロジックが実在するとき**だけ（税率不一致 → 確認フローのコードを書き忘れると業務が壊れる）。「表示して終わり」の失敗に共用体を使うと、同一の表示処理が全 Server Action に複製されるだけで、強制した分岐に書く中身がない。throw なら `handleCommandError()` が一括処理できる。
- **選択肢 B の不採用理由**: (1) 全 VO のコンストラクタを static ファクトリ + Result 化する大改修になる。(2) TypeScript には Result の言語サポート（`?` 演算子・do 記法）がなく、合成ユーティリティの自作か neverthrow 導入が必要で「小手先」では済まない。(3) try-catch → `handleCommandError()` → `ActionResult` という既存の集約変換構造と二重路線になる。(4) ADR-0037 から運用した実績上、共用体が必要になったのは税率クラスタ 1 箇所のみであり、「Result が活きる場面」はこのドメインでは稀だと実証された。
- **選択肢 C の不採用理由**: 税率不一致は失敗ではなく「保存を保留した正常な業務結果」であり、例外化は正常フローの例外制御という最悪のパターンになる。catch 忘れが型で検出されず、汎用メッセージ「処理に失敗しました」に握りつぶされて UX が静かに壊れる。
- **「ユーザーが直せるか」基準の廃棄理由**: ユーザーが直せる失敗（重複コード等）でも呼び出し側に継続処理がなければ共用体の対価は報われない。「直せるか」は事象の性質であり、戻り値設計を決めるのは**呼び出し側の振る舞い**である。

## 影響

- ADR-0037 を置き換える。ただし 0037 の実装上の結論（`TaxCheckedSaveResult`・throw の使い分け・汎用 Result 不導入）はすべて存続し、変わるのは**判定基準の言語化**のみ。既存コードの変更は不要（現実装は本基準にすべて適合している）。
- 今後のコマンド実装では「失敗か、もう一つの結末か」だけを問う。新たな共用体の追加は「専用 UI フローの存在」が根拠になる。
- `handleCommandError()` が業務的失敗の表示経路として公式化される。ドメイン例外のメッセージ品質（ユーザーが読めること）がレビュー観点になる。
- テストは従来どおり `expect().toThrow(ErrorClass)` / `.rejects.toThrow()` を継続する。
- 楽観ロック競合（#301）は本基準で再判定する: 競合時に専用 UI フロー（差分表示・再読み込み誘導など）を設けるなら共用体、表示して終わりなら `ConflictError` の throw。
