# ADR-0064: 見積単価は販売単価マスタを唯一の源とし、手入力を廃して掛率・値引のみで調整する

| 項目 | 値 |
|---|---|
| ステータス | 採用（ADR-0046/0060 の編集可能集合から単価を取り下げ） |
| 起票日 | 2026-06-22 |
| 最終更新日 | 2026-06-22 |

## コンテキスト

#422 で見積の価格決定機構の輪郭を定めるにあたり、「商品を選択したときに見積単価をどう定めるか」が論点になった。現状、見積単価は手入力・編集可能である。

- 単価変更の委譲鎖 `Estimate.changeItemUnitPrice` → `EstimateVariation.changeItemUnitPrice` → `EstimateItem.changeUnitPrice`（および改訂先一括調整の `EstimateVariation` bulk 経路）／ `EstimateItem` 生成時の `unitPrice` 受け取り（通常明細・C4 編集経路）
- 改訂先バリエーションの粒度別調整 `AdjustRevisedVariationCommand`（#390）。ADR-0046/0060 は編集可能集合に「単価」を含めている

一方 #422 は、販売単価マスタ（共通／得意先別／納品先別の3レイヤー）から見積年月日・宛先で売単価を解決する**価格決定**機構を導入する。ここで「価格決定が出した売単価は上書き可能な初期値か、固定される確定値か」を決める必要があり、既存の「単価は手入力で動かせる」設計と正面から衝突する。

## 検討した選択肢

### A. 単価固定（価格決定が唯一の源・手入力廃止・調整は掛率／値引）（採用）

見積単価は価格決定が販売単価マスタから解決した値で確定し、明細上では固定する。担当者は単価を編集できず、金額の調整は掛率・明細値引・全体値引で行う。通常明細・改訂先の双方に一貫適用する。

### B. 初期値供給（マスタ由来だが明細ごとに上書き可）（不採用）

価格決定の出力を初期値として流し込み、担当者が明細ごとに上書きできる。既存の「都度入力」文化を残せるが、価格の源がマスタと手入力の二重になる。

## 決定

見積単価は価格決定が販売単価マスタから解決した値で確定・固定し、手入力での修正を許さない（A を採用）。金額調整は掛率・明細値引・全体値引のみで行う。通常明細・改訂先に一貫適用し、ADR-0046/0060 と #390 が認めていた**単価調整を取り下げる**。

## 根拠

- 監査可能な単一の価格源（販売単価マスタ）に売価を集約できる。opaque な単価手入力を排し、調整を透明な掛率・値引の列に寄せることで、金額の根拠が追える。
- B 案は価格源がマスタ＋手入力の二重になり、「この単価はマスタ由来か手で変えたのか」が判別できず監査・整合が崩れる。
- 文脈（通常明細／改訂先）ごとに単価可否が割れると学習コスト・バグ温床になる。一貫適用が単純で確実。
- 掛率・値引で非負制約内の任意の行金額に到達できるため、粗利目標の調整能力は失われない。改訂先も数量固定（ADR-0060）下で掛率・値引のみで粗利合わせが成立する。

## 影響

- 単価変更の委譲鎖を一括で撤去する: `Estimate.changeItemUnitPrice` → `EstimateVariation.changeItemUnitPrice` → `EstimateItem.changeUnitPrice`（および改訂先一括調整の bulk 経路）、`EstimateItem` 生成時の `unitPrice` 手入力、通常編集（C4）の単価入力、`AdjustRevisedVariationCommand`（#390）の単価入力。委譲元・委譲先のどちらかだけ消し残さないよう鎖全体を対象とする（出荷済みコードの差し替え。#351 に前例あり）。
- ADR-0046/0060 の改訂先編集可能集合は「掛率・明細値引・全体値引・メモ」に縮小（単価を除外）。両 ADR には本決定への改訂注記を付す。
- 価格決定が走る契機は「明細追加／見積複製先生成／得意先改訂先生成／見積年月日・宛先変更」。いずれも単価を（再）解決し、見積年月日に有効な販売単価が無く解決不能なら操作を拒否する（暫定値・0 円明細を作らない）。
- 改訂先の単価は改訂元から引き継がず、改訂先の提出区分で価格決定を再解決する（得意先別販売単価→共通）。数量固定（ADR-0060）と併せ、改訂先は単価・数量とも固定、調整は掛率・値引のみ。
- セット構成明細も価格付き末端行として価格決定の対象（セット群は自前の価格を持たないため対象外）。
- **UI 要件（価格決定の実装 Issue で拾う）**: 見積年月日・宛先の変更のように既存明細の単価が一斉に再解決され値が変わりうる操作では、再解決を実行する**前に**ユーザへ確認・警告を出す。単価が手入力できないぶん、価格が黙って変わる事故を防ぐためである。本 ADR のスコープは輪郭定義（#422）であり、この確認 UI は価格決定を実装する Issue の受け入れ要件とする。
- 本決定は価格決定機構の前提。販売単価マスタのスキーマ（3レイヤー・適用期間・NULL 排除/番兵）の確定は別 Issue（ADR-0065 と同じ実装フェーズ）に送る。
