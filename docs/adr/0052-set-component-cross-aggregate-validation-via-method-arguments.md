# ADR-0052: セット構成の集約越え検証はドメインポートではなく ADR-0030（メソッド引数）で行う

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-16 |
| 最終更新日 | 2026-06-16 |

## コンテキスト

見積詳細画面のセット明細編集（issue #333 / S5）で、セット群の構成明細が満たすべき集約越えのルールを書き込み経路で検証する必要がある。検証対象は次の 2 つ:

- **構成商品の区分は個別商品・消耗品のみ**（セット商品を構成にできない＝ネスト禁止）。ハードエラー。
- **無効化された構成商品**（`Product.isActive === false`）の検出。非ブロッキング警告（無効でも追加・保存は許す。CONTEXT.md「セット商品」／商品設計書 §5.2）。

これらは `Estimate` 集約が単独で判定できない。区分・有効性は **Product 集約**の事実であり、`Estimate` 集約は Product に依存しない（集約境界）。

ADR-0047 の補足は、この検証を「`estimate/domain/services/` のルール検証型 Domain Service を**ドメインポート経由**で実装（実装は S5）」と前約束していた。しかし、このリポジトリには **ADR-0030（pass-cross-cutting-context-via-method-arguments）** という確立した方針が既にある — 「集約をまたぐ文脈は、アプリ層が事実を集めてドメインのメソッド引数で渡す」。ADR-0047 補足はこの ADR-0030 を踏まえずに「ポート」を選んでいた。

加えて探索の結果、このコードベースには**集約をまたぐドメインポートの前例が 1 つも無い**ことが分かった（`TaxRateConsistencyCheckDomainService` は estimate 内の `TaxRateRepository` に閉じ、`EstimateDuplicationService` も estimate 内で完結）。ADR-0047 補足を字義どおり実装すると、S5 が「最初の集約越えドメインポート」という重い前例を新設することになる。

なお、商品事実は**作成中は read-through（ライブ）**で扱い、申請時以降にスナップショット凍結する（ADR-0048）。S5/C4 はすべて作成中の局面なので、検証はライブな Product 事実に対して行う。

## 検討した選択肢

### A. ADR-0030 方式 — アプリ層が事実を集め、純粋ドメインに引数で渡す（採用）

アプリ層コマンド（`UpdateVariationCommand` / `CreateEstimateCommand`）が、セット群の全 `productId`（群本体＋構成）について `ProductQueryService`（product サブドメインの query）で区分・`isActive` をライブ取得し、純粋なドメイン検証（`SetComponentRule` 関数 / 引数受け取り型 Domain Service）へ method-args で渡す。

```ts
// アプリ層: 事実を集める（read-through・ライブ）
const facts = await productQueryService.resolveCategories(productIds); // { id → {category, isActive} }
// ドメイン: 事実を引数で受け取り、純粋に判定する
SetComponentRule.assert(setGroups, facts); // 区分外はハードエラー / 無効は warning を集約
```

- 新しい集約越えポートを導入しない（ADR-0030 の前例どおり）。
- 自動展開しか UI が無くても、保存時のライブ検証が**ペイロード防御**として効く。

### B. ADR-0047 補足どおりドメインポート（不採用）

`estimate/domain` に Product 区分照会のポート interface を定義し、infra が product を呼んで実装する。

- ADR-0047 補足の前約束に忠実。
- しかし ADR-0030 と非対称な新パターン（集約越えポート）を増やし、「最初の集約越えポート」という前例を S5 で背負う。

## 決定

**セット構成の集約越え検証は選択肢 A（ADR-0030 方式）で行う。** アプリ層コマンドが `ProductQueryService` で区分・`isActive` をライブ取得し、純粋なドメイン検証へメソッド引数で渡す。構成商品の区分外（ネスト含む）はハードエラー、無効構成は非ブロッキング警告として結果に集約する。ADR-0047 補足の「ドメインポート」は採らない。

## 根拠

- **既存方針との整合**: 同じ「集約越えの事実参照」を、リポジトリ確立済みの ADR-0030 で実現でき、「ポート」という新概念を増やさない。集約越えドメインポートの前例がゼロのコードベースに、その前例を S5 で作らずに済む。
- **ドメインの純粋性を保つ**: 検証ロジックはドメイン層の純粋関数のまま（Product 事実は引数で受け取るのみ）。DDD レイヤリング規約（ドメインは他層・外部に依存しない）を破らない。
- **作成中はライブ検証（ADR-0048）**: 商品事実はスナップショット凍結（申請時以降）まで read-through。S5 は作成中の局面なので、保存時にライブ取得して検証するのが正しい。
- **不採用理由（B）**: ADR-0047 補足は ADR-0030 を踏まえていなかった。ポートは ADR-0030 と非対称で、集約越えポートの初導入という重い設計判断を伴う。

## 影響

- **ADR-0047 補足の上書き**: ADR-0047 補足が指定した「ドメインポート経由のルール検証型 Domain Service」は本 ADR で**メソッド引数方式に差し替える**。ADR-0047 補足末尾に本 ADR へのポインタを追記する。
- **手動構成追加の繰り延べ**: 区分/ネスト違反が **UI から到達可能になるのは構成明細の手動追加経路**のみ（自動展開はマスタ由来で常に正しい）。手動構成追加は S5 のスコープ外とし issue #350 へ繰り延べた。S5 は本検証を**ペイロード防御**として導入する（自動展開経路でも保存時にライブ検証が効く）。
- **無効構成の警告は read 関心**: 非ブロッキング警告は UI で状態導出する（インライン）。そのため `LineDTO` に `isActive`（read-through）を追加し、展開時・再編集時のどちらでも作業コピーの `isActive` から導出する。
- **関連**: ADR-0030（集約越え文脈は method-args）、ADR-0047 補足（差し替え元）、ADR-0048（作成中は read-through・申請時スナップショット）、issue #333（S5）、#350（手動構成追加）。
