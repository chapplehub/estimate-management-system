# ADR-0044: 改訂系譜は集約の内側に置き、凍結は系譜からの導出状態とする

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-12 |
| 最終更新日 | 2026-06-12 |

## コンテキスト

C7 ReviseForCustomer（得意先改訂・#296）では、改訂元の納品先宛バリエーションを「凍結」（メモ以外編集・行追加削除不可）する必要がある（設計書 §7.2）。この凍結状態をどう表現するかが問題になった。

似た構造を持つ複製系譜（`EstimateVariationCopy`）は ADR-0040 で「集約外の兄弟成果物」として扱い、`insertWithCopies` で永続化すると決めた。改訂系譜（`EstimateVariationRevision`）はスキーマ上ほぼ対称な交差テーブルであり（§11.2.1）、同じ扱いにするのが一見自然に見える。しかし凍結は集約が守るべき編集ガード（不変条件）であり、その根拠データが集約の外にあると、集約は再構築時に自分の不変条件を知り得ない。

## 検討した選択肢

### A. 凍結を系譜から導出し、改訂系譜を集約の内側に置く（採用）

「同一見積内に、自分を改訂元とする改訂系譜が存在する ⟺ 凍結」。`EstimateVariation` が自分の出自（`revisedFrom`: 改訂元 ID、高々1）を保持し、集約ルートが横断して凍結を導出する。永続化は既存の `EstimateVariationRevision` テーブルへ集約保存（`update` の差分 upsert・ADR-0032 の延長）の一部として書き込む。

### B. `VariationStatus` に FROZEN を追加（不採用）

enum 拡張のマイグレーションが必要な上、「凍結かつ無効」が表現できない。凍結（内容の保全）と有効/無効（検討対象か否か）は直交概念であり、1 軸に潰すと歪む。

### C. boolean 列 `isFrozen` を追加（不採用）

系譜と二重管理になり、「系譜はあるのに凍結フラグが立っていない」という不整合状態が表現可能になる。

### D. 複製と同様に兄弟成果物として集約外に置き、凍結はフラグで持つ（不採用）

ADR-0040 との対称性は得られるが、凍結ガードの根拠が集約外に出るため、結局 B か C のフラグを併設することになり C の二重管理問題を引き継ぐ。

## 決定

改訂系譜は Estimate 集約の内側に属するデータとして保持し、凍結はその存在から導出する（A を採用）。

## 根拠

- **集約境界の原則**: 複製系譜は複製元と複製先という**2つの集約インスタンスにまたがる**ためどちらにも属さない（ADR-0040 の前提）。改訂系譜は改訂元・改訂先とも**同一集約インスタンス内**にあり、前提が異なる。同じ交差テーブルでも所属が逆になるのは原則に忠実な帰結であり、表面的な対称性より整合性境界を優先した。
- **不変条件は境界の内側で**: 凍結ガード（C4 全置換拒否・明細操作拒否）は集約が単独で強制できる必要がある。系譜が内側にあれば再構築だけで判定できる。
- **導出によるNULL/二重管理排除**: 凍結は「改訂された」という事実の帰結であり、解除という操作が存在しない。状態として保存しないことで凍結・解除 API も不整合状態も構造的に生まれない。スキーマ変更もゼロ。
- 副次的な帰結として、改訂先バリエーションを削除すると系譜が消え、改訂元の凍結が自動的に解ける（改訂のやり直しが特別な操作なしに成立する）。

## 影響

- リポジトリに `insertWithRevisions` のような専用メソッドは増えない。`update`（差分 upsert）が改訂系譜行の挿入を担う。
- マッパーは集約の読み込み時に `revisionTarget` を `EstimateVariation.revisedFrom` へ写像する必要がある。
- 「改訂以外の理由で凍結したい」要求が将来生まれた場合、導出では表現できず本決定の見直しが必要になる（現設計書にそのような要求はない）。
- 複製系譜（集約外）と改訂系譜（集約内）で扱いが非対称になる。混同しないよう CONTEXT.md の用語定義（改訂系譜・凍結）に所属の違いを明記した。
