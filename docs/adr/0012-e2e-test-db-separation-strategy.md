# ADR-0012: E2Eテスト専用DB環境の構築方針

| ステータス | 採用 |

## コンテキスト

E2Eテストのデータ管理に以下の問題がある。

- テストデータをUI操作で作成 → 遅い、前提テストの失敗で連鎖
- 開発用Seedデータへの依存 → Seed変更でテストが壊れる
- afterEachでUI経由の削除 → 失敗時にゴミデータ蓄積

テスト専用DB環境を導入するにあたり、以下の3点で設計判断が必要になった。

1. テストサーバーのポート
2. 環境変数ファイルの管理方法
3. DB初期化のタイミング（Playwright setup project vs 外部スクリプト）

また、#217 で Playwright テスト内から Prisma クライアントで直接DB操作する方式を検討したが、Prisma 7.x の TypeScript 生成クライアントが `import.meta.url` を使用しており、Playwright の esbuild トランスパイラと ESM/CJS の不整合が発生するため断念した。テストデータ管理の代替方式について追加の設計判断が必要になった。

4. テストデータのセットアップ・クリーンアップ方式

## 検討した選択肢

### 判断1: テストサーバーのポート

#### A. 開発と同じ port 3000（不採用）

シンプルだが、開発サーバー起動中に `pnpm e2e` を実行すると `reuseExistingServer` により開発DBに対してテストが走る。テスト実行前に開発サーバーの停止が必要。

#### B. テスト専用 port 3001（採用）

開発サーバー(3000)とテストサーバー(3001)が共存可能。開発サーバーを止めずにテスト実行・UIモードでのデバッグが可能。

### 判断2: 環境変数ファイルの管理

#### A. `.env.test` をリポジトリにコミット（不採用）

全開発者が同じ設定を使えるが、PostgreSQLの接続情報（ユーザー名・パスワード）はローカル環境に依存するため、統一が困難。

#### B. `.env.test.example` をコミットし、`.env.test` は各開発者が作成（採用）

既存の `.env` / `.env.example` パターンと統一的。接続情報をローカル環境に合わせて設定可能。

### 判断3: DB初期化のタイミング

#### A. Playwright setup project（`setup db`）を使用（不採用）

Playwright公式ドキュメントで推奨されるproject dependenciesパターン。テスト実行のたびにDB初期化が走る。HTMLレポートやトレースでセットアップ状況が確認可能。

#### B. 外部スクリプト `pnpm e2e:setup` で明示的に実行（採用）

DB準備とテスト実行を分離。開発者が任意のタイミングでDB状態をリセットできる。

### 判断4: テストデータのセットアップ・クリーンアップ方式

#### A. Prisma クライアントで直接DB操作（不採用）

テスト内から Prisma クライアントを使い、`beforeEach` / `afterEach` でレコードの作成・削除を行う。データ操作が高速かつ確実だが、Prisma 7.x の TypeScript 生成クライアントが `import.meta.url` を使用しており、Playwright の esbuild トランスパイラでは ESM/CJS の不整合エラーが発生するため技術的に実現不可。

#### B. CRUD テストの直列化（`test.describe.serial`）でデータ管理（採用）

副作用のあるテスト（create → update → delete）を `test.describe.serial` で直列実行し、create テストがデータセットアップ、delete テストがクリーンアップを兼ねる。副作用のないテスト（参照系・権限テスト等）は並列実行のまま維持する。

## 決定

テストサーバーは port 3001 で分離、環境変数は `.env.test.example` テンプレート方式、DB初期化は外部スクリプト `pnpm e2e:setup` で行う。テストデータ管理は CRUD テストの直列化（`test.describe.serial`）で対応する。

## 根拠

- **port 3001**: UIモードを頻繁に使用するワークフローにおいて、開発サーバーとの共存が不可欠。port分離のコストは低い（設定変更のみ）。
- **`.env.test.example`**: PostgreSQL接続情報がローカル環境に依存する（peer auth/password auth、ユーザー名等が開発者ごとに異なる）ため、テンプレート方式が適切。既存パターンとの統一性も確保。
- **外部スクリプト**: Playwright setup projectは毎回のテスト実行時にDB初期化が走り、UIモードでの繰り返し実行時に煩わしい。外部スクリプトなら必要なときだけ明示的に実行でき、テスト実行のオーバーヘッドがない。
- **CRUD 直列化**: Prisma 7.x の TypeScript 生成クライアントは `import.meta.url` を使用しており、Playwright の esbuild トランスパイラでは ESM/CJS 不整合が発生するため、テスト内からの直接DB操作は断念。代わりに `test.describe.serial` で CRUD テストを直列化し、create がセットアップ、delete がクリーンアップを兼ねる方式を採用。副作用のないテスト（参照系・権限テスト）は並列実行を維持し、テスト実行速度への影響を最小限に抑える。

## 影響

- 開発者は初回セットアップ時に `cp .env.test.example .env.test` + 接続情報の編集が必要
- テストデータが汚れた場合は `pnpm e2e:setup` の再実行が必要（自動リセットはされない）
- CIでは環境変数を直接設定するため `.env.test` は不要
- CRUD テスト（create → update → delete）は `test.describe.serial` で直列実行されるため、途中のテストが失敗すると後続テストはスキップされる
- シリアルチェーン途中の失敗でテストデータが残留した場合は `pnpm e2e:seed` で再シードが必要
