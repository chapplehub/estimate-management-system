# ADR-0039: 楽観ロックは集約ルートの version 列＋フォーム往復＋リポジトリ insert/update 分割で横断適用する

| 項目         | 値         |
| ------------ | ---------- |
| ステータス   | 採用       |
| 起票日       | 2026-06-11 |
| 最終更新日   | 2026-06-12 |

## コンテキスト

Issue #301。更新系コマンドは「findById → メモリ上で変更 → save」型で、保存時の競合チェックが無く、2人が同じ集約を同時編集すると後勝ちで片方の変更が静かに消える（lost update）。

被害は上書き型コマンドに限らない。ADR-0032 の差分 upsert は「メモリ上の集約 = 真実」として集約に無い子を `deleteMany` で消すため、**追加型コマンド（AddVariation 等）でも stale な集約からの保存が、並行して追加された他人の子を物理削除しうる**。

守るべき競合ウィンドウは2つある。

```
ユーザーAが画面を開く ──── 数分間編集 ──── 保存 → findById → save
                          ↑ 編集ウィンドウ            ↑ レースウィンドウ
                     （lost update の本体）          （数ミリ秒）
```

コマンド内で findById した version を保存条件に使う設計は、findById が常に最新値を読んでしまうためレースウィンドウしか守れない。編集ウィンドウを守るには「ユーザーが画面で見た時点のトークン」を保存時まで運ぶ必要がある。

## 検討した選択肢

### 1. 楽観ロックトークンの実体

#### A. 集約ルートに `version Int @default(1)` を新設（採用）

整数の等値比較は決定的で、シリアライズ往復で壊れる余地がない。専用列なので意図がスキーマ上明示され、監査列と責務が分離される。

#### B. 既存の `updatedAt` を流用（不採用）

マイグレーション不要だが、等値比較の成立が「全リンクが ISO 8601 ミリ秒精度を保つ」という暗黙の規律に依存する。将来誰かがタイムスタンプ整形を1箇所変えると全保存が偽競合し、型エラーにもテスト失敗にもならない。同一ミリ秒内の2更新を区別できない理論穴と、監査列との兼務（touch と競合検知の要件衝突）も残る。

#### C. PostgreSQL システム列 `xmin` を流用（不採用）

列追加不要で raw SQL 経由の更新も検知できるが、Prisma スキーマで表現できず読み書き両経路に `$queryRaw` が侵食する（`ESTIMATE_FULL_INCLUDE` の多階層 include と両立しない）。pg_dump/リストア・VACUUM FREEZE で値が変わり、データが無変更でも偽競合が一斉発生する。

### 2. 期待 version の通り道

#### A. リポジトリメソッドの引数で渡す（採用）

フォーム（hidden input）→ コマンド Input → `update(aggregate, expectedVersion)` と素通しし、リポジトリの条件付き UPDATE 1箇所でチェックする。

```typescript
const result = await tx.estimate.updateMany({
  where: { id: estimateId, version: expectedVersion }, // フォーム由来の値で直接条件更新
  data: { ...scalar, version: { increment: 1 } },
});
if (result.count === 0) throw new ConflictError("...");
```

WHERE がフォームの値を直接使うため、「比較→更新」が DB 上で原子化され、編集ウィンドウとレースウィンドウを1回の条件付き UPDATE で同時に守る。

#### B. ドメインエンティティに version を保持（不採用）

エンティティの version は findById がたった今読んだ最新値なので、DB の WHERE 条件としては編集ウィンドウを検知できない。同等の保護には「コマンド層でフォーム値とエンティティ値を比較」＋「リポジトリの条件付き UPDATE」の2段チェックが必要になり、早期チェックの書き忘れを型が守らない。また version は業務概念ではなく永続化の並行制御メタデータであり、ドメインモデルに載せない方が筋が良い（横断コンテキストを引数で渡す ADR-0030 の延長）。

### 3. 渡し忘れの防御（リポジトリのシグネチャ）

#### A. `insert(aggregate)` / `update(aggregate, expectedVersion)` にメソッド分割（採用）

`save` を廃止。更新経路で expectedVersion を渡さないコードはコンパイルエラーになる。副産物として save 冒頭の存在確認 `findUnique` プローブが消え、各メソッドが一直線になる。呼び出し側の意図（新規/既存）と DB の現実がズレた場合も暗黙の create/update に化けず明示的にエラーになる。

#### B. `save(aggregate, options?: { expectedVersion })` のオプショナル引数（不採用）

更新コマンドで渡し忘れてもコンパイルが通り、楽観ロックが静かに無効化される。「何も書いていない」誤りはレビューでも目立たない。

#### C. `save(aggregate, expectedVersion: number | null)` の必須引数（不採用）

意識的な宣言を強制できるが、1つの関数が引数の値で create/update に分岐する設計になる。メソッド分割の方が各経路が単純で、関数に処理分岐の状態を持たせない方針とも整合する。

## 決定

**既存集約を変更する全コマンドに楽観ロックを横断適用する。トークンは集約ルートの `version Int @default(1)` 列、期待値は編集画面表示時の version をフォームで往復させ、リポジトリを `insert` / `update(aggregate, expectedVersion)` に分割して条件付き UPDATE（`WHERE id AND version` + `version: { increment: 1 }`）で原子的にチェックする。不一致（count = 0）は `ConflictError` を throw する。**

### 横断ポリシーの細目

1. **version 列は集約ルートのみ**。子テーブルへの書き込みは必ずルートの update を通る（ADR-0027/0032）ため、集約全体（子・孫・1:1 詳細まで）がルートの version 1つで守られる。子に持たせると「部分競合」という新しい複雑さを生むだけで保護は増えない。
2. **対象は既存集約を変更する全コマンド**。上書き型（UpdateXxx）だけでなく、追加型（AddVariation）・状態変更型（Activate/Deactivate/SetXxx）も含む。差分 upsert の `deleteMany` がある限り「上書きしないから安全」は成立しない。除外は新規作成（insert 経路）のみ。
3. **Delete 系もポリシー上は対象**（`deleteMany({ id, version })` の count 検査）。stale な画面を見て下した削除判断も競合の一種。実装優先度は更新系より下げてよい。
4. **競合は `ConflictError` の throw**。ADR-0038 の基準で「メッセージを表示して終わり」の失敗に分類。専用サブクラスは作らず既存 `ConflictError` を再利用する。将来、差分マージ等の専用 UI フローを設ける場合は ADR-0038 の昇格ルールに従い共用体の結末へ昇格させる。
5. **count = 0 は「version 不一致」と「行の消失」を区別しない**。区別には再 SELECT が要るが、コマンドが直前に findById で NotFound を弾くため残る窓は極小。メッセージは両方を覆う文言とする：「他のユーザーによって更新または削除されています。画面を再読み込みして最新の内容を確認してください。」（ADR-0038 のユーザー向け品質要件）。
6. **Prisma 実装は `updateMany` を使う**。単数形 `update` は WHERE にユニーク列しか書けず、version を条件に足せないため。返り値の `{ count }` で成否を判定する。
7. **ルートを経由しない横断一括書き込みは、影響を受けた全ルートの version を無条件増分する**（#306）。`replaceInRelationsAndComponents`（入れ替えつき無効化）のように複数集約の子行を直接書き換えるシステム操作は、細目1の前提「子への書き込みは必ずルートの update を通る」の例外となる。version を進めないと、参照元商品の編集フォームを開いていたユーザーの stale な保存（差分 upsert の deleteMany）が入れ替え結果を静かに巻き戻す。同一トランザクション内で影響ルートへ `version: { increment: 1 }` を打つ（ユーザー編集ではないため expectedVersion チェックは行わない）。なお、この種のコマンドが持つ複数トランザクションの非原子性は楽観ロックとは独立の既存リスクとして受容する。ConflictError は必ず先頭の条件付き update で発生するため新しい中間状態は生まれない（残余リスクは #322）。

## 根拠

- **整数 version は失敗モードを型レベルで消す**: updatedAt 流用の事故確率は低いが、起きたときの症状（全保存が偽競合）と原因（整形1行）の距離が遠くデバッグコストが高い。楽観ロックの存在意義は等値比較の絶対的信頼であり、それを規律ではなく構造で担保する。
- **偽競合はユーザーの信頼を摩耗させる**: 偽競合が頻発すると「このエラーは無視して再保存」が運用に染みつき、本物の競合警告まで無視される。トークンには「データへの正当な更新でのみ変わる」性質が必須で、xmin はこれを満たさない。
- **フォーム往復だけが編集ウィンドウを守れる**: lost update の本体は数分間の編集ウィンドウで起きる。コマンド内 findById の version を使う設計は本イシューの受け入れ条件「後勝ちによる静かな変更喪失が起きない」を実質満たせない。
- **チェック1箇所・型強制が横断適用の前提**: 7サブドメインに統一適用する以上、コマンド側に「比較を書く規律」を要求する設計は必ず漏れる。リポジトリ1箇所の原子的チェック＋シグネチャによる強制で、規律をゼロにする。
- **増分漏れリスクの受容**: Prisma を経由しない直接 UPDATE は version を進めず検知から漏れるが、書き込みは全てリポジトリ経由という DDD レイヤリング（ADR-0027）が既に強制されており、現実の露出は無視できる。

## 影響

- **スキーマ移行**: 集約ルート7テーブル（estimate / customer / employee / product / delivery_location / role / department）に `version Int @default(1)` を一括追加する。列は使われるまで不活性なため、コード適用はサブドメインごとに段階展開できる。
- **リポジトリインターフェース**: 7サブドメインの `save` を `insert` / `update` に分割。`PrismaEstimateRepository` は既存の private `update()`（ADR-0032 の差分 upsert）を公開に昇格し、トランザクション先頭のルート更新を条件付き `updateMany` に置き換える。version チェックはトランザクション内の最初の文として実行し、count = 0 なら throw して全体をロールバックする。
- **プレゼンテーション層**: 全編集フォームに version の hidden input、Zod スキーマに `version` 追加。クエリ側 DTO に version を含める。追加型・状態変更型の画面（一覧のボタン含む）も親集約の version を持ち回る。
- **段階展開**: #301 では7テーブル一括マイグレーション＋ estimate 縦切り（C2/C3/C4/AddVariation・フォーム・テスト）＋本 ADR を実装し、マスタ系6サブドメインと Delete 系は後続イシューで適用する。受け入れ条件「全 Update 系コマンドへの統一適用」との差分はイシュー側で明示する。
- **テスト**: 実 DB のリポジトリ統合テストで stale トークンを**逐次**再現する（insert → update(v1) 成功 → update(v1) が ConflictError、先行変更が残存）。防御の実体は条件付き UPDATE 1文の原子性（PostgreSQL 保証）なので、`Promise.all` による真の並行テストは flaky の割に証明力が増えず、主役にしない（ADR-0035 と同じ思想）。コマンド単体テストは expectedVersion の素通しを検証する。
- **採番衝突との共存**: `ConflictError` は採番衝突（ADR-0035 の P2002 翻訳）と楽観ロック競合の両方で使われる。区別はメッセージ文言とコマンド文脈で行う。
- 関連: ADR-0030（横断コンテキストは引数で渡す）、ADR-0032（差分 upsert = 本 ADR が守る永続化方式）、ADR-0035（ConflictError + 手動リトライの先例）、ADR-0038（throw / 共用体の判定基準と昇格ルール）
