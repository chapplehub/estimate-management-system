# ADR-0054: 承認免除を申請テーブルに含めず専用テーブルで表現する

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-17 |
| 最終更新日 | 2026-06-17 |

## コンテキスト

見積には承認不要のケースが存在する（ADR-0004 ほか）。

- 見積合計金額が 10 万円未満
- 消耗品のみの見積（金額無関係、ADR-0004）
- 事後見積（すでに作業済みのため承認スキップ、`システム設計書(見積).md` §6.3）

承認不要の見積から受注を作成できる（`システム設計書(見積).md` §4.3「承認済 or 承認不要から作成可能」）。この「承認不要（免除）」という状態をデータとしてどう表現するかを決める必要がある。

見積側は表示ステータスを **「行の存在」から導出**する設計（statusカラムを持たない。受注の確定/取消を `OrderConfirmation` / `OrderCancellation` の行の存在で導出。§1.3 / §11冒頭注記）を採用している。承認免除の表現もこのパターンに整合させたい。

## 検討した選択肢

### A. 申請レコードを作らない（不採用）

承認不要見積は申請レコードを作らず、受注作成を直接許可する。データは最小だが、ステータス導出（§1.3）に「申請レコードが無くても承認不要なら受注可」という特例分岐が必要になる。表示ステータスが「作成中」のまま受注可能となり分かりにくく、承認不要と判断した記録（誰が・いつ）も残らない。

### B. 0ステップの承認済申請レコードを生成（不採用）

承認不要時に `estimate_applications` に「ステータス=承認済・承認ステップ0件・最終承認役職=NULL」のレコードを作る。ステータス導出は一様化できるが、申請テーブルに **変則レコード**（最終承認役職 NULL・ステップ0件）が混入する。完全分離（ADR-0053）で得た「申請＝必ず承認フローを通る」という強い不変条件（最終承認役職 NOT NULL・承認ステップ1件以上）が緩む。

### C. 承認免除専用テーブル（採用）

`estimate_approval_exemptions` を新設し、免除を **承認とは別概念**として独立テーブルの行で表現する。免除理由・操作者・日時を記録する。

## 決定

**C を採用する。** `estimate_approval_exemptions`（承認免除）テーブルを新設し、承認不要の見積を確定した際にこのテーブルへ1行インサートする。

- カラム: `variation_id`（FK・NOT NULL・ユニーク）/ `estimate_id`（FK）/ `reason`（免除理由 enum）/ `exempted_by_employee_id`（操作者・FK）/ `exempted_at`
- `variation_id` は **単純ユニーク**（1バリエーションにつき免除は高々1件）。
- 免除レコード生成後のバリエーションは、承認済と同じく **編集不可**とする（§3.4 の編集制約に準拠）。これにより免除状態が行き来せず、履歴を複数残す必要がない＝単純ユニークで足りる。
- 見積側ステータス導出（§1.3）に「**承認免除レコードあり → 承認不要（受注可）**」を追加する。

## 根拠

- **「行の存在で状態を表現」する既存設計と一貫**: 受注状態を `OrderConfirmation` / `OrderCancellation` の行で導出するのと同じパターンで、承認免除も専用テーブルの行で導出する。§1.3 の導出表に1行追加するだけで乗る。
- **申請テーブルの不変条件を維持**: 申請テーブルを「実際に承認フローを通る申請」だけの純粋な集合に保てる（最終承認役職 NOT NULL・承認ステップ1件以上）。これは NULL 排除（ADR-0034 / ADR-0043）の精神に合致する。
- **概念を正直にモデル化**: 「免除」は「承認された」のではなく「承認が不要だった」状態であり、承認とは別のイベントである。専用テーブルはこの違いを正直に表す（B の偽装を避ける）。
- **記録が残る**: 免除理由・操作者・日時が残り、A の「記録が残らない」欠点を解消する。

不採用理由:
- **A**: 特例分岐が必要で、表示ステータスが不明瞭、記録も残らない。
- **B**: 申請テーブルに nullable・変則レコードを持ち込み、ADR-0053 で得た不変条件を緩める。

## 影響

- テーブルが1つ増えるが、関心の分離（申請＝承認を通る／免除＝承認不要）として正当化される。
- 見積ステータスの導出が「申請テーブル」「免除テーブル」「Order系」の3ソース参照になる。
- 免除レコードは **確定操作時に生成**する（編集の副作用で作らない）。承認要否の **判定** は副作用のない純関数（`ApprovalRequirementPolicy`）として作成・更新・画面表示・確定操作のいずれからも呼べるが、**永続化（免除レコード生成）は確定操作時のみ**とする（判定と永続化の分離）。
- 免除後は編集不可のため、内容を変えたい場合は見積標準の複製（C6）で新バリエーションを作るフローに乗る（承認済見積と同じ扱い）。
- 免除理由 enum（消耗品のみ／金額未満／事後見積）の値は、承認要否判定ポリシーの分岐と1対1で対応する。
