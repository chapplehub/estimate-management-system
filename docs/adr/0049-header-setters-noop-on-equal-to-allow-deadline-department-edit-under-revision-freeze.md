# ADR-0049: 見積ヘッダーのセッターを同値 no-op 化し、改訂凍結下でも締切日・部署の編集を成立させる

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-06-15 |
| 最終更新日 | 2026-06-15 |

## コンテキスト

C2 `UpdateEstimate` は集約をロード後、ヘッダー項目のセッター（`changeEstimateDate` / `changeDeadline` / `changeCustomer` / `changeDeliveryLocation` / `changeDepartment` / `changeTaxRoundingType` 等）を**無条件に全呼び出し**してから保存する。フォームが全ヘッダー項目を送る前提の配線である。

一方、改訂（C7 得意先改訂）が 1 件でも存在する見積は §7.2 により 見積年月日・税率・税端数区分・得意先・納品先 を変更できず、該当セッターは `assertHeaderMutable()` で `BusinessRuleViolationError` を投げる（締切日・部署のみ変更可・CONTEXT.md「凍結」「得意先改訂」参照）。

この 2 つが衝突する。改訂済み見積で「締切日だけ変えたい」場合でも、フォームは全項目を送るため `changeEstimateDate` 等が**現在値と同じ値**で呼ばれ、`assertHeaderMutable()` が即 throw する。結果、改訂済み見積はヘッダーを一切保存できない（変更可能なはずの締切日・部署すら不能）。これは「改訂済みでも締切日・部署は変更できる」という業務ルールと矛盾し、C2 テストでも未カバーの欠陥だった。S3（#331）でヘッダー編集 UI を配線するにあたり塞ぐ必要がある。

## 検討した選択肢

### A. ドメインのセッターを同値 no-op 化する（採用）

各ヘッダーセッターの冒頭で「新値が現在値と等しければ何もせず return」する。`assertHeaderMutable()` より**前**に置くことで、ロック項目でも**同値なら素通り**し、実変更だけが mutability 検証を通過する。等価判定は各 VO の `equals` ・ Date の `getTime` が持つ責務に収まる。コマンドは全セッター呼び出しのまま無変更。

```ts
changeEstimateDate(newDate: Date): void {
  if (this._estimateDate.getTime() === newDate.getTime()) return; // 同値は変更でない
  this.assertHeaderMutable();
  this._estimateDate = newDate;
  this.touch();
}
```

### B. コマンド側で差分適用する（不採用）

コマンドが現在値と入力を比較し、差分があるフィールドのセッターだけ呼ぶ。ドメインは無変更だが、Date / VO の等価判定がアプリ層に漏れ、同じ比較を将来の更新系コマンドでも再実装することになる。

### C. 改訂済み見積は編集ボタンごと無効化する（不採用）

S3 で改訂済み見積の編集を全面禁止する。最小だが「締切日・部署は変更可」という業務ルール（§7.2）を画面都合で削ってしまう。

## 決定

見積集約のヘッダーセッターを「新値＝現在値なら no-op（`assertHeaderMutable()` より前で早期 return）」にする（A を採用）。

## 根拠

- 不変条件の意味は「改訂後はこれらを**変更**できない」であり、同値設定は変更ではない。no-op はこの不変条件の意図に忠実で、ロックを弱めない（異値での変更は従来どおり throw）。
- 等価判定をドメイン（VO / Date）に置けるため、アプリ層に比較ロジックを散らさずに済む（B との差）。「フォームは全項目を送る／コマンドは全セッターを呼ぶ」という単純な配線を保てる。
- 副次効果として、`changeTaxRate` / `changeTaxRoundingType` の同値再設定で走っていた**全バリエーション再計算（ADR-0028）の無駄打ちが消える**。
- C は業務ルールを画面都合で削るため不採用。

## 影響

- 全ヘッダーセッターが「同値なら `updatedAt` を更新しない・税伝播しない」挙動になる（非改訂パスにも適用される挙動変化点）。「変更がなければ何も起きない」という直感に沿う。
- 改訂済み見積のヘッダー編集は「締切日・部署のみ実変更可能」が成立する。フォームは UX としてロック項目を disabled 表示するが、整合の最終強制はドメイン（assert）が担う二重防御。
- 同値判定の正しさは各 VO の `equals` / Date の `getTime` 比較に依存する。新たなヘッダー項目を増やす際は同じ no-op ガードを付ける規律が必要。
- 本 ADR の実装は S3（#331）で行う。修理情報の編集（`changeRepairDetail` / `changeAfterRepairDetail`）は価格に無関係なため `assertHeaderMutable()` を呼ばず、改訂済みでも編集可とする（同 issue の別判断）。
- 関連: ADR-0039（楽観ロック・version 往復）、ADR-0028（ミューテータでの自動再計算）、ADR-0044（改訂凍結＝導出状態）。
