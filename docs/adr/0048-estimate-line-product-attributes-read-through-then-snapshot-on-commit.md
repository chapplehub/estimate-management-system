# ADR-0048: 見積明細の商品由来属性（コード・区分）は編集中マスタ read-through・確定時スナップショット凍結とする（機構は申請スライスへ先送り）

| 項目 | 値 |
|---|---|
| ステータス | 採用（原則のみ。凍結機構は申請スライスへ先送り） |
| 起票日 | 2026-06-15 |
| 最終更新日 | 2026-06-15 |

## コンテキスト

見積詳細画面の閲覧スライス（issue #330 / S2）で、明細行に商品の**コード**・**区分（category）**を表示する（D10: category はドメイン enum 値で表示）。

既存スキーマでは `EstimateItem` がスナップショットするのは **`itemName` と `unit` だけ**で、商品 `code` / `category` は持たず `productId` 参照のみ（`EstimateSetGroup` も同様）。`itemName` / `unit` は「商品マスタからの複写」を**作成時に**固める（§8「金額の保存形式」＝見積時点の値を保持）。これは両者が**行ローカルで編集可能**な値（複写後に行ごとに書き換えうる）だからである。

一方 `code` / `category` は行で編集する対象ではない純粋なマスタ属性であり、見積時点で固める必然が `itemName` / `unit` ほど明確でない。表示に必要なため、何らかの形で値を得る必要がある。論点は「`code` / `category` も凍結すべきか、凍結するならどの時点か」。

なお凍結の起点になりうる「確定」イベント（申請・承認）は本実装スコープ外（D9、C5 は申請テーブル ADR-0001 依存で未実装、「申請テーブル実装時に接続」）。**凍結を起こすトリガーが現時点で存在しない**。

## 検討した選択肢

### A. 作成時スナップショット（`itemName`/`unit` と同様に作成時に複写）（不採用）

明細作成時に `code` / `category` も複写して列に固める。

**問題**: 編集中にマスタがリネーム・区分変更されても古い値が出続ける。`code` / `category` は行ローカル編集対象ではなく、編集中はマスタの最新を反映したい（マスタ修正が編集中の見積へ即座に効くべき）。作成時凍結はこの要求に反する。

### B. 恒久 read-through（凍結せず常にマスタ join）（不採用）

確定後も含め、常に現在マスタを join して表示する。

**問題**: 受注・帳票で歴史的正しさが崩れる。確定後にマスタがリネーム・区分変更・無効化されると、過去に確定した見積の表示が後から変わる。§8「見積時点の値を保持」・ADR-0033（確定した集計値は保存し再計算しない）と同じ思想に反する。

### C. 編集中 read-through ＋ 確定時スナップショット凍結（採用）

確定点（申請後など）までは現在マスタを read-through（join）で表示し、確定時にスナップショットテーブルへ凍結、以後はスナップショットから読む。

## 決定

**選択肢 C を原則として採用する。** ただし確定トリガー（申請・承認）が未実装のため、**原則のみを採用し、凍結機構（スナップショットテーブル＋確定時の freeze 書き込み＋read の凍結優先切替）は申請スライスへ先送りする**。

S2 では全見積が「編集中（pre-freeze）」フェーズしか存在しないため、read は**現在マスタ join（read-through）のみ**を実装する。

## 根拠

- **値の性質で凍結タイミングを分ける**: `itemName` / `unit` は行ローカル編集可能 → 作成時スナップショット。`code` / `category` は純粋マスタ属性 → 確定まで read-through・確定で凍結。凍結タイミングは2系統に分かれるが、値の性質駆動で整合的（分類でなく性質で割る）。
- **早すぎる凍結を避ける**: 編集中はマスタの最新（リネーム・区分変更）を反映したい。確定前に固めると編集中の見積に古い値が残る。
- **歴史的正しさ**: 確定後はマスタ変更から独立させる（§8・ADR-0033 と同じ「確定値は保存」思想）。
- **YAGNI**: 凍結イベントが存在しない現時点でスナップショットテーブルを建てても、書き込む主体も読む分岐の出番も無く dead code になる。read の source 差し替えは DTO の形（`productCode` / `productCategory` フィールド）を変えないため、機構は後付けできる。

## 影響

- **S2（本スライス）**: `code` / `category` は現在マスタ join（read-through）。スナップショットテーブルは作らない。DTO は `productCode` / `productCategory` をフィールドとして持ち、**出所（マスタ or スナップショット）は query service の実装詳細に閉じる**（将来の切替で DTO 形は不変）。
- **申請スライス（将来）**: 確定時に商品由来属性を凍結するスナップショットテーブルを新設し、確定の freeze 書き込みと read の「凍結済みなら snapshot 優先・未凍結ならマスタ」切替を追加する。**確定トリガーの定義**（申請後 / 受注確定 / 凍結のいずれか）もそのスライスで確定する。
- **非対称の明文化**: `itemName` / `unit`（作成時凍結）と `code` / `category`（確定時凍結）で凍結タイミングが分かれることを意図的に許容する。
