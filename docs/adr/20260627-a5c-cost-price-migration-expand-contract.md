# ADR-20260627-a5c: 原価移設は expand/contract で分割し、移行は固定起点・カテゴリ分岐の期間なし写像・遅延疎生成で行う

## コンテキスト

ADR-0065（原価を Product 集約から外し共通レイヤーへ移設）と ADR-0066（原価と共通売単価は独立改定の別集約）は採用済みだが**未実装**で、原価は今も `Product` 集約（`products.cost_price`）が単一値・時間有効性なしで保持している。#465（原価移設）はこの実体化スライスだが、次が未決だった。

1. 集約境界を動かす破壊的変更を**回帰なく**どう刻むか。原価を Product から剥がすと現状唯一の編集導線（Product 作成/編集画面の原価欄）の書き込み先が消え、原価が編集不能になる。
2. 既存 `products.cost_price`（NULL あり・複合品は `canHaveComponents` で `CostPrice(0)` 強制）を、新集約（商品id × 適用期間 × 原価）の**どの形**へ移すか。
3. 移行先の適用期間の**開始日に何を入れるか**。既存データは時間有効性を持たず開始日が無い。

技術前提として、適用期間 VO `ApplicablePeriod`（ADR-0067）は**下端は必ず閉じた開始日が必須で、無期限にできるのは上端のみ**。下端無限（「全時代に適用」）は表現できない。また既存の同型集約 `CommonSellingPrice` は `create()` で空集約を作れるが、`findByProductId` は親行が無ければ `null` を返す**遅延生成**で運用されている。

典拠: ADR-0065, ADR-0066, ADR-0067, ADR-0039, ADR-0048。切り出し元 #426。

## 検討した選択肢

### 分割: A. expand/contract（加法スライス → カットオーバースライス）（採用）

#465 を2スライスに割る。

- **Issue A（加法・可逆）**: 新原価集約＋スキーマ（`CommonSellingPrice` と同型の親＋期間テーブル、`daterange`＋EXCLUDE＋version）＋既存データのバックフィル移行。`products.cost_price` 列は温存・Product 集約は無改変。誰もまだ新集約を正本として読まない。
- **Issue B（カットオーバー・破壊的）**: Product 作成/編集画面から原価UIを**完全撤去**（新集約へ付け替えず編集導線を一旦消す）＋表示参照（`ProductMapper`/`ProductDTO`/`PrismaProductQueryService`／商品詳細画面の原価表示）の**除去**＋Product から `CostPrice` 除去＋`products.cost_price` 列削除＋旧金額VO `product/domain/values/CostPrice.ts` の削除。新集約を読む新規配線は持たない**純減算**。原価の編集・閲覧は後続の専用原価管理画面（#470）で初めて配線する。

### 分割: B. 一括移設（不採用）

集約剥離・移行・画面付け替え・列削除を1スライスで行う。回帰窓（列削除と画面付け替えの間に編集不能になる隙）を手続き順序だけで閉じることになり、部分的なロールバックができない。

### 移行起点: B'. 一律・固定の本番投入日（暦日ハードコード）（採用）

全行の開放端期間を `[本番投入日, )` で作る。投入日は移行 SQL に暦日定数で直書きする。

### 移行起点: created_at 行別 / now()（いずれも不採用）

行ごとに `products.created_at` の JST 暦日を起点にする案は「商品が存在して以降」と局所的に正当だが、粗利は売単価と原価の**両側**が同じ見積年月日をカバーして初めて成立し、共通売単価は legacy 列を持たない going-forward なので、原価だけ過去を厚くカバーしても粗利には無駄。`now()` はマイグレーションに非決定性が入り再現不能。

### 特殊行の写像: カテゴリで分岐し複合品・NULL は期間なし（採用）

移行は格納値ではなく**カテゴリ**で分岐する。

- 複合品（`canHaveComponents()` 真）→ 期間を作らない（格納値が 0 でも無視）。
- 非複合品 ＆ `cost_price IS NULL` → 期間を作らない（「期間なし＝原価未設定」）。
- 非複合品 ＆ `cost_price` 非NULL → `[本番投入日, )` を1本作成（本物の 0 も保存）。

### 特殊行の写像: 値 0 をそのまま期間化（不採用）

`cost_price = 0` を一律に「原価0円の期間」として実体化すると、複合品の強制0（＝単一値原価を持たないプレースホルダ）が「原価0円」になり、粗利接続後に 粗利 = 売単価 − 0 = 売単価全額 という誤りを生む。

### 生成戦略: 遅延・疎（採用） / 全商品に空ルート eager（不採用）

ルート＋期間は「非複合品 ＆ 非NULL」の行のみ生成する。複合品・NULL は親行も作らない。`CommonSellingPrice` の遅延パターンに揃え、後で原価を入れた初回に Issue B が `create()`→`addPeriod`→`save` でルートを生成する。全商品に version 付き空ルートを撒く eager 案は、原価集約がそもそも適用されない複合品にまで空行を作るノイズで採らない。

## 決定

#465 を expand/contract で Issue A（加法バックフィル・列温存・Product 無改変）と Issue B（カットオーバー・列削除）に分割する。バックフィル移行は **(1) 全行一律・固定の本番投入日を開放端期間の開始日とし、(2) カテゴリで分岐して複合品・NULL は期間を作らず本物の非複合品0のみ1期間を作り、(3) ルート＋期間は「非複合品 ＆ 非NULL」のみ生成する遅延・疎方式**で行う。

## 根拠

- **破壊的変更を隔離する（分割 A vs B）**: 加法スライスは新集約を建ててバックフィルするだけで挙動を変えず、単独で出荷・ロールバックできる。破壊的変更（列削除・原価UI撤去・参照除去）をカットオーバースライスに隔離し、各スライスを独立に検証できる。当初は Issue B で編集導線を新集約へ即時付け替えて回帰窓（原価が編集不能になる隙）を閉じる想定だったが、pre-production（開発段階・ユーザ無し）ゆえ編集不能を一時的に受容し、編集の再配線は専用の原価管理画面（#470）へ送る方が、時間有効性を持つ集約を単一値フォームへ写すインピーダンスミスマッチを #468 に持ち込まずに済む（下記 影響 参照）。
- **起点は粗利の対称性で決まる（B' vs 行別 created_at）**: 粗利＝売単価 − 原価は両側が同じ見積年月日をカバーして初めて成立する。共通売単価は legacy を持たず going-forward なので、原価の起点を売単価のカバレッジより過去へ伸ばしても粗利は成立せず無駄。加えて原価が解決不能でも見積単価は売単価から解決でき**操作は拒否されない**（原価は粗利表示専用で価格決定の入力ではない）ため、起点より前の穴は売単価側の穴より軽い。固定定数は決定的で監査可能、`now()` の非決定性を避ける。
- **0 の出自はカテゴリでしか分からない（カテゴリ分岐 vs 値分岐）**: 複合品の強制0は「単一値原価を持たない」プレースホルダ、非複合品が明示入力した0は本物。両者は `cost_price` の値だけでは区別できず、`canHaveComponents()` で分岐するほかない。複合品行は過去のドリフトもありうるため格納値を信用せずカテゴリで弾く。
- **情報は失われない（期間なし写像）**: 複合品・NULL がともに「期間なし（空タイムライン）」に落ちても、「セットだから構成品から導出」と「単に未設定」の区別は Product のカテゴリが保持しており、原価集約に焼き付ける必要はない。
- **既存の遅延パターンに揃える（疎生成）**: `CommonSellingPrice` が「親 null → 初回保存時に生成」で運用されているため、原価も遅延・疎にすると Issue B のオーケストレーションは売単価と同じ分岐（null なら create、在れば load→mutate→save）を再利用でき、新しい複雑性が生まれない。

## 影響

- **VO の一時併存**: Issue A を真に加法的（Product 無改変）に保つため、金額VO `product/domain/values/CostPrice.ts` は Issue A では動かさず、pricing 側に新規 `CostUnitPrice` VO を建てて一時併存させる。旧 `CostPrice` VO の削除は Issue B（contract 相）で行う。命名は `SellingUnitPrice`（共通売単価側の金額VO）と対をなす。集約ルートは概念名 `CostPrice`、期間エンティティは `CostPricePeriod`。「Common」修飾は原価が宛先非依存の単層で「共通でない原価」が存在しないため付けない。
- **起点より前の遡及見積で原価が解決不能**: 本番投入日より前の見積年月日では原価期間が無く粗利を解けない。ただし同条件で売単価側も解決不能になりうるため粗利は元々不成立で、見積単価は売単価から解決でき操作自体は拒否されない。
- **複合品の原価は後続スライス**: セット品の原価（構成品の合算）は本移行では扱わない。原価集約は複合品に対し「単一値原価なし」を正しく表す。
- **Issue B のスコープ変更（付け替え → 純粋契約）と未決事項の移送**: 当初 Issue B は原価編集を新集約へ付け替える想定だったが、純粋契約（原価UIの完全撤去・参照除去・列削除）に変更し、編集の再配線は後続の専用原価管理画面（#470）へ送る。これに伴い「画面の2集約オーケストレーション（更新順・部分失敗時の整合・トランザクション境界）」「原価編集の楽観ロック（新集約 version の読出経路・ADR-0039）」「単一値編集 → append-only 期間タイムラインへの写像（現行期間の上書き／close-and-open／close-period のリポジトリ追加）」はいずれも #470 で設計する。#468 完了から #470 配線までの間、原価は編集・閲覧不能（時点解決 read も無消費）になるが、pre-production ゆえ無害。
- スキーマは `CommonSellingPrice`（ADR-0066/0067）と完全同型（親＋期間＋`daterange`＋EXCLUDE＋version）で、スキーマ分解の論点は対称性で充足する。
