# ADR-0008: 役割一覧の役職フィルタにpositionId（ID）を使用する

| 項目 | 値 |
|---|---|
| ステータス | 採用 |
| 起票日 | 2026-04-03 |
| 最終更新日 | 2026-04-03 |

## コンテキスト

役割一覧画面（`/roles`）の検索パラメータにおいて、役職フィルタのクエリパラメータを`positionId`（内部ID）にするか`positionCd`（役職コード）にするかを決める必要がある。

現状のURL例:
```
/roles?name=課長&roleCd=ROLE015&positionId=n89urlqmz8eyy52lizn6b1kj
```

`name`と`roleCd`はテキスト入力による検索で人間が読めるコード体系を使用しているが、`positionId`だけがCUID形式のIDになっている。

## 検討した選択肢

### A. `positionId`（内部ID）を使用する（採用）

```
/roles?positionId=n89urlqmz8eyy52lizn6b1kj
```

### B. `positionCd`（役職コード）を使用する（不採用）

```
/roles?positionCd=POS004
```

## 決定

`positionId`（内部ID）を使用する。

## 根拠

### セレクトボックス選択のためURLの可読性は不要

`name`や`roleCd`はテキスト入力フィールドであり、ユーザーがURLを見て値を理解・編集する可能性がある。一方、役職フィルタはセレクトボックスからの選択であり、ユーザーがURLの値を手で組み立てるユースケースがない。

### DBクエリがシンプルになる

RoleテーブルはPositionテーブルへの外部キーとして`positionId`を持っている。`positionId`で検索すればJOINなしで直接WHERE句に指定できる。`positionCd`を使う場合、Positionテーブルとの結合またはサブクエリが必要になり、クエリが複雑化する。

### 他画面との一貫性

部門一覧画面（`/departments`）の親部門フィルタでも`parentId`（ID）を使用しており、セレクトボックス選択にはIDを使うパターンが確立されている。

### 不採用理由

- **選択肢B（positionCd）**: URLの可読性は向上するが、セレクトボックス選択のためその恩恵が小さい。一方でPositionテーブルとの結合が必要になり、検索のQueryServiceが複雑化する。また部門一覧の`parentId`パターンとの一貫性が崩れる

## 影響

- 役職フィルタのURLはブックマーク時に人間が読めない形になるが、セレクトボックス選択のため実用上の影響はない
- 将来的にURLの可読性が求められる場合（例：URLをメールで共有するユースケース）は、この決定を見直す
