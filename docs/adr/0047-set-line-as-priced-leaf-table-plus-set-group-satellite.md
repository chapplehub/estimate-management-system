# ADR-0047: 見積のセット商品明細は「価格付き末端行 1 表 ＋ 薄いセット群衛星 ＋ 所属交差表」でモデル化する

| ステータス | 採用 |

## コンテキスト

見積詳細画面の本実装（issue #326 の検討）に向けて、セット商品（`ProductCategory.SET`）を見積明細に取り込む際のモデルを決める必要がある。

セット商品は「それ自体の価格は常に 0 円で、金額はすべて構成商品の価格 × 数量から算出される」概念的な商品（CONTEXT.md「セット商品」）。デザイン v0.3.1 の行コンポーネント改良指示書は、これを明細テーブルで「セット親行（単価 0・数量 1 固定・最終金額＝子合計）」とその配下の「セット子行（構成商品）」として表現し、サンプルデータ（`docs/design/estimate-detail/data.js`）も `kind: 'set-parent' | 'set-child'` ＋ `parentItemId` のフラット配列で作り込んでいる。

しかし既存の `EstimateItem` には `kind`・`parentItemId` 相当のフィールドは無く、セット親子は未モデル化。商品サブドメイン側は `ProductCategory.SET` / `SetProductComponent`（セット構成）/ `ProductRelation`（周辺商品）/ `isActive` をすべて実装済みで、見積側だけが空白という状態。

「画面に親行→子行の入れ子が見える」ことから素朴に「基底テーブル ＋ サブタイプ ＋ 子テーブル（CTI 風の入れ子）」へ写したくなるが、これは **UI の見た目を永続化構造へ漏らす**判断であり、本プロジェクトのスキーマ分解方針（列単位で割る・NULL 徹底排除＝kawasima デシジョンツリー）および ADR-0043（取引先 CTI を対価ゼロとして廃した前例）と衝突する。

## 検討した選択肢

### A. 単一表 ＋ `kind` 判別子 ＋ 自己参照 `parentItemId`（不採用）

`estimate_items` に `kind`(normal/set-parent/set-child) と自己参照 FK `parentItemId` を足す。デザイン data.js・D&D ロジックにそのまま乗り、diff-upsert（ADR-0032）の階層数も増えない。

**問題**: セット親行は単価・数量・掛率・値引・各金額が全列「適用不能」で、行が無意味なゼロで埋まる。これは NULL 排除方針（kawasima ルール 2「特定種別でだけ意味を持つ→排他的サブタイプ表」）と「別概念を `kind` で 1 表にまとめない」方針に反する。

### B. 基底テーブル継承（CTI）＋ 入れ子（不採用）

`estimate_items`(基底) → `estimate_normal_items` / `estimate_set_items` のサブタイプ ＋ セット配下に子行テーブル。

**問題**: (1) 基底＋サブタイプは ADR-0043 で「対価ゼロ」と判断して廃した CTI そのもの。(2) 通常明細と構成明細は列構成が完全一致する「同じ価格付き末端行」であり、別表に割ると価格列が複製され、フレーム観点 4「共通列の N 表複製＝変更コスト・ドリフト ×N」に反する。(3) 改訂明細の粗利スナップショット `RevisedEstimateItemDetail` は価格付き末端行（通常明細・構成明細）にぶら下がるため、両者を別表に割ると衛星の FK が多態参照になる（ADR-0043 が CTI を否定した典型理由の逆）。

### C. 価格付き末端行 1 表 ＋ 薄いセット群衛星 ＋ 所属交差表（採用 / Shape ③-a）

- `estimate_items`（**現行のまま**・列不変）= **価格付き末端行**。通常明細と構成明細を同居させる。`RevisedEstimateItemDetail` の単一 FK 先も維持。
- `estimate_set_groups`（新規・薄い衛星）= **セット群**のヘッダ。`productId`・名称/単位スナップショット・メモのみ。**価格列・並び順列・金額列を持たない**（金額＝構成明細の合計、表示位置＝構成明細の最小並び順から導出）。
- `estimate_set_components`（新規・交差表）= 構成明細がどのセット群に属すかを**行の存在で表現**（`item_id` 一意）。nullable FK を作らず NULL ゼロ。
- ネスト禁止・構成明細の連続配置・親削除での構成カスケード・セット群の単価 0／数量 1 などの不変条件は、表トポロジではなく **Estimate 集約（ドメイン層）** で担保する（ADR-0019/0029 と同方針）。

## 決定

**見積のセット商品明細は選択肢 C（Shape ③-a）でモデル化する。** 価格付き末端行（通常明細＋構成明細）を `estimate_items` 1 表に保ち、セット群を価格を持たない薄い衛星 `estimate_set_groups` に分離し、構成明細の所属を交差表 `estimate_set_components` で（行の存在により）表す。セット群の金額・表示位置は導出とし列に持たない。区分駆動の不変条件はドメイン層で担保する。

## 根拠

- **列単位で割ると結論が一意に出る**: 価格系列はセット群で適用不能（→衛星分離・kawasima ルール 2）。通常明細と構成明細は列が完全一致（→同居・観点 4）。「3 種別だから 3 表」という分類駆動を退け、列の適用不能性で割る（[[schema-table-decomposition-framework]]）。
- **導出を構造にしない**: 「セット群の金額＝構成明細の合計」は計算であって格納すべき包含関係ではない。合計を保持するための表階層を建てない（「導出可能な値は列で持たない」方針）。永続化する派生列がゼロなので ADR-0033（集計値の永続化）とも素直に両立する。
- **NULL ゼロを保つ**: 所属は nullable FK ではなく交差表（行の存在）で表現する（kawasima ルール 3）。
- **既存資産への波及を最小化する**: `estimate_items` の列を変えないため、`RevisedEstimateItemDetail` の単一 FK・既存の金額計算（`EstimateVariation.computeTotals` は全行の `finalAmount` を合算。セット群は `estimate_items` に行を持たないため二重計上が起きない）・diff-upsert（ADR-0032）の既存階層がそのまま生きる。新規はセット群衛星と交差表の差分処理（命令的 upsert ＋ `deleteMany(notIn)`）の追加に閉じる。
- **UI の見た目を永続化に漏らさない**: 親子は「表現（UI 入れ子）／ドメイン（不変条件を持つ集約）／永続化（正規化）」の三層でそれぞれ最適な形を取る。デザインの入れ子は UI の成果物であり、DDL を選ばせない。

### 不採用理由まとめ

- **A（単一表＋kind）**: セット群行が無意味なゼロで埋まり NULL 排除・「kind で混ぜない」方針に反する。
- **B（CTI＋入れ子）**: ADR-0043 で廃した CTI の再導入。通常／構成の同形列を複製し、`RevisedEstimateItemDetail` を多態参照に追い込む。

## 影響

- **スキーマ移行**: `estimate_set_groups`（セット群ヘッダ）と `estimate_set_components`（所属交差表・`item_id` を主キーとして排他所属を構造担保）を追加する。`estimate_items` は無変更。
- **集約の再構築・差分保存**: Mapper はセット群衛星と交差表を読み、構成明細を `estimate_items` 行へ再構成する。`PrismaEstimateRepository.save()` の diff-upsert（ADR-0032）にセット群・交差表の階層を追加する（生存・新規は id キー upsert、消えたものは `deleteMany(notIn)`）。
- **金額計算は無変更**: セット群は `estimate_items` に行を持たないため、`EstimateVariation.computeTotals` の「全 `finalAmount` 合算」は構成明細を正しく集計する。セット群の表示金額（子合計）は読み取り DTO の導出値とする。
- **C1/C4 への波及**: バリエーション内容の作成・更新（C1 / C4）がセット群＋構成明細の生成・差替を扱えるよう拡張する。構成商品の自動展開（`SetProductComponent` から構成明細を生成）・無効構成商品の警告・set-aware な D&D 制約はアプリ／ドメイン層で実装する。
- **空のセット群の禁止**: 表示位置・金額を構成明細から導出するため、構成明細を 1 件も持たないセット群は不正とし、最後の構成明細削除時にセット群ごと削除する（ドメイン不変条件）。
- **用語**: CONTEXT.md に「セット群 / 構成明細 / 通常明細」を追加。
- **関連**: ADR-0043（CTI を廃した前例）、ADR-0032（多階層差分 upsert）、ADR-0033（集計値の永続化）、ADR-0019/0029（区分駆動の不変条件はドメイン層）、ADR-0046（改訂先の行構成固定。セット群を含む改訂先でも構成明細の増減は不可）。

## 補足: テーブル名のリネーム（2026-06-13・issue #329 着手前の /grill-with-docs で確定）

衛星テーブルを当初の `estimate_set_items` から **`estimate_set_groups`** へリネームした。理由:

- **グロッサリ整合**: CONTEXT.md「セット群」は _Avoid_ に「セット明細（明細＝価格付き末端行と紛らわしい）」を挙げている。`set_items`（≒ set 明細）は価格を持たない**群ヘッダ**を指すのに「明細／価格付き行が入っている」と誤読させ、用語集の意図に反する。
- **三層統一**: 用語（セット群）・ドメインエンティティ（`EstimateSetGroup`）・永続化（`estimate_set_groups` / Prisma `EstimateSetGroup`）を同一語に揃え、Mapper の対訳体操を無くす。

これに伴い、ドメイン層の区分駆動チェック（ネスト禁止・構成商品は個別/消耗品のみ）は、`Estimate` 集約が `ProductCategory` を知らない（Product 集約に依存しない）ため、ADR-0029 の集約越え例外に従い **`estimate/domain/services/` のルール検証型 Domain Service（ドメインポート経由）＋実装は S5** に置くことも併せて確定した。S1 のドメインが担保するのは集約内で完結する構造的不変条件（**空群禁止・参照整合・排他所属**）に限る。`item_id` を交差表の主キーとし、排他所属を DB レベルでも構造担保する。

> **2026-06-16 追記（S5 着手時の /grill-with-docs で差し替え）**: 上記「ドメインポート経由のルール検証型 Domain Service」は **ADR-0052 で取り下げ**た。集約越えの区分/有効性検証は、ドメインポートを新設せず **ADR-0030（アプリ層が事実を集めメソッド引数で渡す）** で行う。理由・経緯は [ADR-0052](0052-set-component-cross-aggregate-validation-via-method-arguments.md) を参照。あわせて、区分/ネスト違反が UI から到達する経路（構成明細の手動追加）は S5 スコープ外とし issue #350 へ繰り延べ、S5 の検証はペイロード防御として導入する。
