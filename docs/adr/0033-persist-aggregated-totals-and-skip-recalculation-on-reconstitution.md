# ADR-0033: 集計値を永続化し、再構築時には再計算せず保存値から復元する

## コンテキスト

Estimate 集約は、入力値（明細の単価・数量・割引・税率）から導出される**集計値（派生値）**を多数持つ。

- EstimateItem: `baseAmount` / `discountedAmount` / `finalAmount`
- EstimateVariation: `subtotal` / `discountSubtotal` / `finalSubtotal` / `taxAmount` / `finalTotal`

ADR-0028 で「集計値を持つ集約は全 mutator で自動再計算を強制する」と決めており、**書き込み時の集計の正しさはドメイン側で保証されている**。これらの集計列は Prisma スキーマにも実列として存在する。

Issue #290 のリポジトリ実装で、読み取り（`findById` / `findByEstimateNumber` の `toDomain`）時に集計値をどう扱うかを決める必要が生じた。すなわち **「DB に保存された集計値をそのまま使うか、ロード時にドメインロジックで再計算するか」**。

## 検討した選択肢

### A. ロード時に再計算する（recompute-on-load）（不採用）

`toDomain` では入力値（単価・数量・割引・税率）だけを復元し、集計値はドメインの計算ロジック（ADR-0023 の Policy）を再実行して求める。DB の集計列は冗長として読み捨てる。

**利点**: DB の集計列が古くても常に「今のロジックでの正しい値」が得られる。

**問題**:
- 集計列を永続化している意味が読み取り側で失われる。一覧クエリ・レポート・他コンテキストが DB の集計列を直接 SELECT する用途（DTO に金額を載せる等）と、集約経由のロード値が**二系統に分岐**し、同じ見積で値が食い違う温床になる。
- 再構築は本来「保存した状態をそのまま起こす」操作。ここに計算を挟むと、計算ロジックを改訂した瞬間に**過去に保存・承認・印刷した見積の金額が後から変わる**。確定済み帳票の金額が再計算で動くのは業務的に許容できない。
- `reconstruct()` に「計算する責務」を持たせると、`create()`（計算する）と `reconstruct()`（しない）の役割分担が崩れる。

### B. 保存値をそのまま復元する（persist + restore）（採用）

集計列も含めて永続化し、`toDomain` では DB の集計値を `reconstruct()` にそのまま渡して復元する。ロード時に再計算しない。集計の正しさは書き込み時（ADR-0028 の自動再計算）にのみ確定する。

## 決定

集計値（item の base/discounted/final、variation の subtotal 系・tax・total）は **DB 列として永続化し、再構築時には再計算せず保存値から復元する**（選択肢 B）。集計の確定は書き込み経路（ADR-0028）に一本化し、読み取り経路は確定済みの値を忠実に復元するだけにする。

## 根拠

### 「書くとき計算、読むとき復元」で計算の単一責任点を作る

集計が変わるのは mutator が走ったとき（= 書き込み時）だけ、と決めれば、計算の発生点が 1 か所に集約される。ADR-0028（mutator で自動再計算）と本 ADR は表裏一体で、**書き込み側が計算の単一責任点、読み取り側はその結果を忠実に再現する**という対称をなす。

### 確定済みの金額は後から動いてはならない

見積金額は承認・提出・印刷を経て確定する業務データ。計算ロジックの将来改訂で過去の確定値が遡って変わるのは事故。保存値をそのまま復元すれば、保存時点の金額が不変に保たれる。

### 永続化した集計列を読み取りの単一情報源にできる

集計列を DB に持つ以上、一覧 DTO・レポートはそれを直接読む（ADR-0013）。集約経由のロードも同じ列から復元すれば、**両経路が同一情報源を見る**ことになり値の食い違いが原理的に起きない。

### `reconstruct()` の責務を純粋に保つ

`reconstruct()` は「保存済みの値を検証も計算もせずに組み立て直す」操作に徹する。`create()`（不変条件を検証し集計を計算する）との役割分担が明確になり、Mapper も入力値・集計値を一様に流し込むだけで済む。

### 不採用理由

- **A（recompute-on-load）**: 確定済み金額が計算ロジック改訂で遡って変わる業務リスク、読み取り値の二系統分岐、`reconstruct` の責務肥大。

## 影響

### 書き込み側の集計正当性が全体の前提になる

読み取りで再計算して取り繕う安全網が無いため、**DB の集計列が常に正しいことは書き込み経路（ADR-0028 の自動再計算）が完全に保証している**ことに全面的に依存する。mutator を増やす際は必ず再計算を通す規約を守る必要がある。

### 集計列とロジックの不整合は別途検知が要る

万一、計算ロジック改訂で「保存値」と「現ロジックの計算値」が乖離しても、本方針では読み取り時に気づけない。乖離検知が必要なら、移行バッチや整合性チェックを別途用意する（本 ADR のスコープ外）。

### 今後の集計値を持つ集約に適用する

派生値を列として永続化する集約（Order / Invoice 等）でも本方針を踏襲する。「書くとき計算（ADR-0028）／読むとき復元（本 ADR）」をペアの規約として扱う。
