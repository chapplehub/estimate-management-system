# ADR-0046: 改訂先バリエーションは C4 全置換を拒否し、調整は粒度別メソッドに限定する

| 項目 | 値 |
|---|---|
| ステータス | 採用（数量の調整可否は ADR-0060、単価の調整可否は ADR-0064 で改訂） |
| 起票日 | 2026-06-12 |
| 最終更新日 | 2026-06-22 |

## コンテキスト

得意先改訂（C7・#296）で生まれた改訂先バリエーションには行構成固定（§7.2: 明細の追加・削除不可）が課される。改訂元との明細 1:1 対応を保全し、明細単位の粗利（`deliveryPrice` − 得意先価格・§8.4）を常に計算可能に保つためである。

一方、既存の C4 UpdateVariation（`replaceContent`）は「明細の識別子を保持せず新セットで全置換」する設計（ADR-0032 の差分 upsert が旧行削除・新行挿入で永続化）である。これを改訂先に適用すると 2 つの問題が起きる。

1. 全置換は構造上「全行削除＋全行追加」であり、行構成固定と原理的に矛盾する
2. 明細に付随する `RevisedEstimateItemDetail`（deliveryPrice スナップショット）が、明細識別子の引き継ぎがないため**黙って消失する**

## 検討した選択肢

### A. 改訂先への C4 `replaceContent` をドメインエラーで禁止（採用）

改訂先（出自 `revisedFrom` を持つバリエーション）は `replaceContent` / `addItem` / `removeItem` を `BusinessRuleViolationError` で拒否する。調整（単価・掛率・明細値引・全体値引・数量・メモ）はドメインに既存の粒度別メソッド（`changeItemUnitPrice` / `changeItemDiscountRate` / `changeItemDiscount` / `changeOverallDiscount` / `changeItemQuantity` / メモ変更）で行う。

> **改訂（ADR-0060・2026-06-19）**: ここに挙げた**数量**は後に固定へ改められた。改訂価格 `deliveryPrice` が改訂元の**行金額**スナップショットである以上、数量可変は粗利（§8.4）の基準を黙って壊すため、改訂先は数量を変更できない（`changeItemQuantity` はドメインの `assertQuantityImmutable` で拒否）。改訂先の調整可能集合は「単価・掛率・明細値引・全体値引・メモ」に確定。
>
> **改訂（ADR-0064・2026-06-22・#422）**: 上の**単価**も後に固定へ改められた。見積単価は価格決定が販売単価マスタから解決した値で確定・固定し、手入力での修正を許さない（通常明細・改訂先に一貫適用）。改訂先の単価は改訂元から引き継がず提出区分で再解決する。改訂先の調整可能集合は「掛率・明細値引・全体値引・メモ」に縮小（単価を除外）。

### B. `replaceContent` を改訂先対応に拡張（不採用）

新旧明細の対応関係を検証し、対応する旧明細から `RevisedEstimateItemDetail` を引き継ぐ。「どのキーで対応付けるか」（位置か、productId か、複合自然キーか）という選定問題を今解く必要があり、入力次第で対応付けが壊れる脆さを常に抱える。

### C. 改訂先専用の一括調整メソッドを新設（不採用）

明細 ID を明示した調整入力（id → 新単価等のマップ）を受ける専用メソッド。設計としては健全だが、現時点で利用者（プレゼンテーション層の調整画面）が存在せず、C7 のスコープ（ドメイン＋アプリ層の縦スライス）には過剰。

## 決定

改訂先バリエーションへの C4 全置換をドメインエラーで禁止し、調整は粒度別メソッドに限定する（A を採用）。

## 根拠

- 行構成固定は「全置換を安全に解釈し直す」より「全置換を入口で拒否する」方が単純かつ確実。B の対応付けキー問題を解かずに済む
- スナップショット消失（問題 2）は検証では防げない種類のバグ（型上は正しい入力で起きる）であり、操作自体を構造的に閉じるのが確実
- 粒度別メソッドは集約境界規約（ADR-0027）に沿って既に完備しており、追加実装ゼロで調整経路が成立する
- C（専用一括調整）は将来プレゼンテーション層が必要とした時点で、その画面のフォーム形状に合わせて設計すればよい（YAGNI）。本決定はそれを妨げない

## 影響

- 集約には強度の違う 2 つのガードが共存する: **凍結**（改訂元: メモ・ステータス以外すべて不可）と**行構成固定**（改訂先: 行追加削除・全置換のみ不可）
- 改訂先の調整 UI を実装する際、C4 コマンドは使えない。粒度別操作のアプリ層コマンド、または明細 ID 明示の一括調整コマンド（選択肢 C）の新設が必要になる（将来 issue）
- C4 UpdateVariation の適用対象は「改訂で生まれていないバリエーション」に実質的に限定される
- （2026-06-19・ADR-0060 で改訂）改訂先は数量も固定する方針に改めた。選択肢 A 説明・本影響欄が許容としていた「数量」調整は取り消され、選択肢 C（明細 ID 明示の一括調整）が `AdjustRevisedVariationCommand`（#390）として実装された。詳細は ADR-0060 を参照
