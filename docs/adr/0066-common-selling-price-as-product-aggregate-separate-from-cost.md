# ADR-0066: 共通販売単価は商品単位集約とし、原価と共通売単価を独立改定の別集約に分離する

| 項目 | 値 |

| ステータス | 採用 |
| 起票日 | 2026-06-23 |
| 最終更新日 | 2026-06-23 |

## コンテキスト

#426 で共通販売単価（と原価）の集約境界を定めるにあたり、2つの論点が出た。

1. 共通販売単価を「商品単位で1集約（適用期間行を内包）」とするか、TaxRate マスタ（ADR 外・`effectiveFrom` 単列の前例）のように「期間行1件＝1フラット集約」とするか。
2. ADR-0065 は原価を「共通販売単価と同じ期間付き共通レイヤーへ同居」させると決めたが、この「同居」が「原価と共通売単価を1集約・1行・同一適用期間で持つ」ことまで意味するのかが未確定だった。

共通販売単価には TaxRate に無い性質がある: タイムラインが商品ごとに分かれ、各行が「区間（適用期間）」を持ち、**同一商品内で適用期間が重複してはならない**という不変条件が発生する。

## 検討した選択肢

### A. 商品単位集約＋原価と共通売単価を別集約に分離（採用）

共通販売単価は商品単位で1集約とし、適用期間行のコレクションを内包する（集約ルートの identity は ProductId を流用、期間行はサロゲート UUIDv7）。原価も商品単位集約とするが、共通販売単価とは別集約に分ける。両者は同じ ProductId を参照するが、整合性境界は別。

### B. TaxRate 型フラット（期間行1件＝1集約）（不採用）

期間行をサロゲート ID 持ちの独立集約とし、商品 id は検索キー列にとどめる。集約は1行しか知らないため、期間重複禁止をドメイン層で表現できず DB 制約のみが頼りになる。

### C. 原価と共通売単価を1集約に同居（不採用）

原価行と売単価行を同一集約（同一ルート version）にまとめる。両者の適用期間を独立に改定すると、無関係な改定どうしが同一ルートの楽観ロックで競合する。

## 決定

共通販売単価は商品単位で1集約（適用期間行を内包）とし、原価とは別集約に分離する（A を採用）。原価と共通売単価はそれぞれ独立した適用期間を持ち、別々に改定されうる。

## 根拠

- **期間重複禁止を集約内で構造保証できる（A vs B）**: 商品単位集約なら `findByProductId` で同一商品の全期間行を1集約にロードでき、`addPeriod` が既存区間との重なりを集約内で判定して `overlaps` 違反を弾ける（ADR-0029 の構造的不変条件）。DB の EXCLUDE 制約（ADR-0067）は二重防御の最後の砦に回す。B では集約が1行しか知らず、この不変条件をドメインで表現できない。
- **跨る不変条件が無いので別集約が正しい（A vs C）**: 原価と共通売単価の独立改定を許す（仕入値上げで原価だけ改定し売価据え置き＝粗利縮小、は通常業務）以上、「売単価 ≥ 原価」のような跨る不変条件は持たない。集約をまとめる唯一の正当な理由が無い。
- **独立改定＝独立トランザクション＝独立 version**: C で同居させると、原価改定が集約ルートの version を上げ、無関係な売単価改定と楽観ロック競合（ADR-0039）を起こす。独立と決めたのに人工的競合を生むのは矛盾。
- **拡張軸が異なる**: ADR-0065 により後続フェーズ A2 で共通売単価は得意先別・納品先別の3層に拡張されるが、原価は宛先非依存の1層のまま拡張されない。共通売単価は「3層 family の土台」、原価は「単層の別 family」。変更理由の違うものを別集約に置くと、A2 で境界を引き直さずに済む。
- 「商品単位」は cardinality（1 商品に1つ）であって ownership ではない。同じ ProductId に Product 集約・共通販売単価集約・原価集約が並立し ID 参照で結ぶのは DDD として正常（ADR-0048 の EstimateItem→Product と同型）。

## 影響

- 価格決定 family を収める新規サブドメイン `pricing`（`src/server/subdomains/pricing/`）を立ち上げる。バレル＋ESLint 境界は ADR-0027 に従う。
- 共通販売単価集約: ルート identity = ProductId 流用、適用期間行 = サロゲート UUIDv7。永続化は親テーブル（version 保持・ADR-0039）＋期間行テーブルの差分 upsert（ADR-0032）。
- 適用期間の重なり/包含判定を担う `ApplicablePeriod` VO を `shared` に共有基盤として置き、原価集約も再利用する。
- 原価集約・原価の Product からの剥離・既存データ移行は #426 から切り出し、別 issue（原価移設）に送る。#426 は共通販売単価集約に純化する。
- 時点解決（見積年月日で有効な売単価を引く）は read 関心として価格決定フェーズで QueryService に置く。共通販売単価 Repository は `findByProductId`/`save` に限定する。
- 適用期間の物理表現・重複禁止制約は ADR-0067 で定める。
