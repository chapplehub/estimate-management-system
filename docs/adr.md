# Architecture Decision Records

## ADR-001: 申請テーブルを見積・価格で共通化する

### Status

Accepted

### Context

承認フローは「見積」と「価格」の2種類に対して存在する。申請テーブルの設計として、種別ごとに別テーブルを作るか、共通の1テーブルにするかを決める必要がある。

別テーブルにすると外部キーが明確になるが、承認フローのロジック（承認ステップの管理・ステータス遷移・差戻処理など）が重複する。共通テーブルにすると承認ロジックを一元化できるが、対象IDがポリモーフィック（見積番号にも価格申請グループIDにもなる）になり、データベースレベルの外部キー制約を張れない。

### Decision

申請テーブルは見積・価格で共通の1テーブルとする。申請種別カラム（見積 / 価格）で区別し、対象IDの参照先は申請種別によって変わる。

### Consequences

- 承認フローのロジック（申請→承認ステップ進行→差戻→取下）を共通化でき、コードの重複を避けられる
- 対象IDに対してデータベースレベルの外部キー制約を張れないため、アプリケーション層でのバリデーションが必要になる
- 将来的に新しい申請種別（例：受注確定の承認）を追加する場合、テーブルの変更なく申請種別の追加のみで対応できる

## ADR-002: 承認ステップを申請時に全ステップ事前生成する

### Status

Accepted

### Context

承認フローは複数ステップ（例：課長→部長→本部長）で構成される。承認ステップの生成方式として以下の2案がある。

1. **事前生成方式**: 申請時に承認チェーン全体を計算し、全ステップを一括で生成する
2. **逐次生成方式**: 前のステップが承認されてから次のステップを動的に生成する

逐次生成方式は承認チェーンの途中変更に柔軟だが、フロー全体の事前確認ができず、差戻時の履歴追跡が複雑になる。

### Decision

申請時に承認チェーン全体を計算し、全ステップを事前生成する。最初のステップのみ「承認待ち」、残りは「未着手」ステータスで生成する。

### Consequences

- 申請時点で承認フロー全体を可視化でき、申請者や管理者がフローの全体像を確認できる
- 差戻時の履歴追跡が容易になる（全ステップが事前に存在するため）
- 申請時にチェーン上の全役割にユーザが割り当てられているかを検証でき、承認者不在によるフロー停滞を未然に防げる
- 承認チェーンの途中で組織変更（役割の上位役割変更など）があった場合、進行中の承認フローには影響しない（事前生成済みのため）
- 差戻後の再申請は新しい承認フローを最初から作り直すため、組織変更後の再申請では新しいチェーンが反映される

## ADR-003: 高位役職者の申請でも常に上位役割に承認を求める

### Status

Accepted

### Context

承認フローのゴール（最終承認者の役職）は見積金額で決まる。例えば100万円以上は「部長承認」がゴールとなる。

ここで「部長自身が100万円の見積を申請した場合」に矛盾が生じる。申請者の上位役割は本部長レベルであり、ゴール（部長）よりスタート（本部長）が上位になる。

設計方針として以下の選択肢がある。

1. 申請者の役職 >= ゴール役職なら承認不要とする
2. 常に上位役割に承認を求める（ゴールとの比較に関係なく）
3. スタート > ゴールなら即承認（自動承認）

### Decision

承認者の役職レベルに関係なく、常にユーザテーブルの上位役割に対して承認を求める。「自分自身の見積は自分で承認できない」という暗黙のポリシーではなく、「常に上位に判断を仰ぐ」というポリシーを採用する。

ただし、承認ステップの役割に申請者自身が含まれている場合（兼務など）は自己承認を許可する。

### Consequences

- 承認フローのロジックがシンプルになる（金額によるゴール判定と申請者の役職を比較する分岐が不要）
- 全ての申請に対して最低1ステップの承認が保証され、内部統制の観点で安全
- 部長が100万円の見積を作成した場合でも本部長の承認が必要になり、ゴールの金額レベルより厳しい承認になるケースがある
- 兼務ケース（例：部長と本部長の役割を持つユーザ）では自己承認が可能なため、実務上の柔軟性は確保される

## ADR-004: 消耗品のみの見積は金額に関係なく承認不要とする

### Status

Accepted

### Context

見積の承認不要条件として「見積合計金額が10万円未満または消耗品のみの場合」というルールがある。「消耗品のみ」の条件に金額上限を設けるかどうかを決める必要がある。

消耗品は機械本体と異なり低リスクの補助品目であるが、金額が大きくなる場合（例：500万円分の消耗品）でも承認なしで通すかという論点がある。

### Decision

消耗品のみの見積は金額に関係なく承認不要とする。

### Consequences

- 消耗品の発注業務が迅速になり、承認待ちによる業務遅延がない
- 消耗品の定義（商品区分）が正しく管理されていることが前提となる。個別商品を誤って消耗品区分にすると承認をバイパスできてしまうリスクがある
- 将来的に高額消耗品の承認が必要になった場合、この決定を見直す必要がある

## ADR-005: 差戻後の再申請は承認フローを最初からやり直す

### Status

Accepted

### Context

承認フローで差戻された見積を修正して再申請する場合、承認フローの扱いについて2つの方式がある。

1. **最初からやり直し**: 新しい承認フローを作成し、全ステップを再度通す
2. **途中再開**: 差戻されたステップから再開し、それ以前の承認は有効のまま

途中再開方式は承認者の負担を軽減するが、差戻後に見積金額が変更された場合にゴール（最終承認者の役職）が変わる可能性があり、既存ステップとの整合性が崩れる。

### Decision

差戻後の再申請は承認フローを最初からやり直す。新しい申請レコードと新しい承認ステップが生成される。

差戻された申請レコードと承認ステップは削除せず履歴として保持する。申請テーブルに「申請回数」（1, 2, 3...）を持たせ、同一対象に対する申請の履歴を `(対象ID, 申請種別, 申請回数)` で追跡可能にする。これにより承認者・申請者の双方が過去の差戻理由やコメントを確認できる。

### Consequences

- 金額変更によるゴール変更を自然に処理できる（再申請時に再計算されるため）
- 組織変更（役割の上位役割変更）があった場合も、再申請時に最新の組織構造が反映される
- 承認者は再度全ステップで承認する必要があり、差戻と再申請が頻繁な場合は承認者の負担が増える
- 申請回数と対象IDにより、同一対象の申請履歴（過去の差戻理由・承認コメント含む）を時系列で参照できる
- 申請回数はシンプルな通番であり、表示（「第2回申請」）やクエリ（最新申請の取得）が容易
