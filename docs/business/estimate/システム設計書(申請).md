## 1. 概要とスコープ

### 1.1 申請とは

見積（特定のバリエーション）を上長へ提出し、承認を求めるアクション。承認された見積からのみ受注を作成できる（承認不要の見積を除く）。

業務フロー上の位置づけ（`business-definition.md` / `システム設計書(見積).md` §1.1）:

```
見積作成 → バリエーション折衝 → [申請] → [承認] → 受注作成
```

### 1.2 スコープ

| 含む | 含まない |
| ---- | -------- |
| 見積の申請（提出）・承認・差戻・取下 | 価格の申請（別 issue） |
| 承認チェーンの構築・承認ステップ進行 | 受注確定時の承認（将来検討） |
| 承認免除（承認不要見積の確定） | 通知・メール連携 |
| 申請プレビュー（確認モーダルの中身） | 承認者への割り当て変更 UI |

### 1.3 基本方針（確定事項）

- 申請テーブルは **見積専用**（ADR-0053）。対象は **バリエーション**で、`variation_id` に DB外部キーを張る。
- 承認ステップは **申請時に全ステップ事前生成**（ADR-0002）。先頭のみ「承認待ち」、残りは「未着手」。
- 承認のスタートは **申請者の上位役割**、ゴールは **金額で決まる役職**（ADR-0003 / 0055）。常に最低1ステップの承認を求める（ADR-0003）。
- 承認不要（消耗品のみ／10万円未満／事後見積）は **承認免除テーブル**で表現（ADR-0054）。
- 差戻後の再申請は **承認フローを最初からやり直す**（ADR-0005）。旧レコードは履歴として保持。
- 承認要否の **判定（純関数）** と **永続化（申請/免除レコード生成）** を分離する。

## 2. サブドメイン境界・集約・配置

### 2.1 サブドメイン配置

見積申請は **`estimate` サブドメイン内**に配置する。既存の `estimate` サブドメインは `Estimate` / `EstimateVariation` / `TaxRateMaster` など見積に密な複数集約を同居させており（「1サブドメイン＝1集約」を厳密には採らない運用）、見積専用テーブル（ADR-0053）である申請もこの既存パターンに従う。

```
src/server/subdomains/estimate/
  domain/
    entities/        EstimateApplication（集約ルート） / EstimateApprovalStep / 承認・差戻・取下イベント / EstimateApprovalExemption（独立集約ルート）
    values/          EstimateExemptionReason ほか（状態 enum は持たず導出。ADR-0058）
    services/        ApprovalChainBuilder
    policies/        ApprovalRequirementPolicy（金額閾値・消耗品判定）
    repositories/    EstimateApplicationRepository / EstimateApprovalExemptionRepository（interface）
  application/
    commands/        SubmitApplication / ApproveStep / RejectStep / WithdrawApplication
    queries/         PreviewApplication / GetApplicationDetail / SearchApprovalInbox
  infrastructure/    Prisma 実装・Mapper・QueryService
```

> 承認進行ロジック（`ApprovalChainBuilder` / `ApprovalRequirementPolicy`）は、将来 **価格申請が追加された時点で** 共通部分を `shared` へ抽出して共有する（YAGNI）。それまでは estimate 内に置く。

### 2.2 集約境界

| 集約ルート | 内包 | 対象への参照 |
| ---------- | ---- | ------------ |
| `EstimateApplication` | `EstimateApprovalStep[]` ＋ 各ステップの承認/差戻イベント ＋ 取下イベント（1:N） | `variationId`（別集約 estimate を ID 参照） |
| `EstimateApprovalExemption` | なし（薄い独立集約） | `variationId`（同上） |

- 申請と承認ステップ・各決定イベントは「申請がそのライフサイクルを完全に支配する」ため同一集約。ステップ・承認/差戻/取下イベントは `EstimateApplication` 経由でのみ生成する（ADR-0027 のバレル＋ESLint 境界強制に準拠）。状態は保存せず行の存在から導出（ADR-0058）。
- 対象バリエーションは **別集約**（estimate）。承認に必要な属性（`finalTotal`・明細の商品区分・見積区分）は、集約境界を越えてドメインに渡す際は **メソッド引数で渡す**（ADR-0030）。リポジトリポートをドメインに持たせない（ADR-0052 と同方針）。

## 3. データモデル

> **本節は #375 のグリルで当初案から大きく改訂した（ADR-0058）。** 要点:
> - model/enum 名は **すべて `Estimate` プレフィックス**で統一（`EstimateApplication` に合わせる）。
> - 承認/差戻/取下は **終端イベント表**として分離し、申請・ステップの状態は **行の存在＋順序から導出**（`status` 保存カラム・enum 2種を廃止。ADR-0058）。
> - 「1見積1前進バリエーション」の **DBバックストップ（冗長 `estimate_id`＋部分ユニーク）は廃止**し、アプリ層（見積アグリゲートの楽観ロック）に一元化（ADR-0058）。
> - `submittedAt`／`exemptedAt`／`decidedAt` は **`createdAt` に集約**（業務日時のcreatedAt集約・Order系と同方針）。
> - `onDelete` は全 FK **既定（Restrict）**。申請・ステップ・イベントは履歴で物理削除しない。
> - 数値カラムは **CHECK 下限のみ**（`attempt >= 1` / `step_order >= 1`。`version` は CHECK 対象外）。

### 3.1 Enum

状態 enum（`ApplicationStatus` / `ApprovalStepStatus`）は §3.6 の導出で表現するため**持たない**。残る enum は免除理由のみ。

```prisma
enum EstimateExemptionReason {
  CONSUMABLE_ONLY  // 消耗品のみ（金額無関係・ADR-0004）
  BELOW_THRESHOLD  // 10万円未満
  AFTER_REPAIR     // 事後見積（作業済みのため・§6.3）
}
```

### 3.2 申請テーブル（estimate_applications）

申請の状態（PENDING/APPROVED/REJECTED/WITHDRAWN）は保存せず §3.6 で導出する。申請行自体は生成後ほぼ不変だが、`version` は **アグリゲート全体（子イベント挿入を含む）の変更を直列化する楽観ロック**として持つ（ADR-0039・ADR-0058）。

```prisma
model EstimateApplication {
  id String @id @db.Uuid // ドメイン層で UUIDv7 生成

  // 対象バリエーション（見積専用なので FK を張れる。ADR-0053）
  variationId String            @map("variation_id") @db.Uuid
  variation   EstimateVariation @relation(fields: [variationId], references: [id])

  // 申請回数（同一バリエーションへの通番。差戻→再申請で +1。ADR-0005）
  // 自然キー・業務可視値として保存（導出可能だが残す。ADR-0028/0033 流儀）
  attempt Int @default(1) // CHECK (attempt >= 1) を手書きSQLで付与

  // 申請者
  applicantEmployeeId String   @map("applicant_employee_id") @db.Uuid
  applicant           Employee @relation("ApplicationApplicant", fields: [applicantEmployeeId], references: [id])

  // 承認フローのゴール役職（金額から算出。NOT NULL ＝ 承認を通る申請は必ずゴールを持つ。ADR-0055）
  finalApprovalPositionId String   @map("final_approval_position_id") @db.Uuid
  finalApprovalPosition   Position @relation(fields: [finalApprovalPositionId], references: [id])

  steps      EstimateApprovalStep[]
  withdrawal EstimateApplicationWithdrawal? // 取下イベント（行の存在＝取下）

  // 楽観ロック（ADR-0039・ADR-0058: アグリゲート変更の直列化）
  version Int @default(1)

  // 申請日時 = createdAt（業務日時 submittedAt は冗長なため持たない）
  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(3)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(3)

  @@unique([variationId, attempt]) // ADR-0005 を見積専用で簡約。自然キー
  @@index([variationId])
  @@map("estimate_applications")
}
```

> **「1見積1前進バリエーション」の担保はアプリ層（ADR-0058）**。状態を導出にしたため部分ユニークは張れない。`SubmitApplication`／免除確定は見積アグリゲートを版管理付きで読み書きし、同一見積への同時操作を `Estimate.version` で直列化する。冗長 `estimate_id` 列は持たない（見積へは variation 経由で辿る）。

### 3.3 承認ステップテーブル（estimate_approval_steps）

申請時に全ステップ事前生成（ADR-0002）。**骨格のみ・完全不変・nullable ゼロ**。承認/差戻の結果は §3.4 のイベント表に分離する。ステップ状態は §3.6 で導出する。

```prisma
model EstimateApprovalStep {
  id String @id @db.Uuid

  applicationId String              @map("application_id") @db.Uuid
  application   EstimateApplication @relation(fields: [applicationId], references: [id]) // onDelete 既定(Restrict)

  stepOrder Int @map("step_order") // 1, 2, 3...  CHECK (step_order >= 1)

  // 承認対象役割（この役割に属する従業員が承認する）
  roleId String @map("role_id") @db.Uuid
  role   Role   @relation(fields: [roleId], references: [id])

  // 結果イベント（行の存在＝決定）
  approval  EstimateStepApproval?
  rejection EstimateStepRejection?

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(3)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(3)

  @@unique([applicationId, stepOrder])
  @@index([roleId]) // 承認Inboxは反結合で引く（§3.6 / 論点: AWAITING部分インデックスは張れない）
  @@map("estimate_approval_steps")
}
```

### 3.4 承認・差戻・取下イベント表

各ステップは高々1つの決定（承認 **or** 差戻）を持ち、`stepId` を自然キー（@id）とする（ADR-0041 と同型）。取下は申請レベルのイベントで `applicationId` を自然キーとする。各イベントの発生日時はイベント行の `createdAt`。

```prisma
// 承認イベント（行の存在＝このステップは承認済）
model EstimateStepApproval {
  stepId String               @id @map("step_id") @db.Uuid
  step   EstimateApprovalStep @relation(fields: [stepId], references: [id]) // 既定(Restrict)

  approverEmployeeId String   @map("approver_employee_id") @db.Uuid
  approver           Employee @relation("StepApprover", fields: [approverEmployeeId], references: [id])

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(3)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(3)

  @@index([approverEmployeeId])
  @@map("estimate_step_approvals")
}

// 差戻イベント（行の存在＝このステップは差戻）
model EstimateStepRejection {
  stepId String               @id @map("step_id") @db.Uuid
  step   EstimateApprovalStep @relation(fields: [stepId], references: [id]) // 既定(Restrict)

  rejectedByEmployeeId String   @map("rejected_by_employee_id") @db.Uuid
  rejectedBy           Employee @relation("StepRejecter", fields: [rejectedByEmployeeId], references: [id])

  // 差戻理由は必須（NOT NULL・default なし）。意味を持つ表にだけ comment を置く
  comment String @db.VarChar(2000)

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(3)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(3)

  @@index([rejectedByEmployeeId])
  @@map("estimate_step_rejections")
}

// 取下イベント（行の存在＝この申請は取下）
model EstimateApplicationWithdrawal {
  applicationId String              @id @map("application_id") @db.Uuid
  application   EstimateApplication @relation(fields: [applicationId], references: [id]) // 既定(Restrict)

  withdrawnByEmployeeId String   @map("withdrawn_by_employee_id") @db.Uuid
  withdrawnBy           Employee @relation("ApplicationWithdrawer", fields: [withdrawnByEmployeeId], references: [id])

  createdAt DateTime @default(now()) @map("created_at") @db.Timestamptz(3)
  updatedAt DateTime @updatedAt @map("updated_at") @db.Timestamptz(3)

  @@index([withdrawnByEmployeeId])
  @@map("estimate_application_withdrawals")
}
```

> **跨ぎレースの締め（ADR-0058）**: 「同一ステップへ承認と差戻を同時」「最終承認と取下を同時」は別表の `@id` では防げない。承認/差戻/取下はすべて申請アグリゲートの変更なので、`EstimateApplication.version` でアグリゲート全体の変更を直列化して防ぐ。

### 3.5 承認免除テーブル（estimate_approval_exemptions）

```prisma
model EstimateApprovalExemption {
```

> 免除は冗長 `estimate_id` を持たない（見積へは variation 経由）。「1見積1前進」はアプリ層（ADR-0058）。

### 3.6 状態の導出（保存しない）

```
# 承認ステップの状態
差戻行(EstimateStepRejection)あり          → REJECTED
承認行(EstimateStepApproval)あり           → APPROVED
決定行なし＋下位stepOrderが全て承認済＋申請PENDING → AWAITING
決定行なし＋下位に未承認あり                → NOT_STARTED

# 申請の状態
取下行(EstimateApplicationWithdrawal)あり  → WITHDRAWN   # 最優先
いずれかのステップに差戻行                  → REJECTED
全ステップに承認行                          → APPROVED
上記いずれも無し                           → PENDING
```

- 承認 Inbox（AQ3）は `status='AWAITING'` で引けず、「承認行も差戻行も無く下位が全て承認済のステップ」を反結合（NOT EXISTS）で引く。
- 見積表示ステータス（見積側 §1.3）は、申請テーブル・ステップ・3種のイベント表・免除テーブルを参照する読み取りモデルの関心。

### 3.7 規約・逆参照

- **CHECK**（ADR-0019・手書きSQL）: `attempt >= 1` / `step_order >= 1`（下限のみ）。`version` は対象外。
- **onDelete**: 全 FK 既定（Restrict）。申請・ステップ・イベント・免除は履歴で物理削除しない。
- **逆参照**: `EstimateVariation`（applications / exemption）/ `Employee`（applicant・各種 approver/rejecter/withdrawer/exemptor）/ `Position`（finalApprovalPosition）/ `Role`（approvalSteps）に逆参照リレーションを追加（実装 issue で対応）。`Estimate` への逆参照は不要（冗長 estimate_id を持たないため）。

## 4. 承認要否の判定（ApprovalRequirementPolicy）

副作用のない純関数。入力はバリエーションのスナップショット属性（ADR-0030 でメソッド引数として渡す）。**永続化を伴わない**ため、画面表示・申請プレビュー・申請実行のいずれからも呼べる。

### 4.1 入力

| 入力 | 由来 |
| ---- | ---- |
| `finalTotal`（税込合計） | `EstimateVariation.finalTotal`（ADR-0055） |
| 明細の商品区分の集合 | 各明細 `EstimateItem.product.category`（セット構成の末端含む） |
| 見積区分 | `Estimate.estimateType`（事後見積 = AFTER） |

### 4.2 判定ロジック（評価順）

```
judgeRequirement(finalTotal, categories, estimateType):
  # 1. 免除条件（上から評価し最初に一致したものを免除理由とする）
  if estimateType == AFTER:                 return Exempt(AFTER_REPAIR)
  if categories がすべて CONSUMABLE:         return Exempt(CONSUMABLE_ONLY)   # ADR-0004（金額無関係）
  if finalTotal < 100_000:                  return Exempt(BELOW_THRESHOLD)

  # 2. 承認必要 → ゴール役職を金額から決定（ADR-0055・税込）
  if finalTotal >= 30_000_000:  return Required(社長)
  if finalTotal >= 10_000_000:  return Required(本部長)
  if finalTotal >=  1_000_000:  return Required(部長)
  return Required(課長)                       # 10万円以上 100万円未満
```

> **「消耗品のみ」の判定範囲（確定）**: セット商品（区分 SET）はそれ自体が 0 円のグループであり、価格を持つのは構成末端の明細（個別商品／消耗品）。判定は **価格を持つ末端明細（通常明細＋セット構成明細）の区分**で行い、セット商品区分そのものは判定対象に含めない。これは金額計算が見ている対象（ADR-0047 の価格付き末端行）と一致し、セットで消耗品を偽装した承認バイパスも塞がれる。

## 5. 承認チェーンの構築（ApprovalChainBuilder）

### 5.1 前提となる組織モデル（既存スキーマ）

- `Employee.superiorRoleId` … 承認フローの起点となる役割（§承認フローのスタート）。
- `Role.superiorRoleId` … 役割の上位役割。これを辿って上位へ進む。
- `Role.positionId → Position` … 役割の役職。`Position.superiorPositionId` で役職階層（課長 < 部長 < 本部長 < 社長）。
- **重要な不変条件**: 「上位役割に指定できるのは役職の上位役職に属する役割のみ」（`business-definition.md:368`）。
  → 役割チェーンを1段上がると役職も**ちょうど1段**上がる。ゴール役職を飛び越さない。

### 5.2 アルゴリズム

```
buildChain(applicant, goalPosition):
  start = applicant.superiorRole
  if start is null:                         raise エラー（上位役割未設定）

  steps = [start]                           # ADR-0003: 常に最低1ステップ
  current = start
  # 起点の役職が既にゴール以上なら while に入らず起点1ステップのみ（ADR-0003）
  while positionLevel(current.position) < positionLevel(goalPosition):
    current = current.superiorRole
    if current is null:                     raise エラー（チェーンが途切れ・ゴール役職に到達不能）
    steps.append(current)

  # 各ステップ役割に承認者が存在するか検証（ADR-0002）
  for step in steps:
    if membersOf(step.role) is empty:       raise エラー（承認者不在: step.role）

  # ステータス初期化（ADR-0002）
  steps[0].status   = AWAITING              # 先頭のみ承認待ち
  steps[1:].status  = NOT_STARTED           # 残りは未着手
  return steps
```

### 5.3 補足ルール（業務定義より）

- 承認ステップの役割に **申請者自身が含まれる場合（兼務）は自己承認可**（ADR-0003）。
- ゴール役職（`finalApprovalPositionId`）は §4.2 で算出した役職。チェーン末尾ステップの役割の役職と一致する（5.1 の不変条件による）。
- 「常に上位に判断を仰ぐ」ため、申請者の役職がゴール以上でも最低1ステップは生成される（ADR-0003）。

### 5.4 例

申請者の上位役割 = 「大阪市南課長」（役職: 課長）、見積 = 税込 150 万円（ゴール: 部長）の場合:

```
steps = [ 大阪市南課長(課長) → 大阪市部長(部長, 最終) ]
        AWAITING               NOT_STARTED
finalApprovalPositionId = 部長
```

## 6. 申請フロー（プレビュー → 確認モーダル → 実行）

UI は常に「申請」ボタンを表示する（承認要否でボタンを出し分けない）。押下時に **確認モーダル**で分岐結果をプレビューし、利用者の確認後に実行する。

### 6.1 シーケンス

```
[申請]ボタン押下
   │
   ▼
PreviewApplication(variationId)  … クエリ（副作用なし）
   │   judgeRequirement + buildChain を試算
   ▼
確認モーダル表示
   ├─ 承認不要      → 「承認不要です（理由: 消耗品のみ）。確定しますか？」
   ├─ 承認必要      → 「承認フロー: 課長 → 部長（最終）。申請しますか？」
   └─ 申請不可      → 「◯◯役割に承認者がいないため申請できません」（実行ブロック）
   │
   ▼ 利用者が確認
SubmitApplication(variationId)   … コマンド（永続化）
   ├─ 承認不要 → EstimateApprovalExemption を1件生成
   └─ 承認必要 → EstimateApplication + EstimateApprovalStep[] を生成
```

### 6.2 PreviewApplication（クエリ）の出力

```
PreviewResult =
  | { kind: 'EXEMPT',    reason: EstimateExemptionReason }
  | { kind: 'REQUIRED',  goalPosition, steps: [{ order, roleName, positionName }] }
  | { kind: 'BLOCKED',   reason: 'NO_SUPERIOR_ROLE' | 'NO_APPROVER', detail }
```

- モーダルでは承認必要時に **チェーン全体**（課長 → 部長 → … ）を表示する（ADR-0002 の「フロー全体を可視化」を UI で実現）。
- 申請不可（BLOCKED）は実行に進ませない。ADR-0002 の「承認者不在による停滞を未然に防ぐ」がここで効く。

### 6.3 SubmitApplication（コマンド）の分岐

`judgeRequirement` を再評価し（プレビューと同じ純関数）、結果に応じて:
- `EXEMPT` → `EstimateApprovalExemption` を生成（§8）。
- `REQUIRED` → `buildChain` で `EstimateApplication` ＋ 全 `EstimateApprovalStep` を生成（§3.2 / §3.3）。`attempt` は同一バリエーションの過去申請の最大 +1（初回は 1）。
- `BLOCKED` → 例外（ADR-0038: 失敗は例外）。

> プレビューと実行の間に金額・組織が変わる TOCTOU は、対象バリエーションの楽観ロック（version）で検知する（ADR-0039）。

## 7. 承認・差戻・取下フロー

### 7.1 承認（ApproveStep コマンド）

- 入力: `stepId` ＋ 承認者従業員 ID。
- 前提: 当該ステップが `AWAITING`、かつ承認者が `step.role` のメンバー（兼務の自己承認も可・ADR-0003）。
- 処理:
  1. 当該ステップに `EstimateStepApproval` を1件生成（承認者・承認日時=createdAt を記録）。
  2. これにより次ステップは導出上 `NOT_STARTED → AWAITING` に進む（明示更新は不要）。
  3. 全ステップに承認行が揃えば、申請状態は導出上 APPROVED になる（§3.6）。
- 同時実行（同一ステップへの承認/差戻、最終承認と取下の競合）は `EstimateApplication.version` で直列化する（ADR-0058）。

### 7.2 差戻（RejectStep コマンド）

- 入力: `stepId` ＋ 差戻者従業員 ID ＋ コメント（差戻理由）。
- 処理:
  1. 当該ステップに `EstimateStepRejection` を1件生成（差戻者・差戻理由 comment[必須]・差戻日時=createdAt を記録）。
  2. これにより申請状態は導出上 REJECTED になる（§3.6）。
- 再申請: 差戻された申請・ステップ・イベントは **削除せず履歴保持**。利用者がバリエーションを修正し再度 §6 の申請を行うと、**新しい申請レコード（attempt+1）＋新チェーン**が生成される（ADR-0005）。金額が変われば §4 でゴールも再計算される。

### 7.3 取下（WithdrawApplication コマンド）

- 入力: `applicationId`（申請者本人）。
- 処理: `EstimateApplicationWithdrawal` を1件生成。これにより申請状態は導出上 WITHDRAWN になる（§3.6・最優先）。進行中ステップも導出上終了扱い。履歴は保持。再申請は §6 と同じく新レコード。

### 7.4 承認権限 ［§9 保留事項の確定］

承認できるのは **当該ステップの承認対象役割（`step.role`）に属する従業員**（`EmployeeRole` 経由）。その役割の従業員が **1人でも承認すればステップクリア**。**部署スコープ等の追加制限は設けない**（見積側 §9「見積承認: 保留」を本書で確定）。承認チェーンは §5.2 で申請者の上位役割ラインに限定されるため、役割メンバーシップのみで統制が閉じる。

---

## 8. 承認免除フロー

§6.3 の `EXEMPT` 分岐で `EstimateApprovalExemption` を1件生成する。

- `reason` は §4.2 の免除理由（AFTER_REPAIR / CONSUMABLE_ONLY / BELOW_THRESHOLD）。
- `exemptedByEmployeeId` = 操作者、確定日時 = `createdAt`（業務日時 exemptedAt は持たない）。
- 生成後のバリエーションは **編集不可**（承認済と同じ扱い・ADR-0054）。内容変更は複製（C6）で新バリエーションを作る。
- 免除後はそのまま受注作成に進める（§4.3 見積側）。

## 9. 見積表示ステータスへの統合（§1.3 拡張）

見積側 §1.3 の導出ルールに「承認免除」行を追加する（上から評価し最初に一致）。

| 条件 | 表示ステータス | 次のアクション |
| ---- | -------------- | -------------- |
| `OrderCancellation` が存在 | 受注取消済 | 再見積（複製） |
| `OrderConfirmation` が存在 | 受注確定済 | - |
| `Order` が存在 | 受注作成済 | 受注確定 |
| 申請に承認済レコードあり | 承認済 | 受注作成 |
| **承認免除レコードあり** | **承認不要（受注可）** | **受注作成** |
| 申請中レコードあり | 申請中 | 承認 / 差戻 |
| 差戻レコードあり | 差戻 | 再申請 |
| 取下レコードあり | 取下 | 再申請 |
| いずれも無し | 作成中 | 申請 |

> 申請テーブルは履歴を複数持つため、導出では **最新の有効なイベント**（最大 attempt の申請、または免除）を評価する。承認済と免除は通常共存しないが、差戻履歴と免除が併存しうる場合は最新の確定（承認済 or 免除）を優先する。

## 12. 不変条件・バリデーション

ドメイン層（構造的不変条件・ADR-0029）:
- `EstimateApplication` は承認ステップを **1件以上**持つ（免除はこのテーブルに入らないため・ADR-0054）。
- `finalApprovalPositionId` は **NOT NULL**。
- `stepOrder` は 1 から連番・`@@unique(applicationId, stepOrder)`。
- 先頭ステップのみ `AWAITING`、それ以外は生成時 `NOT_STARTED`。

ユースケース層（業務イベント制約・ADR-0029）:
- 無効（INACTIVE）バリエーションは申請不可（§3.4 見積側）。
- 申請中・承認済・免除済のバリエーションは無効化・編集不可（§3.4）。
- 1見積につき前進（申請中・承認済・免除）は1バリエーションのみ。アプリ層＝見積アグリゲートの楽観ロックで担保（ADR-0058。DB部分ユニークは廃止）。
- 申請者の上位役割未設定はエラー。チェーン上に承認者不在の役割があればエラー（§5.2）。
- 承認は当該ステップの役割メンバーのみ（§7.4）。

## 13. 確定事項と後続スコープ

設計レビュー（issue #361）で以下を確定した。

1. **サブドメイン配置**（§2.1）: **`estimate` サブドメイン内**に配置（既存の集約同居パターン準拠）。承認共通ロジックは価格申請追加時に `shared` へ抽出。
2. **「消耗品のみ」判定**（§4.2）: **価格を持つ末端明細**（通常明細＋セット構成明細）の区分で判定。セット商品（0円グループ）は判定対象外。
3. **承認権限**（§7.4）: **役割メンバーシップのみ**（部署スコープ等の追加制限なし）。
4. **承認 Inbox**（AQ3）: 存在と権限ロジックのみ本書で定義。**一覧 UI・表示仕様は別 issue**。

後続スコープ:

5. 本書は **テーブル・フロー設計まで**（issue #361）。**Prisma スキーマ追加・migration・ドメイン実装・UI は別 issue** に分割する。
