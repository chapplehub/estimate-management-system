## 1. システム概要

### 1.1 業務フロー

```
1. 見積作成
   ↓
2. バリエーション作成 (複数の選択肢を作成)
   ↓
3. 客先と折衝
   ↓
4. 見積申請 (上長へ)
   ↓
5. 見積承認
   ↓
6. 受注作成
   ↓
7. 受注確定
   ↓
8. 受注票出力
```

### 1.2 見積の種類

| 見積区分 | 接頭辞     | 説明                       | 備考                           |
| -------- | ---------- | -------------------------- | ------------------------------ |
| 新規見積 | N (New)    | 新規の機械を設置する場合   | -                              |
| 修理見積 | R (Repair) | 設置済み機械の修理 (事前)  | 修理予定日が必要               |
| 事後見積 | A (After)  | 緊急対応で修理後に見積作成 | 修理実施日・緊急対応理由が必要 |

### 1.3 見積ステータス

見積の状態は単一のステータスカラムでは管理せず、**3 つの独立した関心事の合成（導出値）**として表現する。

| 関心事   | 真実の源                        | 取り得る値                             |
| -------- | ------------------------------- | -------------------------------------- |
| 見積本体 | 申請・受注が未着手か            | 作成中                                 |
| 申請状態 | 共通申請テーブル (ADR-0001)     | 申請中 / 差戻 / 取下 / 承認済          |
| 受注状態 | `Order` / `OrderConfirmation` / `OrderCancellation` の存在 | 受注作成済 / 受注確定済 / 受注取消済 |

#### 表示ステータスの導出ルール（上から評価し、最初に一致したもの）

| 条件                               | 表示ステータス | 次のアクション |
| ---------------------------------- | -------------- | -------------- |
| `OrderCancellation` が存在         | 受注取消済     | 再見積（複製） |
| `OrderConfirmation` が存在         | 受注確定済     | -              |
| `Order` が存在（確定/取消なし）    | 受注作成済     | 受注確定       |
| 申請テーブルに承認済レコードあり   | 承認済         | 受注作成       |
| 申請テーブルに申請中レコードあり   | 申請中         | 承認 / 差戻    |
| 申請テーブルに差戻レコードあり     | 差戻           | 再申請         |
| 申請が取下げられた                 | 取下           | 再申請         |
| 申請レコードなし                   | 作成中         | 申請           |

> **設計判断**: `Estimate` / `EstimateVariation` / `Order` にライフサイクルの `status` カラムを持たせない（`OrderStatus` enum は廃止）。受注の確定/取消は `OrderConfirmation` / `OrderCancellation` の行の存在で表現し、表示ステータスは導出する。理由は §11 冒頭の注記を参照。`VariationStatus`（有効/無効, §3.3）はライフサイクルとは別の関心事のため保持する。

## 2. 見積番号の採番ルール

### 2.1 番号形式

```
N2500001
│││└──┴─ 連番(5桁): 00001〜99999
││└───── 年度(2桁): 25 = 2025年度(2025/4〜2026/3)
│└────── 区分(1文字): N(新規) / R(修理) / A(事後)
└─────── 固定
```

### 2.2 採番仕様

- **年度**: 4 月始まり (2025 年 4 月〜2026 年 3 月 = 2025 年度)
- **リセット**: 年度ごとにリセット
- **容量**: 年間 99,999 件まで対応可能 (想定: 年間 1,000 件)
- **欠番**: 許容される (削除した見積番号は再利用しない)

### 2.3 採番タイミング

見積保存時に自動採番

### 2.4 実装例

```typescript
// 見積番号生成例
N2500001; // 2025年度の新規見積 1番
R2500123; // 2025年度の修理見積 123番
A2510500; // 2025年度の事後見積 10500番
```

## 3. 見積バリエーション

### 3.1 概要

- 1 つの見積番号に対して複数のバリエーション(選択肢)を作成可能
- 見積と見積バリエーションは 1 対多 の関係
- 客先との折衝用に複数案を用意するための機能

### 3.2 バリエーション番号

```
N2500001-01
│       └─┴─ バリエーション番号(2桁): 01〜99
└─────────── 親見積番号
```

### 3.3 バリエーションのステータス

| ステータス      | 説明           | 制約                         |
| --------------- | -------------- | ---------------------------- |
| 有効 (ACTIVE)   | 使用可能な状態 | 編集・申請可能               |
| 無効 (INACTIVE) | 使用しない状態 | 編集・申請不可、有効化は可能 |

### 3.4 バリエーションの制約

#### 申請の制約

- **1 見積番号につき申請できるバリエーションは 1 つのみ**
- 申請中の見積に対して、他のバリエーションを申請することは不可

#### 無効化の制約

- 申請中・承認済みのバリエーションは無効化不可
- 無効状態のバリエーションは編集・申請不可
- 無効 → 有効への変更は無条件で可能

### 3.5 バリエーションテーブルの項目

```
- バリエーション番号
- ステータス (有効/無効)
- 客先メモ
- 内部メモ
- 全体値引金額
- 見積明細 (1対多)
```

> 申請・承認情報はバリエーション自身では保持しない。共通申請テーブル（ADR-0001）が「対象ID = バリエーションID」として一元管理する（§11 冒頭注記）。

## 4. 見積から受注への流れ

### 4.1 基本フロー

```
見積承認済
  ↓
受注作成 (DRAFT)
  ↓ 納期・メモ入力
受注確定 (CONFIRMED)
  ↓
受注票出力
```

### 4.2 見積と受注の関係

- **見積番号 = 受注番号** (受注番号は別途持たない)
- **1 見積バリエーション : 1 受注** (1 対 1 の関係)
- 受注テーブルは見積テーブルを参照 (スナップショット不要)
- **受注明細は持たない**: 受注明細は見積明細と常に完全一致するため、`Order` は `EstimateVariation → EstimateItem` を辿って明細を得る（独立した受注明細テーブル `OrderItem` は作らない）

### 4.3 受注作成の条件

```
✅ 承認済みバリエーションから作成可能
✅ 承認不要な見積も作成可能
✅ 誰でも作成可能 (権限: 作成者 + 同部署メンバー)
```

### 4.4 受注ステータス

| 表示ステータス | 説明   | 導出条件                       | 編集可能項目 |
| -------------- | ------ | ------------------------------ | ------------ |
| 受注作成済     | 作成中 | `Order` のみ存在               | メモ         |
| 受注確定済     | 確定   | `OrderConfirmation` が存在     | メモのみ     |
| 受注取消済     | 取消   | `OrderCancellation` が存在     | 編集不可     |

> ステータスは列ではなく `OrderConfirmation` / `OrderCancellation` の行の存在から導出する（§1.3 / §11.4）。納期は DRAFT 中は保持せず、確定操作で `OrderConfirmation.deliveryDate` に入力する（§4.5）。

### 4.5 受注確定の必須項目

```
必須:
- 納期

任意:
- メモ
```

### 4.6 受注票の出力

- **DRAFT 状態**: SAMPLE 印字で出力可能
- **CONFIRMED 状態**: 正式版を出力可能

### 4.7 受注取消

```
取消処理:
1. `OrderCancellation` を 1 件生成（取消された `Order` は旧バリエーションに紐づいたまま残り、表示ステータスは「受注取消済」に導出される）
2. 見積データを複製
3. 新バリエーション作成画面へ遷移
4. 新バリエーションは申請レコード・受注ともになし → 表示ステータスは自動的に「作成中」（§1.3 の導出ルール）
```

### 4.8 データ整合性

- **受注作成後、見積バリエーションは編集不可**
- 受注作成 = `Order` レコード生成（status カラムは持たず、確定/取消行なし＝「受注作成済」に導出。§1.3）
- 受注確定 = `OrderConfirmation` レコード生成（納期を必須入力）。表示は「受注確定済」に導出
- 受注は見積データを参照 (金額等の変更は不可)

## 5. 見積の複製

### 5.1 複製可能条件

- **すべてのステータスから複製可能**
- **誰でも複製可能** (他人の見積も可)
- **複製回数無制限**

### 5.2 バリエーション選択

```
選択可能:
- 複数選択可能
- 無効も含めてすべて選択可能
- 選択なしもOK (見積のみ複製)

選択順序:
- 保持される
- 連番で振り直し (01, 02, 03...)
```

> **注記（ADR-0042）**: 「選択なしもOK（見積のみ複製）」は、複製を保存するアプリ層コマンド
> （C6 DuplicateEstimate）の責務外とする。集約の空見積不可（§C1）と衝突するため、C6 は最低 1
> バリエーションの選択を要求する。「見積のみ複製」はヘッダ（得意先・提出区分・税率・メモ等）を
> 引き継いだ新規作成として、将来プレゼン層で別途実現する。

### 5.3 引き継ぎ項目

#### 見積テーブル

| 項目       | 引き継ぎ | 備考               |
| ---------- | -------- | ------------------ |
| 見積区分   | ✅       | 変更可能           |
| 見積年月日 | 🔄       | 複製時の日付に更新 |
| 締切年月日 | 🔄       | 複製時から計算     |
| 提出区分   | ✅       | -                  |
| 得意先     | ✅       | -                  |
| 納品先     | ✅       | -                  |
| 消費税率   | 🔄       | 複製時点の税率     |
| メモ       | ✅       | -                  |
| 見積番号   | ❌       | 保存時に新規採番   |
| ステータス | ❌       | 作成中             |

#### バリエーションテーブル

| 項目                  | 引き継ぎ | 備考            |
| --------------------- | -------- | --------------- |
| メモ                  | ✅       | -               |
| 見積明細 (品目、数量) | ✅       | -               |
| 単価                  | ❌       | クリア (要入力) |
| 申請・承認情報        | ❌       | 複製先には申請レコードを作らない (共通申請テーブル / ADR-0001) |
| ステータス (有効/無効) | ❌       | すべて有効      |

### 5.4 複製元情報の保持

```
EstimateVariationCopy テーブル (交差テーブル / §11.2.1):
- copiedVariationId (複製先バリエーションID, @unique)
- sourceVariationId (複製元バリエーションID)

※ 見積・バリエーション本体には複製元の列を持たない（NULL 排除）。
※ 複製元見積は sourceVariation.estimateId から導出する。
```

### 5.5 複製フロー

```
1. 複製ボタン
   ↓
2. バリエーション選択ダイアログ
   ↓
3. 見積新規作成画面へ遷移 (プレビュー)
   - 複製データが表示される
   - 見積区分は変更可能
   - 単価は空欄 (要入力)
   ↓
4. ユーザーが編集
   ↓
5. 保存ボタン → 見積番号採番 & DB保存
```

## 6. 修理見積と事後見積

### 6.1 見積区分の構造

```
見積区分 (EstimateType): 3 値の enum
- NEW          (新規)
- REPAIR       (修理 / 事前)
- AFTER_REPAIR (事後修理)
```

UI 上は 3 つのセレクトボックス: [新規 / 修理 / 事後]
データ上も: estimateType（3 値 enum）で 1:1 管理（旧 isAfterService フラグは廃止）

> **設計判断**: `EstimateType{NEW, REPAIR}` + `isAfterService: Boolean` をやめ、3 値 enum に統合。理由:
>
> - 採番接頭辞 N/R/A（§2.1）・見積書タイトルの 3 分岐（§6.4）と 1:1 で対応し、(estimateType, isAfterService) → 3 値の変換が不要になる
> - `NEW × isAfterService=true` のような不正な組み合わせを型レベルで表現不可能にする（DDD / 型駆動設計）
> - 修理系の共通判定（修理対象機器・故障内容の必須化など）はドメイン層の `isRepairType()` 等のヘルパで吸収する

### 6.2 見積区分別の必須項目

#### 共通項目

```
- 見積年月日
- 締切年月日
- 得意先 or 納品先
- 消費税率
- 見積明細
```

#### 修理見積 (事前) のみ

```
- 修理対象機器 (商品ID)
- 故障内容
- 修理予定日
```

#### 事後見積のみ

```
- 修理対象機器 (商品ID)
- 故障内容
- 修理実施日 (過去日付)
- 緊急対応理由
```

### 6.3 事後見積の特殊制約

#### 承認フロー

```
✅ 承認不要 (すでに作業済みのため)
✅ 作成 → 即受注作成が可能
```

#### 金額チェック

```
10万円以上の場合:
- 保存時: 警告表示
- 受注作成時: 警告表示

警告内容:
「事後見積ですが、10万円を超えています。よろしいですか?」
[OK] [キャンセル]
```

#### 緊急対応理由

```
✅ 必須入力
✅ 見積書に特記事項として表示
```

### 6.4 見積書タイトル

| 見積区分 | タイトル       | 特記事項           |
| -------- | -------------- | ------------------ |
| 新規見積 | 新規設置見積書 | -                  |
| 修理見積 | 修理見積書     | -                  |
| 事後見積 | 事後修理見積書 | 緊急対応理由を表示 |

### 6.5 業務フロー比較

#### 修理見積 (事前)

```
1. 故障連絡
2. 見積作成
3. 申請 → 承認
4. 受注作成
5. 修理実施
6. 受注確定
```

#### 事後見積

```
1. 故障連絡 (緊急)
2. 修理実施 ★先に実施
3. 見積作成 (承認スキップ)
4. 受注作成 (必須)
5. 受注確定
```

## 7. 得意先と納品先の関係

### 7.1 基本構造

```
得意先 (Customer)
  └─ 納品先 (DeliveryLocation) ※1対多

取引フロー:
X社 → 得意先Y (卸値100万) → 納品先Z (販売価格120万)
```

### 7.2 提出区分による業務フロー

#### ステップ 1: 納品先見積作成

```
提出区分: 納品先
金額: 120万円 (販売価格)
制約: 申請・受注作成不可
用途: 得意先改訂の元データ
```

#### ステップ 2: 得意先改訂

```
「得意先改訂」ボタン
  ↓
納品先バリエーションを複製(改訂元の納品先バリエーションはメモ以外は編集・行追加削除不可にする = 凍結)
得意先バリエーションについても行追加削除は不可 (= 行構成固定。価格・数量・メモ等の調整は可)
  ↓
提出区分: 納品先 → 得意先
金額: 120万円 → 100万円 (調整)
粗利: 20万円を表示
```

#### ステップ 3: 申請・受注作成

```
提出区分: 得意先の見積のみ
申請・承認・受注作成が可能
```

### 7.3 見積書の出力内容

#### 納品先向け見積書

```
to: △△薬局 御中
from: 〇〇商事 (得意先)
金額: 120万円
粗利: 非表示
```

#### 得意先向け見積書

```
to: 〇〇商事 御中
from: X社
金額: 100万円
粗利: 20万円 ★表示
納品先: △△薬局 (参考表示)
```

### 7.4 受注票の内容

```
受注先: 得意先Y (〇〇商事)
納品先: 納品先Z (△△薬局)
金額: 100万円 (得意先への卸値)

※実際の納品は納品先Zへ行う
```

### 7.5 データ構造

```
見積テーブル:
- submissionType: CUSTOMER / DELIVERY_LOCATION
- customerId: 得意先ID (常に設定)
- deliveryLocationId: 納品先ID (常に設定)

得意先改訂の出自 (EstimateVariationRevision / §11.2.1):
- revisedVariationId: 改訂で生まれた得意先バリエーションID (@unique)
- sourceVariationId: 改訂元の納品先バリエーションID
※ 見積本体に revisedFromVariationId は持たない（NULL 排除・粒度をバリエーションに統一）

制約:
- 得意先選択時、配下の納品先のみ選択可能
- 両方常に設定、submissionTypeで用途を制御
```

## 8. 金額計算

### 8.1 金額計算フロー

```
明細レベル:
1. 基本金額 = 数量 × 単価 (端数切捨)
2. 掛率適用後金額 = 基本金額 × 掛率 (端数切捨)
3. 最終明細金額 = 掛率適用後金額 - 明細値引金額

見積レベル:
4. 最終明細金額小計 = Σ(最終明細金額)
5. 全体値引後金額 = 最終明細金額小計 - 全体値引金額
6. 消費税額 = 全体値引後金額 × 消費税率 (端数処理: 選択式)
7. 最終合計金額 = 全体値引後金額 + 消費税額
```

### 8.2 値引きの種類

#### ① 掛率値引 (明細レベル)

```
場所: 見積明細テーブル
計算: 単価100,000円 × 掛率0.95 = 95,000円
用途: 割引率による値引き
```

#### ② 明細値引 (明細レベル)

```
場所: 見積明細テーブル
計算: 掛率適用後95,000円 - 明細値引5,000円 = 90,000円
用途: 明細単位の値引き額
```

#### ③ 全体値引 (見積レベル)

```
場所: 見積バリエーションテーブル
計算: 明細小計1,000,000円 - 全体値引50,000円 = 950,000円
用途: 合計額からの値引き
```

### 8.3 消費税の計算

#### 計算タイミング

```
小計後に一括計算 (明細ごとではない)
```

#### 端数処理

```
見積テーブルに保持:
- taxRoundingType: ROUND_DOWN / ROUND_UP / ROUND

選択肢:
- 切り捨て (ROUND_DOWN)
- 切り上げ (ROUND_UP)
- 四捨五入 (ROUND)
```

### 8.4 粗利の計算 (得意先向け見積)

```
粗利 = 納品先価格 - 得意先価格

例:
納品先価格: 1,200,000円
得意先価格: 1,000,000円
粗利: 200,000円
粗利率: 16.7% (200,000 / 1,200,000)
```

> **格納先**: 計算式中の「納品先価格」は改訂明細ごとに `RevisedEstimateItemDetail.deliveryPrice`（改訂元納品先明細価格のスナップショット・§11.3.1）として保持する。改訂元バリエーションは凍結される（§7.2）が、得意先向け見積書に印字される確定値（§7.3）のため導出ではなくスナップショットを真実の源とする。

## 11. データベーススキーマ

> **本章の Prisma スキーマは概念設計であり、実装規約が別途優先される**
>
> - ID は UUIDv7（ドメイン層生成 / `@db.Uuid`）、日時は `@db.Timestamptz(3)`、カラム/テーブル名は snake_case（`@map` / `@@map`）、文字列は `@db.VarChar(N)`、数値は CHECK 制約 — ADR-0009 / 0010 / 0019 および実装済み `prisma/schema.prisma` に従う（本章の `@default(cuid())` / `DateTime` / `@db.Text` 表記は概念上のもの）
> - **申請・承認の状態は本章のモデルに持たせない**。ADR-0001 の共通申請テーブル（申請種別＋ポリモーフィックな対象ID）が唯一の真実の源。見積側は対象IDとして参照されるのみ
> - **ライフサイクルのステータスカラムを持たせない**。表示ステータスは「共通申請テーブル」と「`Order` / `OrderConfirmation` / `OrderCancellation` の存在」から導出する（§1.3）
> - **NULL を設計で排除する**（デシジョンツリー方針）。NULL になりうる列は次の順で処理し、原則テーブルに NULL を残さない:
>   1. 後で必ず埋まる値（状態・イベント）→ イベントテーブルに分離し「行の存在＝その状態」で表現（例: `OrderConfirmation` / `OrderCancellation`）
>   2. 特定の見積区分・提出区分でだけ意味を持つ値（適用不能）→ サブタイプテーブルに分割（例: `RepairEstimateDetail` / `AfterRepairEstimateDetail` / `RevisedEstimateItemDetail`）
>   3. オプショナルな関連 → 交差テーブル（非依存リレーションシップ）に分離（例: `EstimateVariationCopy` / `EstimateVariationRevision`）
>   4. それ以外の真に任意の値属性（メモ等）のみ NULL を許容する

### 11.1.1 修理見積詳細（サブタイプ）

`estimateType` でだけ意味を持つ「適用不能 NULL」を排除するため、修理関連を 2 つの排他的サブタイプテーブルに分割する。`NEW` はどちらの行も持たず、`REPAIR` / `AFTER_REPAIR` はそれぞれ対応する 1 行を持つ。共通列（対象機器・故障内容）は両表に重複させるが、いずれも **全列 NOT NULL** となり Estimate から NULL が消える。

```prisma
// REPAIR (事前修理) 専用。estimateType = REPAIR のときのみ 1:1 で存在
model RepairEstimateDetail {
  id                String   @id @default(cuid())
  estimateId        String   @unique
  estimate          Estimate @relation(fields: [estimateId], references: [id], onDelete: Cascade)

  targetProductId   String              // 修理対象機器 (必須)
  targetProduct     Product  @relation(fields: [targetProductId], references: [id])
  faultDescription  String   @db.Text   // 故障内容 (必須)
  scheduledRepairDate DateTime          // 修理予定日 (必須)

  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  @@index([targetProductId])
}

// AFTER_REPAIR (事後修理) 専用。estimateType = AFTER_REPAIR のときのみ 1:1 で存在
model AfterRepairEstimateDetail {
  id                String   @id @default(cuid())
  estimateId        String   @unique
  estimate          Estimate @relation(fields: [estimateId], references: [id], onDelete: Cascade)

  targetProductId   String              // 修理対象機器 (必須)
  targetProduct     Product  @relation(fields: [targetProductId], references: [id])
  faultDescription  String   @db.Text   // 故障内容 (必須)
  actualRepairDate  DateTime            // 修理実施日 (過去日付・必須)
  emergencyReason   String   @db.Text   // 緊急対応理由 (必須)
  afterServiceWarningAcknowledged Boolean @default(false)  // 10万円超警告の確認済フラグ

  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  @@index([targetProductId])
}
```

> **設計判断**: 共通項目を持つ親 `RepairInfo` ＋ 2 子テーブルの 3 テーブル構成も検討したが、`scheduledRepairDate`（REPAIR のみ）と `actualRepairDate`/`emergencyReason`（AFTER_REPAIR のみ）が親に同居すると結局 NULL が残るため、共通列を重複させてでも **2 サブタイプに完全分割**して全列 NOT NULL を優先した。`estimateType` と詳細テーブルの整合（排他的サブタイプ制約）はドメイン層で担保する（ADR-0019）。

### 11.2.1 バリエーション系譜（複製・得意先改訂）

複製（§5）と得意先改訂（§7）は「あるバリエーションから新しいバリエーションを派生させる」点で形は似るが、業務操作・ルール・将来増える属性が異なる**別概念**である。`kind` 判別子で 1 表に統合すると、改訂固有の属性（提出区分・粗利・改訂明細固有属性 `RevisedEstimateItemDetail`）が増えた時点で複製側が NULL になり、本章の NULL 排除方針に反する。よって**複製と改訂を別々の交差テーブルに分離**する（修理詳細を 2 サブタイプに分割したのと同じ判断軸）。

- いずれも `target` 側を **@unique** とし、1 つの派生先バリエーションは高々 1 つの出自を持つ
- どちらの表にも行が無い＝オリジナル（複製でも得意先改訂でもない）
- 全列 NOT NULL（nullable な自己FKを持たない）
- 「同じバリエーションが複製かつ改訂」になる二重派生の禁止（クロス表排他）はドメイン層で担保する（ADR-0019。修理サブタイプと同方針）
- 「見積単位の複製元」が必要な場合は `sourceVariation.estimateId` を辿って導出する（冗長保持しない）

```prisma
// 見積複製による派生 (§5)
model EstimateVariationCopy {
  id                String            @id @default(cuid())

  // 複製で生まれたバリエーション。1 件のみ → @unique
  copiedVariationId String            @unique
  copiedVariation   EstimateVariation @relation("VariationCopyTarget", fields: [copiedVariationId], references: [id], onDelete: Cascade)

  // 複製元バリエーション
  sourceVariationId String
  sourceVariation   EstimateVariation @relation("VariationCopySource", fields: [sourceVariationId], references: [id])

  createdAt         DateTime          @default(now())

  @@index([sourceVariationId])
}

// 得意先改訂による派生 (§7)
model EstimateVariationRevision {
  id                 String            @id @default(cuid())

  // 改訂で生まれた得意先バリエーション。1 件のみ → @unique
  revisedVariationId String            @unique
  revisedVariation   EstimateVariation @relation("VariationRevisionTarget", fields: [revisedVariationId], references: [id], onDelete: Cascade)

  // 改訂元の納品先バリエーション
  sourceVariationId  String
  sourceVariation    EstimateVariation @relation("VariationRevisionSource", fields: [sourceVariationId], references: [id])

  createdAt          DateTime          @default(now())

  @@index([sourceVariationId])
}
```

### 11.3.1 得意先改訂明細詳細（サブタイプ）

`deliveryPrice`（納品先価格スナップショット）は得意先改訂で生まれた明細でだけ意味を持ち、納品先見積明細では**適用不能**である。`EstimateItem` に nullable 列として持たせると本章の NULL 排除方針に反するため、得意先改訂明細の固有属性を排他的サブタイプ表 `RevisedEstimateItemDetail` に分離する（修理詳細を `RepairEstimateDetail` 等に分割したのと同じ判断軸）。**行が存在する ⟺ この明細は得意先改訂バリエーション（`submissionType = CUSTOMER` かつ `EstimateVariationRevision` 系譜あり）の明細**。全列 NOT NULL となり `EstimateItem` から NULL が消える。

`deliveryPrice` は改訂元の納品先明細価格の**スナップショット**として保持する。改訂元の納品先バリエーションは凍結される（§7.2）が、粗利は得意先向け見積書に印字される確定財務数値（§7.3 / §8.4）のため、導出に依存せずスナップショットを粗利の真実の源とする（§8.5）。

```prisma
// 得意先改訂で生まれた明細だけが持つ固有属性。行の存在＝改訂明細 (1:1)
model RevisedEstimateItemDetail {
  id                String       @id @default(cuid())
  estimateItemId    String       @unique
  estimateItem      EstimateItem @relation(fields: [estimateItemId], references: [id], onDelete: Cascade)

  // 改訂元の納品先明細価格のスナップショット (粗利の真実の源・§8.4)
  deliveryPrice     Decimal

  createdAt         DateTime     @default(now())
  updatedAt         DateTime     @updatedAt
}
```

> **設計判断**: `deliveryPrice` ただ 1 列のため最小の子テーブル（価格専用）も検討したが、改訂明細には今後 粗利関連列・改訂元明細リンク・提出区分固有データ等が増える見込みであり、価格専用名（例 `RevisionItemPrice`）では拡張時に名前と内容が乖離する。よって役割で命名した `RevisedEstimateItemDetail` に固有属性を集約する方針とした（全て NOT NULL で追加し NULL を再生産しない）。`submissionType`／改訂系譜と本表の存在の整合（排他的サブタイプ制約）はドメイン層で担保する（ADR-0019。修理サブタイプと同方針）。

### 11.4 受注

```prisma
model Order {
  id                String              @id @default(cuid())

  // 見積番号を受注番号として使用
  variationId       String              @unique
  variation         EstimateVariation   @relation(fields: [variationId], references: [id])

  // status カラムは持たない（OrderStatus enum は廃止）。
  // 確定/取消は後で埋まる「状態」のため、列ではなくイベントテーブルの行の存在で表現する。
  //   OrderConfirmation あり → 受注確定済 (旧 CONFIRMED)
  //   OrderCancellation あり → 受注取消済 (旧 CANCELLED)
  //   どちらもなし           → 受注作成済 (旧 DRAFT)
  // 納期は DRAFT 中は保持しない（納期入力＝確定操作の一部）。OrderConfirmation.deliveryDate に NOT NULL で持つ。

  orderDate         DateTime            @default(now())
  orderNote         String?             @db.Text  // 任意メモ (真に任意の値属性 → NULL 許容)

  // 作成者
  createdBy         String
  creator           User                @relation(fields: [createdBy], references: [id])

  // 確定/取消イベント (それぞれ高々 1 件)
  confirmation      OrderConfirmation?
  cancellation      OrderCancellation?

  // 受注明細は持たない。明細は variation → items (EstimateItem) を辿る
  // (受注明細 = 見積明細が常に完全一致するため。§4.2)

  createdAt         DateTime            @default(now())
  updatedAt         DateTime            @updatedAt

  @@index([variationId])
  @@index([createdBy])
}

// 受注確定イベント。確定したときのみ 1:1 で存在する（行の存在＝確定済み）
model OrderConfirmation {
  id                String   @id @default(cuid())
  orderId           String   @unique
  order             Order    @relation(fields: [orderId], references: [id], onDelete: Cascade)

  deliveryDate      DateTime            // 納期 (確定時必須 → NOT NULL)
  confirmedAt       DateTime
  confirmedBy       String
  confirmer         User     @relation("OrderConfirmedBy", fields: [confirmedBy], references: [id])

  createdAt         DateTime @default(now())
}

// 受注取消イベント。取消したときのみ 1:1 で存在する（行の存在＝取消済み）
model OrderCancellation {
  id                String   @id @default(cuid())
  orderId           String   @unique
  order             Order    @relation(fields: [orderId], references: [id], onDelete: Cascade)

  cancelledAt       DateTime
  cancelledBy       String
  canceller         User     @relation("OrderCancelledBy", fields: [cancelledBy], references: [id])

  createdAt         DateTime @default(now())
}
```

### 11.5 消費税率マスタ

適用期間を `effectiveFrom` / `effectiveTo` の 2 カラムで持つと「終了日 = NULL（現在も有効）」という開区間 NULL が避けられない。税率タイムラインは制度上**単調（途切れ・ギャップなし）**であることを利用し、**`effectiveFrom` 1 カラムのみ**で表現する。各行は「この日時から税率は X」というイベントで、前期間の終わりは**次の行の `effectiveFrom` が暗黙に決める**（「行の存在＝そこから新税率が有効」パターン。`OrderConfirmation` の存在＝確定済と同型の時系列版）。終端列を持たないので NULL も発生しない。

```prisma
model TaxRate {
  id              String    @id @default(cuid())
  rate            Decimal                   // 0.10 (10%)

  // この日時から有効な税率。前期間の終わり = 次の行の effectiveFrom（暗黙）。
  // @unique で同時刻重複を DB レベルで禁止（同時に B-tree 索引も提供）
  effectiveFrom   DateTime  @unique

  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt
}

// 初期データ例（effectiveFrom 昇順で並ぶ。終端列なし。隙間・重複なし）
// { rate: 0.08, effectiveFrom: 2014-04-01T00:00:00Z }
// { rate: 0.10, effectiveFrom: 2019-10-01T00:00:00Z }  ← 「現在も有効」も特別扱い不要
```

> **クエリ**: ある日時 D における適用税率は「`effectiveFrom <= D` の中で `effectiveFrom` が最大の行」で得る:
> `findFirst({ where: { effectiveFrom: { lte: D } }, orderBy: { effectiveFrom: 'desc' } })`。`@unique` が生む B-tree 索引により高速。範囲型 `tstzrange` / `Unsupported` / `$queryRaw` / マイグレーション手編集はいずれも不要で、Prisma ネイティブ API で完結する。
>
> **前提と再検討トリガ**: 本設計は税率タイムラインの単調性（制度上ギャップなし・予約終了なし）に依存する。仮にギャップや「将来予約終了」のような不連続を扱う必要が出た場合は、範囲型 `tstzrange` + `EXCLUDE USING gist (... WITH &&)` 制約への切替えを再検討する。

## 付録 A: 主要な業務ルール

### A.1 見積作成のルール

```
1. 見積番号は保存時に自動採番
2. 見積作成者の部署が自動設定される
3. 消費税率は作成時点の税率が自動設定される
4. 納品先見積は申請・受注作成不可
5. 事後見積は承認不要
```

### A.2 バリエーションのルール

```
1. 1見積につき申請できるバリエーションは1つのみ
2. 無効状態のバリエーションは編集・申請不可
3. 申請中・承認済みのバリエーションは無効化不可
4. バリエーション番号は連番で自動採番
```

### A.3 受注作成のルール

```
1. 受注番号は持たず、見積番号で管理
2. 承認済みバリエーションからのみ作成可能 (事後見積除く)
3. 1バリエーションから1受注のみ作成可能
4. 受注作成後、見積バリエーションは編集不可
5. 納期は受注確定時に必須
```

### A.4 金額計算のルール

```
1. 明細金額 = 数量 × 単価 × 掛率 - 明細値引
2. 消費税は小計後に一括計算
3. 消費税の端数処理は選択式
4. 保存時・申請時に自動再計算
5. 消費税率変更時はエラーチェック
```

### A.5 得意先・納品先のルール

```
1. 納品先見積を先に作成
2. 「得意先改訂」で得意先見積を作成
3. 申請・受注作成は得意先見積のみ可能
4. 納品先価格 - 得意先価格 = 粗利
5. 実際の納品は納品先へ行う
```
